# UI Architecture

## Decision

GameEngine should own the UI contract, not every low-level UI implementation detail.

The production game UI surface starts as first-party retained-mode `mirakana_ui` contracts. The editor UI model starts as `mirakana_editor_ui` contracts layered over GUI-independent editor state. These contracts define what games, tools, and AI agents see.

Low-level implementation details such as font rasterization, complex text shaping, IME integration, accessibility bridges, image decoding, and platform-specific input/window integration may use official platform SDKs or audited dependencies behind adapters. They must not leak into public game APIs.

## First-Party Ownership

The following must be first-party and stable enough for AI-driven game development:

- UI document tree, stable element ids, parent/child hierarchy, visibility, enabled state, and deterministic traversal.
- Style and layout contracts, including rows, columns, stacks, anchors, margins, padding, row/column gap, min/max size, DPI scale, and theme tokens.
- Input focus, pointer capture, keyboard/gamepad navigation, modal layers, and command routing.
- Semantic roles, localization keys, accessibility metadata, and platform-independent UI intent.
- Data binding primitives for one-way display binding and command invocation.
- Animation state for UI transitions without depending on renderer frame-graph internals.
- Runtime game UI APIs under `mirakana::ui` and editor model helpers under `mirakana::editor`.

The current `mirakana_ui` implementation is headless and contract-first. It provides deterministic document primitives, style/layout descriptors, a headless row/column/stack layout solver with row/column gap support, text metadata, image metadata, focus/navigation state, transition state, one-way text binding, command invocation, renderer submission payload construction with `RendererBox`, `RendererTextRun`, and image placeholder records, text adapter payload rows with line/span placeholders that keep resolved label text separate from localization keys and diagnose unresolved localization, an opt-in dependency-free `MonospaceTextLayoutPolicy` overload that produces deterministic logical-order line and glyph placeholder boxes with stable glyph ids, simple ASCII whitespace wrapping, clipping diagnostics, invalid-policy diagnostics, and unsupported right-to-left fallback diagnostics, image adapter payload rows with stable resource and asset URI references, accessibility payload construction with localization key, enabled, focusable, parent, and depth metadata, `AccessibilityPublishPlan` / `AccessibilityPublishResult` helpers that validate payload rows before dispatching to an `IAccessibilityAdapter`, `ImeCompositionPublishPlan` / `ImeCompositionPublishResult` helpers that validate target and cursor bounds before dispatching to an `IImeAdapter`, `PlatformTextInputSessionPlan` / `PlatformTextInputSessionResult` and `PlatformTextInputEndPlan` / `PlatformTextInputEndResult` helpers that validate first-party text-input begin/end requests before dispatching to an `IPlatformIntegrationAdapter`, `TextEditState` / `CommittedTextInput` helpers that validate committed UTF-8 text, cursor byte offsets, and selection byte ranges before applying text insertion/replacement, `TextEditCommandKind` / `TextEditCommand` helpers that validate and apply scalar-boundary cursor movement plus backward/forward deletion to the same `TextEditState`, `TextEditClipboardCommandKind` / `TextEditClipboardCommand` helpers that validate copy/cut/paste command rows and apply them to `TextEditState` through `IClipboardTextAdapter`, `ClipboardTextWriteRequest` / `ClipboardTextReadRequest` helpers that validate target rows, strict UTF-8 writes, empty clear writes, empty clipboard reads, and adapter-returned text before dispatching through `IClipboardTextAdapter`, `TextShapingRequestPlan` / `TextShapingResult` helpers that validate text shaping requests and adapter-returned runs before/after dispatching to an `ITextShapingAdapter`, and `FontRasterizationRequestPlan` / `FontRasterizationResult` helpers that validate font family, glyph id, pixel size, and adapter-returned allocation data before/after dispatching to an `IFontRasterizerAdapter`. It does not own fonts, implement shaping, reorder bidirectional text, decode images itself, publish OS accessibility bridge objects, interpret platform SDK events directly, own platform key mapping/key repeat, own virtual keyboard/session behavior, or open platform UI services directly.

`mirakana_platform_sdl3` provides the first concrete platform text-input begin/end adapter through `mirakana::SdlPlatformIntegrationAdapter`. It maps validated `PlatformTextInputRequest` rows to `SDL_SetTextInputArea` plus `SDL_StartTextInput` and maps validated end rows to `SDL_StopTextInput` for caller-owned `SdlWindow` instances while keeping `mirakana_ui` free of SDL3 headers and native handles. It also translates `SDL_EVENT_TEXT_INPUT`, `SDL_EVENT_TEXT_EDITING`, and `SDL_EVENT_TEXT_EDITING_CANDIDATES` into copied first-party `SdlWindowEvent` rows, including committed text, IME composition range metadata, candidate strings, selected candidate index, candidate-list orientation, SDL keycodes, and key modifiers, without retaining SDL-owned event pointers. `sdl3_ime_composition_from_window_event` and `publish_sdl3_ime_composition_event` can publish copied SDL3 text-editing composition rows through `IImeAdapter`, converting SDL UTF-8 character cursor positions to the byte offsets validated by `mirakana_ui`; `sdl3_committed_text_from_window_event` and `apply_sdl3_committed_text_event` can apply copied SDL3 committed text rows to a caller-owned `TextEditState`; `sdl3_text_edit_command_from_window_event` and `apply_sdl3_text_edit_command_event` can map copied SDL3 key-down rows for Left, Right, Home, End, Backspace, and Delete, including repeated key-down rows, into first-party `TextEditCommand` application; `SdlClipboardTextAdapter` maps the existing `SdlClipboard` text API into `IClipboardTextAdapter`; `sdl3_text_edit_clipboard_command_from_window_event` and `apply_sdl3_text_edit_clipboard_command_event` can map copied Ctrl/GUI + C/X/V SDL3 key-down rows into first-party `TextEditClipboardCommand` application through that adapter. This is only the SDL3 begin/end session, event translation, composition publish, committed-text apply, selected key-command bridge, selected clipboard-shortcut bridge, and text clipboard adapter plus first-party command/request application; full text editing widget behavior, full platform shortcut localization, key repeat policy, selection UI, copy/cut/paste widget behavior, rich clipboard formats, word movement, grapheme-cluster editing, candidate UI, virtual keyboard policy, OS accessibility, and platform-specific text services remain adapter work.

Runtime UI Image Decode Request Plan v1 adds `ImageDecodeRequestPlan` / `ImageDecodeDispatchResult` helpers plus `plan_image_decode_request` and `decode_image_request` around the existing `IImageDecodingAdapter` boundary. The helper validates adapter-safe non-empty asset URIs, non-empty source bytes, explicit `ImageDecodePixelFormat::rgba8_unorm`, dimensions, and pixel byte counts before treating adapter output as usable. Runtime UI PNG Image Decoding Adapter v1 adds `mirakana::PngImageDecodingAdapter` in `mirakana_tools` as an optional host/tool bridge from reviewed PNG bytes to `ImageDecodeResult` through the existing `decode_audited_png_rgba8` / `libspng` path; it fails closed when `asset-importers` is disabled or decoding fails. Runtime UI Decoded Image Atlas Package Bridge v1 adds `PackedUiAtlasAuthoringDesc`, `author_packed_ui_atlas_from_decoded_images`, `plan_packed_ui_atlas_package_update`, and `apply_packed_ui_atlas_package_update` so tool/editor code can turn validated RGBA8 decoded rows into one deterministic `GameEngine.CookedTexture.v1` atlas page plus `GameEngine.UiAtlas.v1` metadata and `.geindex` updates. Runtime UI Glyph Atlas Package Bridge v1 adds structured `UiAtlasMetadataGlyph` rows plus `PackedUiGlyphAtlasAuthoringDesc`, `author_packed_ui_glyph_atlas_from_rasterized_glyphs`, `plan_packed_ui_glyph_atlas_package_update`, and `apply_packed_ui_glyph_atlas_package_update` so already-rasterized RGBA8 glyph pixels can become cooked glyph atlas package rows with `source.decoding=rasterized-glyph-adapter` and `atlas.packing=deterministic-glyph-atlas-rgba8-max-side`. This does not parse SVG/vector data, upload textures, call platform imaging APIs, add codec or font dependencies to `mirakana_ui`, or make broad production UI image/font quality ready.

`mirakana_ui_renderer` is the first concrete engine adapter for this contract. It converts styled `RendererBox` payloads into `mirakana_renderer` `SpriteCommand` submissions through `UiRendererTheme`, can resolve deterministic monospace text glyph rows through `UiRendererGlyphAtlasPalette` / `UiRendererGlyphAtlasBinding` into atlas-backed sprite submissions, can build that glyph palette from runtime `GameEngine.UiAtlas.v1` glyph metadata through `build_ui_renderer_glyph_atlas_palette_from_runtime_ui_atlas`, and can resolve `ImageAdapterPayload` rows through `UiRendererImagePalette` / `UiRendererImageBinding` into image sprite submissions without adding renderer dependencies to `mirakana_ui`. Image resolution is deterministic: `resource_id` bindings are authoritative, and `asset_uri` bindings are used only as fallback when no resource binding exists. Glyph resolution is deterministic by `font_family` and glyph id. A binding may carry a first-party cooked atlas page id plus UV rect so the renderer-owned native overlay can sample an already-cooked texture in the narrow desktop package proof; package tooling can now create decoded-image and rasterized-glyph UI atlas rows through the `mirakana_tools` packed atlas bridges, but renderer upload and runtime residency still live outside `mirakana_ui`. The adapter reports deterministic text adapter row, text glyph availability/resolution/missing/submission, image placeholder, image sprite submission, missing image resource, accessibility node, and adapter diagnostic counts. This is a dependency-free renderer intent adapter; production text shaping implementation, real font loading/rasterization, real font-driven glyph atlas generation, broader source image codecs, SVG/vector parsing, renderer texture upload, broad atlas allocation/packing, and OS accessibility publication remain separate adapter work.

## Adapter Boundaries

The following are adapter responsibilities, not core public UI contracts:

- Submitting UI draw data through concrete `mirakana_renderer`/`mirakana_rhi` backends after `mirakana_ui` has built first-party renderer submission payloads.
- Text shaping, bidirectional text, line breaking, glyph metrics, and glyph fallback. `mirakana_ui` may validate first-party `TextLayoutRequest` rows and adapter-returned `TextLayoutRun` metadata around `ITextShapingAdapter`, while concrete shaping engines, bidi reordering, line breaking, and font fallback remain adapter work.
- Font loading, real font rasterization, shaping, glyph metrics/fallback, and font licensing records. `mirakana_ui` may validate first-party `FontRasterizationRequest` rows and adapter-returned `GlyphAtlasAllocation` metadata around `IFontRasterizerAdapter`, while `mirakana_tools` can package already-rasterized RGBA8 glyph pixels into cooked atlas metadata for `UiRendererGlyphAtlasPalette`.
- IME composition, platform text input, and text clipboard. `mirakana_ui` may validate first-party `ImeComposition` rows and hand them to `IImeAdapter`, validate committed text input rows before applying insertion/replacement to a first-party `TextEditState`, validate first-party `TextEditCommand` rows before applying scalar-boundary cursor movement and backward/forward deletion, validate first-party `TextEditClipboardCommand` rows before applying copy/cut/paste over caller-owned selection state through `IClipboardTextAdapter`, validate text clipboard read/write rows before `IClipboardTextAdapter` dispatch, and validate `PlatformTextInputRequest` begin/end session rows before `IPlatformIntegrationAdapter` dispatch. `mirakana_platform_sdl3` can now service the narrow SDL3 begin/end lane for `SdlWindow`, translate SDL text input/editing/candidate/keycode/modifier events into copied first-party rows, publish SDL text-editing composition rows through `IImeAdapter`, apply SDL committed text rows through `TextEditState`, map selected SDL key-down rows to first-party text edit commands, map selected Ctrl/GUI + C/X/V rows to first-party text edit clipboard commands, and adapt `SdlClipboard` through `SdlClipboardTextAdapter`, but full text editing widget behavior, full platform shortcut localization, key repeat policy, selection UI, copy/cut/paste widget behavior, rich clipboard formats, word movement, grapheme-cluster editing, concrete Win32/TSF, Cocoa text input, ibus/Fcitx, Android, iOS IME session integration, candidate UI, virtual keyboard behavior, and native text-input object ownership remain adapter work.
- OS accessibility bridges and screen-reader integration. `mirakana_ui` may validate and hand off first-party accessibility rows to `IAccessibilityAdapter`, but concrete UI Automation, NSAccessibility, AT-SPI, Android, or iOS publication remains adapter work.
- Image decoding or SVG/vector decoding for UI assets. `mirakana_ui` may validate first-party `ImageDecodeRequest` rows and adapter-returned RGBA8 `ImageDecodeResult` metadata around `IImageDecodingAdapter`; `mirakana_tools` may provide reviewed dependency adapters such as `PngImageDecodingAdapter` and may pack validated decoded RGBA8 rows into cooked UI atlas package artifacts through `author_packed_ui_atlas_from_decoded_images`, but concrete codec execution, vector parsing, renderer texture upload, and broad atlas policy remain adapter work outside the `mirakana_ui` contract.
- Middleware adapters for Qt, NoesisGUI, Slint, RmlUi, or similar systems.

Adapters may depend on platform SDKs or audited third-party libraries only after `license-audit`, `docs/dependencies.md`, `vcpkg.json`, and `THIRD_PARTY_NOTICES.md` are updated.

## Dear ImGui Role

Dear ImGui remains the optional developer/editor/debug shell for fast iteration, diagnostics, and early tools. It is not the production game UI foundation and must not be required by generated game runtime code.

The current `mirakana_editor` shell may render panels with Dear ImGui while durable panel state lives in `editor/core`. Future high-complexity panels should be modeled through `mirakana_editor_ui` first, then rendered by an adapter.

## Middleware Policy

Qt, NoesisGUI, Slint, RmlUi, or similar frameworks are optional adapter candidates. They must not become the core `mirakana_ui` public API, and generated games must not depend on their headers or object models directly.

Qt in particular requires explicit licensing review before any adoption because its open-source and commercial licensing models impose different obligations.

## Non-Goals

- Do not hand-roll mature low-level text shaping, IME, accessibility, or font systems without an accepted architecture decision proving that the official SDK or audited dependency path is worse.
- Do not expose SDL3, Dear ImGui, Qt, NoesisGUI, Slint, RmlUi, OS window handles, or graphics API handles through public game UI APIs.
- Do not add fonts, UI art, UI middleware, or text/font libraries without dependency and notice records.
