# UI Architecture

## Decision

GameEngine should own the UI contract, not every low-level UI implementation detail.

The production game UI surface starts as first-party retained-mode `mirakana_ui` contracts. The editor UI model starts as `mirakana_editor_ui` contracts layered over GUI-independent editor state. These contracts define what games, tools, and AI agents see.

Low-level implementation details such as font rasterization, complex text shaping, IME integration, accessibility bridges, image decoding, and platform-specific input/window integration may use official platform SDKs or audited dependencies behind adapters. They must not leak into public game APIs.

## First-Party Ownership

The following must be first-party and stable enough for AI-driven game development:

- UI document tree, stable element ids, parent/child hierarchy, visibility, enabled state, and deterministic traversal.
- Style and layout contracts, including rows, columns, stacks, anchors, margins, padding, row/column gap, min/max size, DPI scale, and theme tokens.
- Input focus, pointer capture, keyboard/gamepad navigation, modal layers, and command routing.
- Semantic roles, localization keys, accessibility metadata, and platform-independent UI intent.
- Data binding primitives for one-way display binding and command invocation.
- Animation state for UI transitions without depending on renderer frame-graph internals.
- Runtime game UI APIs under `mirakana::ui` and editor model helpers under `mirakana::editor`.

## Adapter Boundaries

The following are adapter responsibilities, not core public UI contracts:

- Submitting UI draw data through concrete `mirakana_renderer`/`mirakana_rhi` backends after `mirakana_ui` has built first-party renderer submission payloads.
- Text shaping, bidirectional text, line breaking, glyph metrics, and glyph fallback. `mirakana_ui` may validate first-party `TextLayoutRequest` rows and adapter-returned `TextLayoutRun` metadata around `ITextShapingAdapter`; `ILineBreakingAdapter` returns separate `TextLineBreakRun` rows so line-breaking adapters do not need to fabricate shaping glyph evidence. Concrete shaping engines, bidi reordering, line breaking, and font fallback remain adapter work.
- Font loading, real font rasterization, shaping, glyph metrics/fallback, and font licensing records. `mirakana_ui` may validate first-party `FontRasterizationRequest` rows and adapter-returned `GlyphAtlasAllocation` metadata around `IFontRasterizerAdapter`, while `mirakana_tools` can package already-rasterized RGBA8 glyph pixels into cooked atlas metadata for `UiRendererGlyphAtlasPalette`.
- IME composition, platform text input, and text clipboard. `mirakana_ui` may validate first-party `ImeComposition` rows and hand them to `IImeAdapter`, validate committed text input rows before applying insertion/replacement to a first-party `TextEditState`, validate first-party `TextEditCommand` rows before applying scalar-boundary cursor movement and backward/forward deletion, validate first-party `TextEditClipboardCommand` rows before applying copy/cut/paste over caller-owned selection state through `IClipboardTextAdapter`, validate text clipboard read/write rows before `IClipboardTextAdapter` dispatch, validate `PlatformTextInputRequest` begin/end session rows including strict UTF-8 surrounding text, scalar-boundary cursor byte offset, and selection byte range before `IPlatformIntegrationAdapter` dispatch, and validate first-party IME parity evidence through `TextInputParityEvidenceRequest` before shell readiness promotion. `mirakana_platform_win32` can translate selected Windows committed-text, edit-key, and clipboard shortcut rows into copied first-party rows, while the private editor TSF adapter can report selected `ITextStoreACP` text-store evidence. Full text editing widget behavior, full platform shortcut localization, key repeat policy, selection UI, copy/cut/paste widget behavior, rich clipboard formats, word movement, custom candidate UI rendering, Cocoa text input, IBus/Fcitx, Android, iOS IME session integration, virtual keyboard behavior, and public native text-input object ownership remain adapter work.
- OS accessibility bridges and screen-reader integration. `mirakana_ui` may validate and hand off first-party accessibility rows to `IAccessibilityAdapter`, but concrete UI Automation, NSAccessibility, AT-SPI, Android, or iOS publication remains adapter work.
- Image decoding or SVG/vector decoding for UI assets. `mirakana_ui` may validate first-party `ImageDecodeRequest` rows and adapter-returned RGBA8 `ImageDecodeResult` metadata around `IImageDecodingAdapter`; `mirakana_tools` may provide reviewed dependency adapters such as `PngImageDecodingAdapter` and may pack validated decoded RGBA8 rows into cooked UI atlas package artifacts through `author_packed_ui_atlas_from_decoded_images`, but concrete codec execution, vector parsing, renderer texture upload, and broad atlas policy remain adapter work outside the `mirakana_ui` contract.
- Middleware adapters for Qt, NoesisGUI, Slint, RmlUi, or similar systems.

Adapters may depend on platform SDKs or audited third-party libraries only after `license-audit`, `docs/dependencies.md`, `vcpkg.json`, and `THIRD_PARTY_NOTICES.md` are updated.

## First-Party Editor Shell

The native `MK_editor` shell uses first-party retained `mirakana::ui` documents and `MK_ui_renderer` submissions in the dependency-free `desktop-editor` lane. It is not a runtime game UI dependency and must not be required by generated game runtime code.

The visible `MK_editor` shell keeps Win32, D3D12, DXGI, swapchains, descriptor heaps, fences, and message bridging in private `editor/src` implementation files. Durable panel state remains in `editor/core`; high-complexity panels should be modeled through `mirakana_editor_ui` first, then rendered by a first-party platform/rendering adapter.

Editor/UI performance evidence is currently narrow baseline telemetry plus deterministic retained diff/cache evidence: `MK_editor_core` exposes value-only `EditorUiPerformanceBudget` / `EditorUiPerformanceSample` summaries, `MK_ui` exposes retained snapshot/diff rows, and the native shell smoke reports `editor_ui_performance_budget_status=ready`, measured p95 layout/document-build/renderer-submission rows, text/render/visible-texture/memory rows, `editor_ui_performance_budget_violations=0`, `editor_ui_performance_diagnostics=0`, `editor_ui_performance_broad_optimization_claimed=0`, `ui_retained_diff_status=ready`, `ui_retained_dirty_rows=0`, positive layout/text/submission cache hit rows, zero layout/text cache miss rows, `ui_retained_submission_rebuilt_rows=0`, and `ui_retained_cache_native_handle_access=0`. These rows prove stable frame-to-frame first-party UI cache evidence for the selected editor smoke only; broad editor/UI optimization remains a later aggregate gate.

First-Party UI Editor Excellence v1 Phase 12 adds the exact selected-row aggregate for the editor lane: `first_party_editor_excellence_status=ready`, `first_party_editor_excellence=1`, `first_party_editor_excellence_windows_d3d12=1`, `first_party_editor_excellence_vulkan=0`, `first_party_editor_excellence_metal=0`, `first_party_editor_excellence_cross_platform=0`, `first_party_editor_excellence_text_parity=0`, `first_party_editor_excellence_accessibility_parity=0`, `first_party_editor_excellence_broad_optimization_claimed=0`, and `first_party_editor_excellence_native_handles_exposed=0`. They do not execute non-Windows shells or make Android/iOS editor shells ready.

## Middleware Policy

Qt, NoesisGUI, Slint, RmlUi, or similar frameworks are optional adapter candidates. They must not become the core `mirakana_ui` public API, and generated games must not depend on their headers or object models directly.

Qt in particular requires explicit licensing review before any adoption because its open-source and commercial licensing models impose different obligations.

## Non-Goals

- Do not hand-roll mature low-level text shaping, IME, accessibility, or font systems without an accepted architecture decision proving that the official SDK or audited dependency path is worse.
- Do not expose Dear ImGui, Qt, NoesisGUI, Slint, RmlUi, OS window handles, third-party windowing APIs, or graphics API handles through public game UI APIs.
- Do not add fonts, UI art, UI middleware, or text/font libraries without dependency and notice records.
