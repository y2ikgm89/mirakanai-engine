# Architecture

## Direction

GameEngine uses a core-first architecture. The first-party C++ API must be usable in headless tests before any desktop window, mobile lifecycle, or graphics backend is required.

## Layers

```text
game code
  -> engine/core
  -> engine/platform
  -> engine/runtime
  -> engine/runtime_host
  -> engine/renderer
  -> engine/rhi
  -> backend implementations
```

`engine/core` owns deterministic runtime concepts:

- application loop contracts
- logging contracts
- dependency-free diagnostics, counters, profile sample capture, and deterministic trace JSON export
- time stepping
- entity identifiers
- component registry
- core errors and versioning

`engine/math` owns standard-library-only 2D/3D math primitives:

- vectors
- 4x4 matrices
- 2D/3D transforms with position, scale, and radians-based rotation

`engine/assets` owns logical asset identity and registry contracts. Game manifests (`games/<target>/game.agent.json`) carry **descriptor-only** AI and tooling surfaces alongside runtime lists; for 3D desktop packages, `registeredSourceAssetCookTargets` rows pair reviewed `cook-registered-source-assets` root keys and dependency expansion with a specific `prefabScenePackageAuthoringTargets` id so `sourceRegistryPath` and `packageIndexPath` stay identical across prefab authoring and registered-source cook plans. Those rows are JSON data validated by repository scripts, not embedded shell commands or argv fragments. Asset Identity v2 is implemented as a foundation-only contract through `AssetKeyV2`, deterministic validation diagnostics, stable `AssetId` derivation from keys, `GameEngine.AssetIdentity.v2` text IO, and `plan_asset_identity_placements_v2` production placement resolution from reviewed keys to `AssetId`/kind/source rows; it does not complete scene/render/UI/gameplay reference cleanup, asset command apply surfaces, renderer/RHI residency, package streaming, or 2D/3D playable vertical slices. The module also provides asset records, an asset dependency graph, hot reload snapshot/event tracking, debounce-based recook request scheduling with dependency invalidation, failed-reload rollback state, runtime asset replacement staging with safe-point commits, recook import plan extraction, cooked package index serialization with content hashes, dependency edges, and deterministic recook decisions, shader source metadata, generated shader artifact tracking, shader descriptor reflection rows, shader compiler command planning including DXC Vulkan SPIR-V commands with `-spirv` and `-fspv-target-env=vulkan1.3`, shell-free shader tool runner validation, compiler output capture contracts, shader include dependency discovery, shader include dependency graph edge generation, deterministic shader artifact manifest serialization, texture/mesh/audio/material/scene import metadata, scene mesh/material/sprite dependency edge kinds, first-party texture/mesh/audio source document decoding including optional hex byte payloads, asset import planning, material definition serialization, material instance serialization/composition, and explicit backend-neutral material binding metadata including uniform-buffer, sampled-texture, and sampler resource rows without depending on renderer, RHI, scene, editor, or tools modules. External PNG decoded bytes, common-audio decoded bytes, glTF triangle mesh vertex/index byte cooking for position-only or lit position+normal+uv+tangent layouts, and first-party `GameEngine.Scene.v1` source validation/canonicalization are owned by `engine/tools` adapters so runtime/game code remains cooked-artifact-only.

`engine/audio` owns audio graph and mixing contracts. edit actions, Inspector transform and camera/light/mesh/sprite component controls, viewport transform delta controls, project bundle save/open commands, dirty status, project document state, a Editor GPU marker/timestamp visualization, allocator diagnostics panels, telemetry upload, crash reports, project trace import/file-open workflows, arbitrary trace conversion, and production flame graphs are still future observability work. Dear ImGui is the developer-tool shell, not the game/runtime UI foundation. It depends on engine public modules, `MK_editor_core`, and optional GUI dependencies, but engine runtime modules do not depend on editor code.

The visible `MK_editor` shell owns explicit dynamic game-module driver loading for Play-In-Editor. It reviews the path and factory symbol through `MK_editor_core`, loads the module through `MK_platform` `DynamicLibrary`, resolves `mirakana_create_editor_game_module_driver_v1`, wraps the returned `GameEngine.EditorGameModuleDriver.v1` function table as an `IEditorPlaySessionDriver`, and keeps the module loaded until the driver is destroyed. Editor core does not load modules or expose native handles, and the shell blocks unload while the isolated play session is active; hot reload, active-session reload/unload, stable third-party ABI, `DesktopGameRunner` embedding, package scripts, validation recipes, renderer/RHI uploads or handles, native handles, and package streaming remain outside this boundary.

## Dependency Rules

- `engine/core` depends only on the C++ standard library. Shared observability contracts such as diagnostics, counters, clocks, profile samples, and capture-to-trace serialization belong here when renderer, runtime host, platform, and editor modules all need them.
- `engine/assets` depends only on the C++ standard library. Scene import metadata stores asset ids and dependency edges only; parsing or canonicalizing scene documents stays outside `engine/assets`.
- `engine/animation` may depend on math.
- `engine/ai` depends only on the C++ standard library. It may own explicit caller-supplied behavior-tree blackboard value snapshots, condition comparisons, and explicit caller-supplied perception target rows projected into blackboard facts, but game-specific leaf action execution, scene queries, physics queries, navigation requests, scripting, blackboard persistence/services, and editor graph tooling must stay outside the core `MK_ai` evaluator or behind later adapters.
- `engine/audio` may depend on assets for logical clip ids but must not depend on platform audio APIs.
- `engine/audio/sdl3` may depend on SDL3 and `engine/audio`; it must not expose SDL handles or SDL headers through public generated-game APIs.
- `engine/math` depends only on the C++ standard library.
- `engine/platform` may depend on math. Public platform contracts must not expose native handles; per-OS implementation files and mobile packaging templates may depend on native OS APIs behind first-party interfaces. Dynamic-library loading must require explicit absolute paths and reviewed symbol names; Windows uses `LoadLibraryExW` with `LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR` and `LOAD_LIBRARY_SEARCH_SYSTEM32` rather than current-directory or `PATH` search. File access through `RootedFileSystem` must remain root-relative and reject absolute or parent-relative paths. File watching must flow through first-party `FileWatch*` contracts and keep native watcher handles backend-private.
- `engine/runtime` may depend on assets and platform contracts. It owns game-facing cooked package loading, runtime asset handles, typed cooked payload access, session documents, and runtime diagnostic reports, while source import/cook/tool execution stays in `engine/tools` and graphics/audio upload stays in renderer/RHI/audio adapters.
- `engine/runtime_rhi` may depend on runtime and RHI contracts. It owns backend-neutral runtime texture upload, mesh vertex/index buffer upload, and material descriptor binding helpers, but must not expose backend-native handles or make `engine/runtime` depend on RHI.
- `engine/runtime_scene` may depend on runtime and scene contracts. It owns renderer-free cooked scene instantiation and reference diagnostics for gameplay, but must not depend on renderer, RHI, scene_renderer, editor code, platform windows, SDL3, Dear ImGui, or native handles.
- `engine/runtime_scene_rhi` may depend on runtime, runtime_rhi, scene_renderer, and RHI contracts. It may build retained scene GPU binding palettes from cooked package payloads, but must not make `engine/runtime` or `engine/scene` depend on RHI, concrete backends, platform windows, editor APIs, or native handles.
- `engine/runtime_host` may depend on core, platform, and renderer contracts. It owns game host loop orchestration and first-party `MK_core` observability adapter points, not platform-native polling, editor UI, or backend-native presentation handles.
- `engine/runtime_host/sdl3` may depend on SDL3, `engine/platform/sdl3`, and `engine/runtime_host`; it must keep SDL headers and handles out of public game APIs and remain optional behind the desktop GUI build lane.
- `engine/physics` may depend on math.
- `engine/navigation` depends only on the C++ standard library.
- `engine/tools` may depend on asset, platform, scene, and UI contracts for development-time cook/validation adapters such as the optional PNG UI image decoding bridge, but runtime game modules must not depend on it for normal gameplay.
- `engine/platform/sdl3` may depend on SDL3 and `engine/platform`.
- `engine/renderer` may depend on asset identity, core/math, and RHI contracts, not on game logic or RHI backends.
- `engine/rhi` depends only on the C++ standard library.
- `engine/rhi/d3d12` may depend on the Windows SDK and `engine/rhi`.
- `engine/rhi/vulkan` depends on `engine/rhi` and platform dynamic-library APIs for runtime loader probing; native Vulkan handles, function pointers, surface queries, swapchains, shader modules, descriptors, synchronization, and presentation details stay private to implementation files.
- `engine/rhi/metal` public headers depend only on `engine/rhi`; native Metal/QuartzCore dependencies stay private to Apple-only implementation files.
- `engine/scene` may depend on math and asset identity contracts, not on renderer or platform code. Renderer-facing extraction must stay renderer-neutral and avoid RHI/backend handles.
- `engine/scene_renderer` may depend on runtime_scene, scene, asset material, and renderer contracts, but must not depend on RHI backends or editor code.
- `engine/ui` must remain a first-party runtime contract layer and must not depend on editor, SDL3, Dear ImGui, native UI middleware, OS windows, or GPU APIs. Text, font, IME, accessibility, image decoding, and renderer integration must sit behind adapters.
- `editor/core` may depend on scene, asset, animation, tool, and UI contracts but must remain GUI-independent.
- `editor` may depend on optional UI/windowing dependencies and public engine modules. It may adapt editor-core or future `MK_editor_ui` models to Dear ImGui, but it must not make Dear ImGui part of public game/runtime APIs.
- Backend modules depend on their graphics API and the renderer interfaces.
- Examples and tests may depend on multiple modules.
