# Architecture

## Direction

GameEngine uses a core-first architecture. The first-party C++ API must be usable in headless tests before any desktop window, mobile lifecycle, or graphics backend is required.

## Layers

```text
game code
  -> engine/core
  -> engine/platform
  -> engine/runtime
  -> engine/runtime_host
  -> engine/renderer
  -> engine/rhi
  -> backend implementations
```

`engine/core` owns deterministic runtime concepts:

- application loop contracts
- logging contracts
- dependency-free diagnostics, counters, profile sample capture, and deterministic trace JSON export
- time stepping
- entity identifiers
- component registry
- core errors and versioning

`engine/math` owns standard-library-only 2D/3D math primitives:

- vectors
- 4x4 matrices
- 2D/3D transforms with position, scale, and radians-based rotation

`engine/assets` owns logical asset identity and registry contracts. Game manifests (`games/<target>/game.agent.json`) carry **descriptor-only** AI and tooling surfaces alongside runtime lists; for 3D desktop packages, `registeredSourceAssetCookTargets` rows pair reviewed `cook-registered-source-assets` root keys and dependency expansion with a specific `prefabScenePackageAuthoringTargets` id so `sourceRegistryPath` and `packageIndexPath` stay identical across prefab authoring and registered-source cook plans. Those rows are JSON data validated by repository scripts, not embedded shell commands or argv fragments. Asset Identity v2 is implemented as a foundation-only contract through `AssetKeyV2`, deterministic validation diagnostics, stable `AssetId` derivation from keys, `GameEngine.AssetIdentity.v2` text IO, and `plan_asset_identity_placements_v2` production placement resolution from reviewed keys to `AssetId`/kind/source rows; it does not complete scene/render/UI/gameplay reference cleanup, asset command apply surfaces, renderer/RHI residency, package streaming, or 2D/3D playable vertical slices. The module also provides asset records, an asset dependency graph, hot reload snapshot/event tracking, debounce-based recook request scheduling with dependency invalidation, failed-reload rollback state, runtime asset replacement staging with safe-point commits, recook import plan extraction, cooked package index serialization with content hashes, dependency edges, and deterministic recook decisions, shader source metadata, generated shader artifact tracking, shader descriptor reflection rows, shader compiler command planning including DXC Vulkan SPIR-V commands with `-spirv` and `-fspv-target-env=vulkan1.3`, shell-free shader tool runner validation, compiler output capture contracts, shader include dependency discovery, shader include dependency graph edge generation, deterministic shader artifact manifest serialization, texture/mesh/audio/material/scene import metadata, scene mesh/material/sprite dependency edge kinds, first-party texture/mesh/audio source document decoding including optional hex byte payloads, asset import planning, material definition serialization, material instance serialization/composition, and explicit backend-neutral material binding metadata including uniform-buffer, sampled-texture, and sampler resource rows without depending on renderer, RHI, scene, editor, or tools modules. External PNG decoded bytes, common-audio decoded bytes, glTF triangle mesh vertex/index byte cooking for position-only or lit position+normal+uv+tangent layouts, and first-party `GameEngine.Scene.v1` source validation/canonicalization are owned by `engine/tools` adapters so runtime/game code remains cooked-artifact-only.

## Memory And Lifetime Boundaries

GameEngine uses explicit ownership boundaries before allocator, job-system, NUMA, SIMD, GPU residency, or vendor-specific optimization work. Public contracts use value rows, immutable snapshots, generation-checked ids, first-party handles, `std::unique_ptr` ownership transfer, `std::span` / `std::string_view` borrowed views, and non-owning raw pointers or references only when the owner and lifetime are clear from the surrounding contract.

Module-owned resources stay inside their module or backend. Runtime package contents are owned by runtime package/store/catalog objects; renderer and RHI resources are owned by renderer/RHI devices and lifetime registries; editor and host shell native state is owned by private shell adapters. Cross-module handoffs use copied rows, stable handles, or explicit safe-point results instead of shared mutable ownership.

Frame-local scratch, upload staging, transient render rows, and borrowed views cannot outlive the frame, call, worker handoff, safe point, or GPU submission/retire boundary that created them. If data must cross a worker/thread boundary, package residency transition, or GPU fence boundary, copy it into owner-owned storage or convert it into a stable first-party handle/id plus explicit diagnostics.

Backend-private objects include Win32, D3D12, Vulkan, Metal, WASAPI, profiler SDK, allocator, queue, fence, descriptor, heap, and GPU memory objects. Official D3D12 and Vulkan models make application-owned memory, synchronization, residency, and fence/ownership transitions explicit, so GameEngine keeps those details behind backend/PIMPL or first-party opaque handles rather than exposing native handles to gameplay, generated games, manifests, or AI command surfaces.

AI-generated games must treat these lifetime rules as part of recipe selection. When the selected recipe lacks a stable handle, budget counter, safe-point result, trace artifact, or host-gated profiler evidence for a requested optimization, the correct outcome is an AI Engine Capability Handoff row, not a game-local raw pointer, native handle, allocator, or backend shortcut.

- `engine/scene` may depend on math and asset identity contracts, not on renderer or platform code. Renderer-facing extraction must stay renderer-neutral and avoid RHI/backend handles.
- `engine/scene_renderer` may depend on environment profiles, runtime_scene, scene, asset material, and renderer contracts, but must not depend on RHI backends or editor code.
- `engine/ui` must remain a first-party runtime contract layer and must not depend on editor, Dear ImGui, native UI middleware, OS windows, GPU APIs, or third-party windowing middleware. Text, font, IME, accessibility, image decoding, and renderer integration must sit behind adapters.
- `editor/core` may depend on scene, asset, animation, tool, and UI contracts but must remain GUI-independent.
- `editor` may depend on optional platform/windowing adapters and public engine modules. Active editor UI must use first-party retained `MK_editor` / `mirakana::ui` contracts; Dear ImGui, Qt, Slint, RmlUi, SDL3, and other UI middleware must not be added as active editor or runtime UI dependencies without a new explicit architecture decision and dependency/license update.
- Backend modules depend on their graphics API and the renderer interfaces.
- Examples and tests may depend on multiple modules.
