# Testing

## Test Shape

The initial project uses a tiny first-party C++ test harness to avoid third-party dependencies during bootstrap.

Default conventions for large **`tests/unit/*.cpp`** sources (clang/clang-tidy friendly aggregates; substring checks on **`std::string` and `std::string_view`** via **`contains`/`starts_with`** where applicable): see **Unit tests** in [`docs/cpp-style.md`](cpp-style.md).

Runtime UI accessibility, IME, platform text-input, font rasterization request, image decode request, and glyph atlas package bridge coverage lives across `MK_ui_renderer_tests`, `MK_tools_tests`, `MK_runtime_tests`, and `MK_asset_identity_runtime_resource_tests`: `AccessibilityPublishPlan`, `AccessibilityPublishResult`, `plan_accessibility_publish`, and `publish_accessibility_payload` validate first-party accessibility rows before `IAccessibilityAdapter` dispatch; `ImeCompositionPublishPlan`, `ImeCompositionPublishResult`, `plan_ime_composition_update`, and `publish_ime_composition` validate first-party `ImeComposition` target/cursor rows before `IImeAdapter` dispatch; `PlatformTextInputSessionPlan`, `PlatformTextInputSessionResult`, `PlatformTextInputEndPlan`, `PlatformTextInputEndResult`, `begin_platform_text_input`, and `end_platform_text_input` validate first-party `PlatformTextInputRequest` begin/end rows before `IPlatformIntegrationAdapter` dispatch; `FontRasterizationRequestPlan`, `FontRasterizationResult`, `plan_font_rasterization_request`, and `rasterize_font_glyph` validate first-party `FontRasterizationRequest` rows plus adapter-returned allocation metadata around `IFontRasterizerAdapter` dispatch; `ImageDecodeRequestPlan`, `ImageDecodeDispatchResult`, `plan_image_decode_request`, and `decode_image_request` validate first-party `ImageDecodeRequest` bytes plus adapter-returned RGBA8 image metadata around `IImageDecodingAdapter` dispatch; and `UiAtlasMetadataGlyph`, `RuntimeUiAtlasGlyph`, `author_packed_ui_glyph_atlas_from_rasterized_glyphs`, `plan_packed_ui_glyph_atlas_package_update`, and `build_ui_renderer_glyph_atlas_palette_from_runtime_ui_atlas` validate cooked glyph atlas metadata, package updates, and renderer palette construction from already-rasterized RGBA8 glyph pixels. OS accessibility bridge publication, native IME/text-input sessions, virtual keyboard behavior, native text-input object ownership, real font loading/rasterization, real font-driven glyph atlas generation, image decoding beyond reviewed adapters, source codecs, SVG/vector parsing, renderer texture upload, and broad atlas policy remain outside the default host-independent tests.

Runtime Resource Residency Hints Execution v1 coverage lives in `MK_asset_identity_runtime_resource_tests`: `execute_selected_runtime_package_streaming_safe_point` now rejects missing required preload assets and disallowed resident resource kinds before staging, preserves the active package/catalog on hint failure, and reports deterministic preload, resource-kind, and selected resident package-count counters on successful commit. Broad/background streaming, arbitrary eviction, resident caches, allocator/GPU budget enforcement, renderer/RHI resource ownership, and native handles remain outside this focused test lane.

Runtime Resource Resident Package Mount Set v1 coverage also lives in `MK_asset_identity_runtime_resource_tests`: `RuntimeResidentPackageMountSetV2` accepts explicit loaded package mounts, rejects zero/duplicate/missing mount ids without mutating the set, rebuilds `RuntimeResourceCatalogV2` from deterministic mount order with overlay policy, and proves unmount/rebuild invalidates stale handles through catalog generation changes. Disk/VFS mount discovery, async streaming, eviction, hot reload, renderer/RHI ownership, and native handles remain outside this focused test lane.

Runtime Resource Resident Catalog Cache v1 coverage lives in `MK_runtime_resource_resident_cache_tests`: `RuntimeResidentCatalogCacheV2` refreshes from a `RuntimeResidentPackageMountSetV2`, reports cache-hit reuse when mount generation, overlay policy, and budget are unchanged, rebuilds when the mount-set generation changes, and preserves the previous catalog and live handles when a changed budget fails. Disk/VFS mount discovery, async streaming, eviction, hot reload, renderer/RHI ownership, and GPU allocator enforcement remain outside this focused test lane.

Runtime Resource Streaming Resident Mount Commit v1 coverage lives in `MK_runtime_package_streaming_resident_mount_tests`: the resident-mount overload of `execute_selected_runtime_package_streaming_safe_point` mounts an already-loaded streaming package under an explicit `RuntimeResidentPackageMountIdV2`, refreshes `RuntimeResidentCatalogCacheV2`, reports `resident_mount` and `resident_catalog_refresh` rows, rejects duplicate mount ids before mutation, rejects duplicate raw loaded-package records before mount/cache mutation, and preserves the previous mount set and cached catalog when the projected resident budget fails. Filesystem/VFS discovery, background streaming, eviction, upload/staging package integration, renderer/RHI ownership, hot reload, and native handles remain outside this focused lane.

Runtime Resource Resident Unmount Cache Refresh v1 coverage lives in `MK_runtime_resource_resident_unmount_tests`: `commit_runtime_resident_package_unmount_v2` removes one explicit `RuntimeResidentPackageMountIdV2` from `RuntimeResidentPackageMountSetV2` and refreshes `RuntimeResidentCatalogCacheV2` through copy-then-commit, reports deterministic unmount and catalog refresh rows, rejects missing ids before live mutation, and preserves the previous live mount set and cached catalog while retaining projected refresh diagnostics when the remaining resident view exceeds the budget. Filesystem/VFS discovery, background streaming, eviction, upload/staging package integration, renderer/RHI ownership, hot reload, and native handles remain outside this focused lane.

Runtime Resource Resident Package Replacement Commit v1 coverage lives in `MK_runtime_resource_resident_replace_tests`: `commit_runtime_resident_package_replace_v2` replaces one existing explicit `RuntimeResidentPackageMountIdV2` with an already-loaded package while preserving the mount slot/order, preflights raw candidate catalog validity before overlay merge, refreshes `RuntimeResidentCatalogCacheV2` through copy-then-commit, rejects invalid and missing ids before live mutation, rejects duplicate raw candidate records before cache refresh, and preserves the previous live mount set and cached catalog on projected budget failure. Filesystem/VFS discovery, background streaming, eviction, upload/staging package integration, renderer/RHI ownership or teardown, broad hot reload productization, and native handles remain outside this focused lane.

## CI validation matrix (GitHub Actions)

Workflow: [.github/workflows/validate.yml](../.github/workflows/validate.yml). Artifact uploads preserve `Testing/**/*.log` and static-analysis evidence when jobs fail or always().

`pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-ci-matrix.ps1` is the local static drift guard for the reviewed CI lanes. It verifies `.github/workflows/validate.yml`, `.github/workflows/ios-validate.yml`, `engine/agent/manifest.json`, and `tools/validate.ps1` still expose the Windows/Linux/sanitizer/static-analysis/macOS/iOS jobs, Node 24-backed first-party GitHub Actions, the explicit Windows Server 2025 + Visual Studio 2026 runner label, package/log/coverage/static-analysis artifacts, C++23 release package artifact validation, read-only workflow token permissions, stale pull-request run cancellation, and host-gated iOS evidence rows. `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` runs this check; it does not execute GitHub Actions locally.

PR check selection policy: branch-protection-required checks must either run for every PR or be represented by an always-running aggregate gate. Do not make path-filtered or otherwise skippable workflows branch-protection-required because GitHub can leave skipped required checks pending. Heavy hosted lanes may be conditional when an always-running gate reports the selection, but run the full hosted matrix on main push, release, scheduled/nightly, and workflow_dispatch runs, and run full or near-full PR coverage when changes touch `.github/`, `CMakeLists.txt`, `CMakePresets.json`, `cmake/`, `vcpkg.json`, `tools/`, `engine/`, `editor/`, `games/`, packaging, validation policy, agent surfaces, or public/runtime contracts. Documentation-only slices may use a narrower hosted/local tier only when the PR body explains why the changed files cannot affect build or runtime behavior. Keep required job names unique across workflows so branch protection cannot resolve an ambiguous status check.

## Linux coverage thresholds

Policy lives in [tools/coverage-thresholds.json](../tools/coverage-thresholds.json) (`minLineCoveragePercent`, optional `lcovRemovePatterns`). On Linux:

- `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-coverage.ps1` runs the **separate** `coverage` preset under `out/build/coverage`, executes CTest including `ctest -T Coverage`, runs **`lcov --capture`** / **`lcov --summary`**, and compares the reported **line %** to the minimum (warnings only unless you use strict modes below).
- **`pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-coverage.ps1 -Strict`** (or `./tools/check-coverage.ps1 -Strict`) **fails** if line coverage is below the minimum or if `lcov`/`gcov` is missing. CI uses **`-Strict`** in [.github/workflows/validate.yml](../.github/workflows/validate.yml).

Install **`lcov`**, **`gcov`** / **`g++`**, and **Ninja** on the host; Ubuntu/Debian: `apt install lcov g++ ninja-build`. Raise `minLineCoveragePercent` only when sustained CI/`lcov --summary` evidence supports it (see [coverage threshold plan](superpowers/plans/2026-05-03-coverage-threshold-policy-v1.md)).

Checked-in CMake build presets inherit `normalized-build-environment`, which sets a single child `Path` from parent `PATH`/`Path` values so raw `cmake --build --preset ...` does not pass duplicate path keys to MSBuild tool tasks on Windows. `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` enforces that every visible build preset inherits this hidden preset. Repository `tools/*.ps1` wrappers use the resolved CMake/CTest tools reported by `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` and normalize the child-process environment before launching build tools.
`pwsh -NoProfile -ExecutionPolicy Bypass -File tools/test.ps1` builds the `dev` preset and then invokes `ctest --preset dev --output-on-failure --timeout 300` directly instead of routing tests through the generator-specific `RUN_TESTS` target, keeping the scripted lane on the same CTest preset path documented above.

Static analysis uses the repository wrapper:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1
pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1 -Strict -MaxFiles 1
pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1 -Files engine/physics/src/physics3d.cpp
```

`tools/check-tidy.ps1` verifies `.clang-tidy`, accepts native `compile_commands.json` from Makefile/Ninja generators, and uses CMake File API codemodel data to synthesize a clang-tidy-compatible compile database for the default Windows Visual Studio `dev` preset. Missing CMake or clang-tidy is reported as a concrete local tool blocker; the compile database is no longer a Visual Studio generator host gate for the default `dev` preset. `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` runs the same wrapper with `-MaxFiles 1` as a fast static-analysis smoke gate; run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1 -Strict` for a full-repository clang-tidy pass using automatic CPU-count parallelism, or pass `-Jobs <N>` to pin a lower or higher explicit throttle. `-Jobs 0` is the default and resolves to the host CPU count before capping at the selected translation-unit count. `-Strict` passes `--warnings-as-errors=*` so surfaced clang-tidy diagnostics fail the lane instead of remaining warning-only, and `-Jobs` parallelizes independent translation units while preserving diagnostics and exit codes. `.clang-tidy` keeps `HeaderFilterRegex` absolute-path aware (`(^|.*[/\\])(engine|editor|examples|games|tests)[/\\].*`) because CI and Visual Studio File API compile databases contain absolute source paths. The wrapper suppresses only clang frontend summary lines like `NN warnings generated.` after preserving actionable diagnostics and exit codes. The GitHub Actions `static-analysis` job runs that full `-Strict` wrapper on `ubuntu-latest` through the `ci-linux-tidy` preset, which keeps module scanning/import-std off so clang-tidy does not depend on build-generated CMake module map files. Local `ci-matrix-check` only verifies the lane exists. Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1 -Files <repo-relative-source.cpp>` for changed-file analysis through the same wrapper and compile database; `-Files` accepts repeated or comma-separated source paths, preserves the requested order before `-MaxFiles`, and rejects headers, missing paths, out-of-repository paths, and sources not present in the active compile database instead of silently falling back to ad hoc analysis. `tools/check-validation-recipe-runner.ps1` checks reviewed DryRun/Execute **rejection** paths for `run-validation-recipe.ps1` without running the full `agent-contract` workload twice; the integration suite `tools/check-ai-integration.ps1` (same as `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-ai-integration.ps1`) runs **once** immediately after that gate.

Current policy follows the official clang-tidy compile database/header filter/warnings-as-errors/config verification model, CMake `CMAKE_EXPORT_COMPILE_COMMANDS` support for Makefile/Ninja generators, and GitHub Actions workflow concurrency plus least-privilege token guidance: [clang-tidy](https://clang.llvm.org/extra/clang-tidy/), [CMAKE_EXPORT_COMPILE_COMMANDS](https://cmake.org/cmake/help/latest/variable/CMAKE_EXPORT_COMPILE_COMMANDS.html), [workflow concurrency](https://docs.github.com/en/actions/how-tos/write-workflows/choose-when-workflows-run/control-workflow-concurrency), and [GITHUB_TOKEN permissions](https://docs.github.com/en/actions/security-guides/automatic-token-authentication).

Optional vcpkg-backed lanes set official vcpkg process environment variables through `tools/common.ps1`: downloads go to `out/vcpkg/downloads`, the file binary cache goes to `out/vcpkg/binary-cache`, and metrics are disabled. This applies to bootstrap, GUI, desktop runtime, desktop runtime package, asset importer, and GUI C++23 evaluation wrappers.

Run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1` before optional vcpkg-backed lanes. It is the only wrapper that runs `vcpkg install`, and it installs the `desktop-runtime`, `desktop-gui`, and `asset-importers` manifest features into `vcpkg_installed`. Optional CMake presets set `VCPKG_MANIFEST_INSTALL=OFF` and `VCPKG_INSTALLED_DIR=${sourceDir}/vcpkg_installed`, so configure/build/package validation consumes that install tree without invoking vcpkg during CMake configure.

In sandboxed agent environments, `bootstrap-deps` may need explicit approval to run outside the sandbox because it is the dependency acquisition/build step. The recurring validation and packaging commands are designed to avoid that vcpkg path after bootstrap.

`tools/run-validation-recipe.ps1` is the reviewed AI-facing runner for the allowlisted validation recipes: `agent-contract`, `default`, `public-api-boundary`, `shader-toolchain`, `desktop-game-runtime`, `desktop-runtime-sample-game-scene-gpu-package`, `desktop-runtime-generated-material-shader-scaffold-package`, `desktop-runtime-generated-material-shader-scaffold-package-vulkan-strict`, `desktop-runtime-sample-game-vulkan-ui-atlas-metadata-package`, and `dev-windows-editor-game-module-driver-load-tests` (Windows-only; runs `tools/run-editor-game-module-driver-load-tests.ps1` with `-HostGateAcknowledgements windows-msvc-dev-editor-game-module-driver-ctest` before `Execute`). `-Mode DryRun` returns the reviewed argv plan and host gates without executing. `-Mode Execute` executes only the reviewed builder for that recipe and returns exit code, duration, stdout summary, and stderr summary. The runner does not evaluate arbitrary shell or raw manifest `command` strings, and free-form validation commands are unsupported. Selected D3D12/Vulkan package execution requires explicit `-HostGateAcknowledgements`.

## Test Policy

- Core behavior should be tested before implementation when local tools allow it.
- Tests should exercise public API behavior, not private implementation details.
- Rendering tests will later add golden/screenshot checks.
- Performance-sensitive changes will later add benchmarks with before/after numbers.
