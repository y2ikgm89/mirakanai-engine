# GameEngine Roadmap

## Principle

Build a clean C++23 engine with no early backward-compatibility burden. Prefer official platform guidance, explicit module boundaries, deterministic tests, and AI-readable contracts. Separate developer-facing immediate-mode tooling from production runtime UI: Dear ImGui is the current editor/debug shell, while long-term game UI and scalable editor surfaces should evolve toward a first-party retained-mode UI layer similar in role to Unreal Slate/UMG/Common UI or Unity UI Toolkit.

## Core-First MVP Closure

`core-first-mvp` is complete for the current MVP scope when the repository default build/test/agent contract is validated and the remaining work is explicitly classified as host-gated or next-phase. This is not a claim that GameEngine is a complete commercial game engine.

The closure plan is `docs/superpowers/plans/2026-05-01-core-first-mvp-closure.md`. Closure requires `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1`, `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/build.ps1`, agent/schema checks, and API-boundary verification where public API or backend interop surfaces are touched. The current MVP foundation includes the source-tree default validation lane, the optional SDL3 desktop game runtime lane, manifest-driven desktop runtime package metadata, generated desktop package scaffolds, cooked-scene package scaffolds, material/shader package scaffolds, and Windows D3D12 plus Windows/toolchain-gated Vulkan proofs behind private backend ownership and first-party diagnostics.

The following remain outside the MVP closure claim: Apple-hosted iOS simulator/device validation, device signing, backend-neutral Metal visible presentation, production material/shader graph authoring, live shader generation, broader authored/cooked packaged-game conventions, production text/font/IME/accessibility/image adapters, richer postprocess and production shadow systems, broad GPU skinning/backend compute morph parity beyond the reviewed D3D12 lanes, navmesh/crowd foundations, advanced physics controllers, dynamic-vs-dynamic TOI, rotational/2D CCD, persistent joint assets, vehicles/ragdolls, and broader authored collision assets, streaming codec integration, HRTF/DSP graphs, and crash/telemetry/profiling/trace workflows beyond the current diagnostics and GPU marker foundations.

## Desktop Runtime Productization

`docs/superpowers/plans/2026-05-01-desktop-runtime-productization.md` is complete as the first practical Windows desktop vertical slice after `core-first-mvp`: source-tree desktop runtime validation, the default desktop runtime package lane, and the selected `sample_desktop_runtime_game` package lane with installed config, cooked scene package files, target-specific shader artifacts, scene GPU binding reports, depth-aware postprocess status, and fixed 3x3 PCF directional-shadow smoke reports.

Optional vcpkg-backed desktop lanes now use explicit dependency bootstrap rather than configure-time restore: `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1` owns `vcpkg install` for `desktop-runtime`, `desktop-gui`, and `asset-importers`, while CMake presets consume `${sourceDir}/vcpkg_installed` with `VCPKG_MANIFEST_INSTALL=OFF`.

D3D12 is the primary verified Windows package lane for this slice. The current Windows host also validates the strict selected Vulkan package command, but Vulkan remains host/runtime/toolchain-gated for other environments. Apple/iOS/Metal productization, production renderer quality, material/shader graph authoring, live shader generation, broader authored/cooked packaged-game conventions, editor UX expansion, concrete runtime UI adapters, telemetry/crash reporting, allocator diagnostics, and GPU marker adapters remain next-phase work.
