## 100% Completion Definition

For this master plan, `100%` means **GameEngine 1.0 is ready as a desktop-first AI-operable production engine for small-to-medium 2D and 3D games**. A coding agent should be able to read the engine and game manifests, choose only supported recipes, author or update scenes/prefabs/materials/assets/gameplay code through reviewed surfaces, cook/package the result, run validation, diagnose failures, and hand the game to an operator without guessing.

This is not a Unity/Unreal completeness claim. It does not require a full commercial marketplace ecosystem, AAA renderer feature parity, visual scripting, network replication, cinematic tooling, or every platform backend at equal maturity. It does require that the supported 2D and 3D production paths be coherent, validated, documented, and machine-readable end to end.

The 1.0 completion claim is allowed only when all of these are true:

- AI-operable contracts, command surfaces, manifests, schemas, docs, skills, and validation recipes agree on the same supported capabilities.
- A generated 2D desktop package and a generated 3D desktop package can be produced, packaged, and validated through reviewed recipes without manual guesswork.
- The D3D12 primary Windows lane has package-visible evidence for the supported renderer, material, animation, UI, runtime package, and gameplay slices.
- Vulkan and Metal capabilities remain either validated through their required host/toolchain gates or explicitly excluded from ready claims.
- Runtime systems that normal generated games depend on have production-minimum paths: persisted input rebinding profiles, runtime UI text/image path, audio output/streaming baseline, physics controller/collision baseline, navigation/pathing baseline, behavior/AI baseline, diagnostics, and profiling.
- Editor productization covers the same first-party authoring model used by AI/headless tools: hierarchy, inspector, viewport, content/package review, asset/material preview, playtest handoff, and undo/redo coverage for supported workflows.
- Release and operations recipes are honest: desktop package release validation is ready, Android and Apple lanes report real host evidence or host-gated blockers, and crash/telemetry/profile/trace handoff rows distinguish ready artifacts, host gates, and unsupported backends.
- The composed `engine/agent/manifest.json` `unsupportedProductionGaps` rows, current-truth docs, and this ledger have no contradiction for the 1.0 ready surface.

## Official Practice Review Gates

Each child plan that touches the listed surface must include a short "official practice check" before implementation and record the source in its validation evidence. These checks are planning gates, not broad ready claims:

- **CMake / install / package:** keep project-wide presets in tracked `CMakePresets.json`, local choices in ignored `CMakeUserPresets.json`, target-based usage requirements, installed headers/modules, `install(EXPORT ...)`, `find_package(Mirakanai)` consumer validation, CTest, and CPack in the repository wrappers. C++ module work must keep `FILE_SET CXX_MODULES` install/export behavior explicit.
- **vcpkg / dependencies:** stay in manifest mode with `builtin-baseline`, additive manifest features, audited optional dependencies, and user/CI-local binary cache configuration. Dependency installation belongs to `tools/bootstrap-deps.ps1`; CMake configure must not restore or download packages.
- **SDL3 / platform adapters:** keep SDL3 in platform/runtime-host adapters, use installed CMake targets such as `SDL3::SDL3`, copy event-owned text before retention, enable text input only for the active text target/window, and keep audio on SDL3 stream/device adapters rather than moving SDL APIs into `MK_ui`, gameplay, or `engine/core`.
- **D3D12 / Vulkan / Metal:** require backend-private resource ownership, barriers/synchronization, queue/fence timelines, upload/staging retirement, debug names/markers, and validation-layer or host-tool evidence before promotion. D3D12 may be the Windows primary lane; Vulkan and Metal must not inherit D3D12 ready claims without their own strict host evidence.
- **Android / Apple release lanes:** Android work must use the GameActivity/Prefab/NDK path accepted by the Android docs and must not store signing keys or SDK caches in the repo. Apple/iOS/Metal readiness requires macOS/Xcode/Simulator or device evidence; Windows diagnostics can only record host-gated blockers for those lanes.
- **AI/operator command surfaces:** every convenience feature must be a reviewed dry-run/apply or reviewed execution surface with typed diagnostics, host-gate acknowledgement where needed, undo/remediation rows when it mutates content, and manifest/static-check synchronization.

Source anchors for these checks: CMake official docs for presets/install/export/C++ module file sets, Microsoft vcpkg docs for manifest mode/versioning/binary caching, SDL3 migration/API docs, Microsoft Direct3D 12 synchronization docs, Khronos Vulkan Guide/spec validation docs, Apple Metal best-practices/capability docs, and Android GameActivity docs. Use newer official docs when a child plan is authored after this ledger edit.

## Physics 1.0 Target

Physics 1.0 for this master plan means a production-minimum gameplay physics surface for the supported desktop-first generated 2D and 3D game recipes. It is not a claim of Unity/Unreal or standalone commercial physics-middleware parity. The ready claim should stay limited to deterministic first-party APIs, reviewed package/gameplay integration, and validation evidence that small-to-medium games can rely on without native handles, hidden middleware coupling, or unsupported backend assumptions.

The physics ready surface can be promoted only when focused child plans prove these capabilities through tests, manifest/docs updates, and package-visible or headless gameplay evidence:

- deterministic body integration, broadphase, contact generation, trigger reporting, contact resolution, collision filtering, raycasts, exact sphere casts, conservative controller movement, and in-memory authored collision rows remain green under default validation;
- first-party joints cover the production-minimum generated-game cases with deterministic constraints, diagnostics, and replay tests;
- exact 3D AABB/sphere/vertical-capsule shape sweeps cover current first-party 3D query primitives, including false-positive rejection beyond conservative bounds queries; any broader 2D exact-query or future primitive expansion needs its own focused evidence before joining the ready surface;
- CCD covers the accepted fast-moving-body cases without changing default deterministic replay behavior or exposing backend-native handles;
- scene/package collision authoring converts reviewed first-party collision rows into runtime-loadable package data with validation diagnostics, asset references, and generated-game smoke evidence;
- character/dynamic interaction policies define the accepted push, step, slope, grounded, and trigger behavior for generated 3D controllers without pretending to solve full platformer, vehicle, or character-action middleware scope;
- physics benchmark and determinism gates record stable solver/query budgets and replay evidence before any optional middleware adapter is promoted;
- optional Jolt or native backend work remains adapter-gated behind dependency/legal/vcpkg/notice updates, benchmark comparisons, fail-closed capability negotiation, and first-party API parity. A Jolt adapter is not required for the first-party ready claim unless a later child plan explicitly moves it into the 1.0 surface.

5. `physics-character-dynamic-interaction-policy-v1` (completed): defines generated-game push, step, slope, grounded, trigger, and dynamic-body interaction semantics as an explicit gameplay policy over the proven query/contact/CCD surface.
