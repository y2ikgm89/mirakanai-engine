# Production Completion v1 - AI Autonomous Game Creation

## AI Autonomous Game Creation Track (Codex / Claude Code)

The long-term AI-operable target is that Codex, Claude Code, and equivalent coding agents can understand this engine, choose supported recipes, create a game, author or update game-owned content, validate it, package it, and hand off evidence with minimal operator decisions. The autonomous game-creation lane is **game-surface only by default**: agents may generate and update game files, game code, game manifests, source assets, placeholder assets, package registration rows, validation evidence, and repo-owned operation commands. They must not modify engine internals, renderer/RHI/backend code, platform adapters, build-system contracts, toolchain policy, or shared agent surfaces unless the operator explicitly starts a separate engine-feature task. This lane must be implemented through reviewed engine/game command surfaces, machine-readable manifests, bounded subagents, and validation recipes rather than unconstrained shell or arbitrary file mutation.

Official agent-surface anchors:

- [OpenAI Codex](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan) is an AI coding agent that can work locally or in the cloud, navigate a repository, edit files, run commands/tests, review code, and delegate tasks in isolated environments for reviewable merge/pull-down workflows.
- [OpenAI Codex CLI](https://help.openai.com/en/articles/11096431-openai-codex-cli-getting-started) uses approval modes for read/edit/execute behavior; autonomous modes still need sandbox and repository-scoped validation expectations.
- [Claude Code memory](https://docs.anthropic.com/en/docs/claude-code/memory) uses project memory (`CLAUDE.md`) for team-shared architecture, commands, and workflow instructions.
- [Claude Code subagents](https://docs.anthropic.com/en/docs/claude-code/sub-agents) are project/user Markdown agents with purpose, tool permissions, and separate context windows; project agents take precedence and should be specialized.
- [Claude Code settings and permissions](https://docs.anthropic.com/en/docs/claude-code/settings) and [hooks](https://docs.anthropic.com/en/docs/claude-code/hooks) support project-scoped permissions and validation hooks, but hooks execute shell commands and require conservative security design.

AI autonomous game creation must follow these rules:

- **Manifest-first context:** agents start from `engine/agent/manifest.json`, `game.agent.json`, and `tools/agent-context.ps1 -ContextProfile Minimal|Standard`, not broad source-tree exploration. The manifest must expose supported game templates, validation recipes, host gates, package surfaces, command surfaces, unsupported claims, and remediation hints.
- **Game-owned mutation boundary:** autonomous game creation may write under `games/<game_name>/`, game-local source asset roots, game-local docs/evidence, and explicitly reviewed generated/runtime package registration rows. Engine directories (`engine/`, `editor/`, shared `tools/`, global schemas, shared skills/rules/subagents, CMake/vcpkg, and repository-wide validation policy) are read-only for this lane unless a separate engine-change plan is selected.
- **Developer handoff for missing engine capability:** if a requested game requires an unsupported engine feature, the game-creation agent must not implement it in `engine/`, weaken validation, or add game-specific engine shortcuts. It must emit a developer-facing handoff row with the requested capability, current supported workaround if any, blocked game feature, affected game files, desired public contract, validation evidence needed, and links to the relevant manifest unsupported claim. The developer then decides whether to implement a separate engine-feature task; the game-creation lane continues only after the manifest exposes the new supported capability.
- **Reviewed mutation surfaces:** agents must create or change game content through reviewed dry-run/apply tools, typed command rows, or documented file formats. Direct edits to generated scene/package/runtime files are allowed only when the file is explicitly listed as an AI-owned source artifact and has validation recipes.
- **Game design before generation:** generated games need a structured game design contract before code/content generation: gameplay family, target template, camera, input map, core loop, win/loss/restart, scene list, asset requests, reusable gameplay systems, game-specific rules, package targets, validation recipes, and quality gates.
- **Safe asset pipeline:** placeholder assets must be generated from first-party templates or reviewed synthetic/AI-generated assets with deterministic provenance, license status, package paths, and replacement instructions. Agents must not copy web/blog/sample/store assets without explicit license evidence and legal records.
- **Playtest and remediation loop:** every generated game needs a validation/playtest evidence loop that records recipe output, package smoke output, gameplay counters, failure diagnostics, and next-step remediation rows. Agents should repair by following remediation recipes, not by loosening validation or deleting evidence.
- **Bounded subagents:** Codex/Claude project subagents should be specialized by ownership: game designer/planner, gameplay builder, asset/package author, validation fixer, renderer auditor, and integration auditor. Read-only/auditor roles stay read-only; write-capable roles get narrow ownership and must not revert unrelated edits.
- **Approval and hook safety:** automated hooks or full-auto modes may run only repository-supported validation and static guards. They must avoid destructive cleanup, secret access, network dependency installation, signing, raw PR state changes, or arbitrary shell execution unless a documented approved surface owns the action.
- **Quality is explicit:** an AI-generated game is not complete merely because it compiles. It needs a small quality rubric: clear objective within 30 seconds, responsive controls, visible feedback, fail/restart path, deterministic package smoke, performance budget, and supported-host evidence.

Recommended AI game creation streams:

| Stream | Goal | Official-practice evidence gate | Non-goals until separately planned |
| --- | --- | --- | --- |
| `ai-game-design-spec-v1` | Add a structured game design document/schema that agents fill before generation. | Schema/static checks, examples for 2D and 3D templates, validation that required gameplay/package fields exist, and docs for Codex/Claude usage. | Free-form natural-language-only generation, visual scripting, market-ready design scoring. |
| `ai-game-generation-orchestrator-v1` | Add a reviewed orchestration recipe from design spec to `tools/new-game.ps1`, game manifest, game-owned source assets, scene/prefab/material/package rows, and validation recipes. | Dry-run/apply plan rows, deterministic game-owned file list, generated 2D/3D package smoke evidence, and no arbitrary shell. | Fully autonomous publishing, external asset downloads, engine-internal edits, unrestricted code generation outside the selected game surface. |
| `ai-safe-content-mutation-ledger-v1` | Make AI-owned game source files, generated files, reviewed command surfaces, forbidden engine/shared paths, and remediation actions machine-readable per game. | `game.agent.json` schema rows, static checks, docs/skills alignment, and validation failure if AI-owned mutation surfaces drift. | Letting agents hand-edit cooked/runtime package artifacts, hidden engine state, shared toolchain policy, or engine internals. |
| `ai-placeholder-asset-pipeline-v1` | Provide legal deterministic placeholder assets and replacement flows for sprites, meshes, materials, audio cues, UI atlas rows, and scene props. | Provenance rows, license status, package rows, hash/eol checks for runtime files, and package-visible placeholder evidence. | Scraping assets, unrecorded AI-generated third-party-looking assets, broad DCC pipeline automation. |
| `ai-generated-game-playtest-loop-v1` | Let agents run supported game validation/playtest recipes, collect evidence, classify failures, and propose fixes. | Recipe output summaries, gameplay counters, package smoke logs, screenshots only when a reviewed browser/desktop lane owns them, and remediation rows. | Subjective fun claims without rubric evidence, unbounded visual testing, native tool automation outside approved surfaces. |
| `ai-validation-remediation-recipes-v1` | Add machine-readable remediation for common generation failures: missing package files, invalid scene references, unsupported host gates, shader/toolchain gaps, gameplay counter failures, and asset/package drift. | Manifest remediation rows, tests for failure classification, docs/skills/subagent updates, and no validation weakening. | Auto-deleting user work, bypassing host gates, loosening static checks. |
| `ai-generated-game-quality-rubric-v1` | Define minimum generated-game quality gates beyond compile: objective clarity, controls, feedback, fail/restart, deterministic simulation, package evidence, and performance budget. | Headless/package assertions, sample rubric reports, 2D/3D examples, and explicit unsupported rows for unmet qualities. | Commercial quality scoring, user-study replacement, store-certification readiness. |
| `ai-codex-claude-agent-surface-v1` | Keep Codex and Claude Code project surfaces aligned so autonomous game generation uses game-owned write scopes, reviewed tools, and validation recipes. | `check-agents`, `check-ai-integration`, project subagent/tool-permission audits, and official OpenAI/Anthropic docs citations when behavior changes. | Broad permission grants, hidden local-only instructions, engine-internal writes from game-generation agents, relying on untracked user settings for production workflows. |
| `ai-engine-capability-handoff-v1` | Add a typed handoff surface for game-creation agents to report missing engine capabilities to the developer without editing engine internals. | `game.agent.json` or evidence-row schema for blocked features, examples, validation that handoff rows include requested capability/current workaround/blocked game feature/desired public contract/evidence needed, and docs/skills/subagent alignment. | Game agents patching engine internals, silent feature workarounds in shared code, validation weakening, automatic promotion of unsupported capabilities. |

When this track is selected, create or update a dated capability/milestone plan. The plan must explicitly say which agent surface it improves, which tools or commands the agent may use, which **game-owned** files it may mutate, which engine/shared paths are read-only, how failures are remediated, and which validation evidence proves that Codex and Claude Code can follow the flow without guessing. If the agent discovers a missing engine capability, it must stop at a typed developer handoff/remediation row; it must not patch engine internals as part of autonomous game creation. Only the developer, through a separate engine-feature task and normal engine validation, may change engine internals and expose the new capability back through the manifest.


AI generation must separate reusable engine capability from game-specific content. Game-specific balance, enemy tables, dialogue, maps, scoring, progression curves, and presentation stay in games/<game_name>/; reusable gaps become developer-owned engine backlog rows.
