## General-Purpose / High-Freedom Game Creation Track (post-1.0 / 1.x)

The Engine 1.0 target is intentionally scoped to AI-operable small-to-medium 2D/3D desktop games. Higher-freedom games such as sandbox, open-ended content-rich adventure, construction/crafting, larger exploration, simulation-heavy, procedural, or mod-friendly games require additional first-party gameplay systems and world-management contracts. Track those capabilities as **post-1.0 / 1.x general-purpose game creation streams** unless a future dated plan explicitly promotes a slice into the 1.0 ready definition.

High-freedom game support must preserve the same AI-operable rules and remain game-surface scoped by default:

- **Data-driven but reviewed:** quests, dialogue, inventory, interaction rules, world regions, save data, procedural generation seeds, and scripts must be first-party documents or reviewed command rows with schema/static validation. Agents must not create hidden runtime-only state that cannot be inspected or repaired.
- **No implicit engine edits:** high-freedom game creation may generate or edit game-owned source, data, assets, and package registration. If a requested gameplay family needs an engine feature that is not supported by the current manifest, the game-creation agent records a developer handoff/remediation row and waits for the developer to implement a separate engine-feature task instead of modifying engine internals during game generation.
- **Stable identity and persistence:** every persistent entity, region, quest state, inventory item, dialogue flag, procedural chunk, and authored/generated asset reference needs stable identity, versioning, migration diagnostics, and save/load validation before ready claims.
- **World scale is explicit:** large maps require region/chunk ownership, streaming boundaries, resource budgets, nav/physics partitioning, LOD/culling evidence, deterministic missing-region diagnostics, and package-visible counters. Do not claim open-world readiness from single-scene package proof.
- **Scripting is sandboxed and optional:** Lua, WASM, or other scripting runtimes can only enter through an optional adapter plan with dependency/legal records, deterministic host API boundaries, execution budgets, capability restrictions, and replay/validation evidence. First-party data/command surfaces remain the ready baseline.
- **Procedural generation is reproducible:** AI or runtime procedural generation needs explicit seeds, inputs, versioned generator ids, deterministic output summaries, validation rows, and remediation paths. Agents must not depend on unrecorded randomness or external services for package-ready content.
- **Simulation scope is bounded:** economy, crafting, NPC schedules, factions, quests, and world state need budgets and deterministic tick/update ownership. Long-running or background simulation must not bypass validation recipes or package evidence.
- **2D/3D evidence remains separate:** shared systems may be engine-level, but generated 2D and generated 3D package lanes must prove their own world/gameplay evidence.

Recommended high-freedom 1.x streams:

As of PR #175 / merge commit `f05136e2`, `ai-gameplay-authoring-tools-v1` implementation is completed and the selection gate returned to the production-completion master plan. `post-1-0-capability-program-v1` now selects the next post-1.0 / 1.x developer-owned milestone, starting in the physics/navigation/AI track with `physics-constraints-and-joints-v1`. High-freedom streams remain sequenced follow-ons in the same milestone until a later phase selects one directly.

When this track is selected, prefer one dated milestone plan that states the target game freedom level, supported gameplay families, AI-operable authoring surface, persistence model, validation recipes, and explicit non-goals. Do not broaden the 1.0 ready claim or add optional scripting/middleware dependencies without dependency, legal, manifest, schema, skills, rules/settings, subagent, and validation updates.


General-purpose boundary: this track should increase the number of safe game shapes the engine can support without coupling engine APIs to any one genre vocabulary. Prefer capability names such as persistence, interaction, inventory, quest state, navigation, behavior, placement, procedural generation, and validation.
