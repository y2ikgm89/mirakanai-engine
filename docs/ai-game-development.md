# AI-Driven Game Development

GameEngine should let Codex, Claude Code, and similar agents build games through an explicit workflow instead of guessing project structure.

## Production Loop Recipes

Before creating or modifying a game, inspect `engine/agent/manifest.json.aiOperableProductionLoop` or the top-level `productionLoop` section from `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1`. Select exactly one recipe id, then mirror its supported subset in `game.agent.json`.

Current recipe ids:

- `headless-gameplay`: ready default for deterministic C++ gameplay and renderer-intent tests.
- `ai-navigation-headless`: ready headless behavior-tree and navigation composition.
- `runtime-ui-headless`: ready first-party `MK_ui` HUD/menu model recipe.
- `desktop-runtime-config-package`: host-gated desktop runtime config package scaffold.
- `desktop-runtime-cooked-scene-package`: host-gated desktop runtime cooked scene package scaffold.
- `desktop-runtime-material-shader-package`: host-gated desktop runtime material/shader package scaffold with D3D12 as the primary Windows lane and Vulkan behind strict toolchain/runtime gates.
- `2d-playable-source-tree`: ready source-tree 2D playable foundation for deterministic input actions, orthographic scene camera validation, sprite renderer intent, first-party HUD submission, device-independent audio cue rendering, and `NullRenderer` proof.
- `2d-desktop-runtime-package`: host-gated 2D desktop runtime package proof with cooked first-party 2D scene, sprite texture/material, cooked audio payload, HUD submission, SDL3 desktop host, installed package validation, deterministic `NullRenderer` fallback, and selected D3D12 native 2D sprite package smoke through `--require-native-2d-sprites`.
- `3d-playable-desktop-package`: host-gated sample and generated 3D desktop package foundation using cooked first-party mesh/material/animation/scene payloads, primary camera/controller status, selected public gameplay systems composition counters, selected safe-point package streaming counters, D3D12 package validation, and strict Vulkan gated validation.
- `renderer-package-quality-gates`: host-gated `sample_desktop_runtime_game` package proof that evaluates existing scene GPU, postprocess, depth-input, directional-shadow filtering, framegraph pass-count, and first-party renderer counter fields through `evaluate_sdl_desktop_presentation_quality_gate`. The package smoke uses `--require-renderer-quality-gates` and expects `renderer_quality_status=ready`, `renderer_quality_ready=1`, `renderer_quality_diagnostics=0`, `renderer_quality_expected_framegraph_passes=3`, and `renderer_quality_framegraph_execution_budget_ok=1`. This is not GPU timestamp telemetry, backend-native stats, cross-backend performance parity, Metal readiness, or general production renderer quality.
- `native-gpu-runtime-ui-overlay`: host-gated sample desktop package proof that routes first-party `MK_ui_renderer` box/image/glyph sprites through public `mirakana::IRenderer::draw_sprite` into a renderer-owned final native GPU overlay. This is not production text shaping, font loading/rasterization, real font-driven glyph atlas generation, runtime image decoding, renderer upload policy, IME, or OS accessibility readiness.
- `native-ui-textured-sprite-atlas`: host-gated sample desktop package proof that binds a cooked texture as a single atlas page for a first-party `MK_ui_renderer` image sprite through public UI/renderer contracts. This is not source image decoding, production atlas packing, production text/font/accessibility, Metal readiness, or broad UI renderer quality.
- `native-ui-atlas-package-metadata`: host-gated sample desktop package proof that builds `UiRendererImagePalette` image bindings from first-party cooked UI atlas package metadata before using the same renderer-owned native textured overlay path. The separate `MK_tools` packed atlas bridge can now author those metadata rows from already validated RGBA8 `ImageDecodeResult` inputs, but this recipe is still not runtime source PNG/JPEG image decoding, renderer texture upload, production text/font/accessibility, Metal readiness, or broad UI renderer quality.
- `future-3d-playable-vertical-slice`: planned broader 3D production recipe. Do not select it for normal generated games yet.

`planned`, `blocked`, or host-gated recipes must not be treated as ready unless the task is explicitly implementing that engine capability or the required host validation recipe has passed. Generated games must not bypass these recipes by exposing SDL3, Win32, D3D12, Vulkan, Metal, Dear ImGui, RHI backend handles, source asset parsers, runtime shader compilers, or game-local UI/physics/rendering systems.

## AI Command Surfaces

`aiCommandSurfaces` describes which AI-facing mutations may be dry-run or applied. Each descriptor has `schemaVersion`, `requestModes`, `requestShape`, `resultShape`, `requiredModules`, `capabilityGates`, `hostGates`, `validationRecipes`, `unsupportedGapIds`, and a placeholder `undoToken`. Agents must treat these rows as capability gates before editing game files, scene/prefab/material/package data, or manifests.

`register-runtime-package-files`, `register-source-asset`, `cook-registered-source-assets`, `migrate-scene-v2-runtime-package`, `update-ui-atlas-metadata-package`, `create-material-instance`, `create-material-from-graph`, `update-scene-package`, `create-scene`, `add-scene-node`, `add-or-update-component`, `create-prefab`, and `instantiate-prefab` currently have ready mutation modes. Runtime package registration is limited to reviewed game-relative `runtimePackageFiles` appends through `tools/register-runtime-package-files.ps1` or the matching editor-core package registration model; source asset registration is limited to `GameEngine.SourceAssetRegistry.v1` rows plus deterministic `GameEngine.AssetIdentity.v2` and import-metadata projections through `mirakana::plan_source_asset_registration` and `mirakana::apply_source_asset_registration`; `cook-registered-source-assets` is limited to explicitly selected source registry rows through `mirakana::plan_registered_source_asset_cook_package` and `mirakana::apply_registered_source_asset_cook_package`, reusing `build_asset_import_plan`, `execute_asset_import_plan`, and `assemble_asset_cooked_package` for deterministic cooked artifacts plus `.geindex` updates; `migrate-scene-v2-runtime-package` is limited to the reviewed bridge from authored `GameEngine.Scene.v2` plus `GameEngine.SourceAssetRegistry.v1` rows into the existing `GameEngine.Scene.v1` `.scene` plus `.geindex` package update surface through `mirakana::plan_scene_v2_runtime_package_migration` and `mirakana::apply_scene_v2_runtime_package_migration`; UI atlas and material instance updates are limited to their first-party `.uiatlas` / `.material` plus matching `.geindex` rows; material graph package binding is limited to explicit `GameEngine.MaterialGraph.v1` text/path inputs lowered through `mirakana::plan_material_graph_package_update` / `mirakana::apply_material_graph_package_update` into runtime `GameEngine.Material.v1` `.material` rows plus matching `.geindex` texture dependencies; `update-scene-package` remains the narrow first-party `GameEngine.Scene.v1` cooked package update through `mirakana::plan_scene_package_update` and `mirakana::apply_scene_package_update`; the Scene/Prefab v2 commands are authored-source `.scene` / `.prefab` updates through `mirakana::plan_scene_prefab_authoring` and `mirakana::apply_scene_prefab_authoring`.

`validate-runtime-scene-package` has ready dry-run/execute modes for non-mutating runtime scene package validation through `mirakana::plan_runtime_scene_package_validation` and `mirakana::execute_runtime_scene_package_validation`. It checks safe `.geindex` package paths, explicit `AssetKeyV2` scene keys, package load failures, `mirakana::runtime_scene::instantiate_runtime_scene` diagnostics, and structured reference summaries without writing files, cooking packages, parsing source assets at runtime, creating renderer/RHI residency, streaming packages, running editor or desktop UI, exposing native/RHI handles, or supporting arbitrary shell/free-form edits. Optional `contentRoot` is prepended to cooked package entry paths by the runtime loader; omit it when package index entries already include `runtime/...`.

For host/editor review of existing package files before validation or safe-point selection, use `mirakana::runtime::discover_runtime_package_indexes_v2` over a caller-owned `IFileSystem` root. Discovery returns sorted reviewed `.geindex` candidate rows with caller-provided `content_root` preserved for later explicit load requests; it does not read index contents, infer package roots, load packages, mutate manifests, mount resident packages, stream packages, or expose renderer/RHI/native handles.

Shader hot-reload and pipeline-cache planning is a `MK_tools` workflow over already reviewed shader compile requests, not a game-code mutation surface. Use `mirakana::build_shader_pipeline_cache_plan` to summarize selected `ShaderCompileExecutionRequest` rows into compile-required, pipeline-recreation, cache-entry-current, and cache-index-repair diagnostics, and `mirakana::reconcile_shader_pipeline_cache_index` only to rebuild selected first-party `ShaderArtifactCacheIndex` files from current artifacts plus `.geprovenance`. This does not execute shader compilers, validators, package scripts, native PSO/Vulkan/Metal cache blob APIs, live shader generation, renderer/RHI residency, public native handles, or package streaming.

Cooked-scene game manifests should expose `runtimeSceneValidationTargets` so agents can choose those `validate-runtime-scene-package` inputs without scraping smoke-test commands. Each row is descriptor-only and contains `id`, `packageIndexPath`, `sceneAssetKey`, optional `contentRoot`, `validateAssetReferences`, and `requireUniqueNodeNames`. Paths are game-relative, `packageIndexPath` must point at a `.geindex` already listed in `runtimePackageFiles`, `contentRoot` must not duplicate prefixes already present in cooked package entry paths, and rows must not contain command strings, source asset files, renderer/RHI/native handle fields, or broad package-cooking claims.

Editor Playtest Package Review Loop v1 is the reviewed pre-playtest sequence for generated or sample packaged games: review editor package candidates, apply only reviewed `runtimePackageFiles` additions, select the runtime scene package validation input from `game.agent.json.runtimeSceneValidationTargets`, run `validate-runtime-scene-package`, then run the selected host-gated desktop smoke recipe through `run-validation-recipe` or the package validation scripts. Keep package registration, runtime scene validation, and desktop smoke as separate steps; do not turn this into broad package cooking, runtime source parsing, renderer/RHI residency, package streaming, editor productization, Metal readiness, native handle exposure, or a general renderer quality claim.

Editor AI Package Authoring Diagnostics v1 is the diagnostics-only review model for AI/editor package authoring state. It summarizes editor package candidate review, manifest descriptor rows such as `runtimeSceneValidationTargets` and `prefabScenePackageAuthoringTargets`, caller-supplied runtime package payload diagnostics, validation recipe status rows, and host-gated desktop smoke preflight through `EditorAiPackageAuthoringDiagnosticsModel`. It does not mutate `game.agent.json`, execute `run-validation-recipe`, run package scripts, parse source assets at runtime, expose renderer/RHI handles, productize play-in-editor, make Metal ready, or claim renderer quality.

Editor AI Validation Recipe Preflight v1 is the preflight-only model for selected validation recipes. `EditorAiValidationRecipePreflightModel` maps manifest `validationRecipes` and selected descriptor recipe ids to caller-supplied `run-validation-recipe dry-run` plans, host gates, missing recipe ids, blocked reasons, and unsupported claim diagnostics. It treats dry-run plans as data only: editor core does not execute arbitrary shell, evaluate raw manifest command strings, run package script execution, or call `run-validation-recipe`. Host gates remain explicit and execution stays in the reviewed runner or external operator workflow.

Editor AI Playtest Readiness Report v1 is the read-only aggregation model before operator validation. `EditorAiPlaytestReadinessReportModel` combines `EditorAiPackageAuthoringDiagnosticsModel` and `EditorAiValidationRecipePreflightModel` rows, reports package-authoring and validation-preflight status, preserves host gates, and emits blocking diagnostics for mutation, validation execution, arbitrary shell, free-form manifest edits, package script execution, play-in-editor productization, renderer/RHI handle exposure, Metal readiness, and renderer quality claims. It does not run commands or mutate manifests; operator validation remains an external reviewed workflow.

Editor AI Playtest Operator Handoff v1 is the deterministic external operator handoff after readiness. `EditorAiPlaytestOperatorHandoffModel` consumes `EditorAiPlaytestReadinessReportModel` and `EditorAiValidationRecipePreflightModel`, then lists selected recipe ids, reviewed command display text, optional argv plan data, host gates, blocked reasons, and the readiness dependency that must be satisfied before an operator runs anything. It treats `run-validation-recipe` dry-run output as data only; editor core still does not execute arbitrary shell, evaluate raw manifest command strings, run validation execution, call package scripts, mutate manifests, productize play-in-editor, expose renderer/RHI handles, make Metal ready, or claim renderer quality.

Editor AI Playtest Evidence Summary v1 is the read-only post-handoff evidence model. `EditorAiPlaytestEvidenceSummaryModel` consumes `EditorAiPlaytestOperatorHandoffModel` and externally supplied validation evidence rows such as `run-validation-recipe execute` result data, then summarizes each recipe as `passed`, `failed`, `blocked`, `host_gated`, or `missing` with exit code or summary text, host gates, blockers, and readiness dependency. Evidence rows are data only: editor core does not execute validation commands, call package scripts, evaluate raw/free-form command text, mutate manifests, productize play-in-editor, expose renderer/RHI handles, make Metal ready, or claim renderer quality.

Editor AI Evidence Import Review v1 is the deterministic paste/review layer for externally supplied evidence. `EditorAiPlaytestEvidenceImportModel`, `EditorAiPlaytestEvidenceImportReviewRow`, `make_editor_ai_playtest_evidence_import_model`, and `make_editor_ai_playtest_evidence_import_ui_model` parse `GameEngine.EditorAiPlaytestEvidence.v1` text, validate recipe ids/status/exit codes/host gates/blockers, expose retained `ai_evidence_import` ids, and produce imported evidence rows only when the input stays external, non-mutating, and non-executing. Editor AI Reviewed Validation Execution v1 plus Host-Gated Validation Execution Ack v1 is the narrow local execution path for reviewed recipes. `EditorAiReviewedValidationExecutionModel`, `EditorAiReviewedValidationExecutionDesc`, `make_editor_ai_reviewed_validation_execution_plan`, and `make_ai_reviewed_validation_execution_ui_model` validate an existing operator handoff argv row, rewrite `-Mode DryRun` to `-Mode Execute`, and return a safe `mirakana::ProcessCommand` for the optional editor shell to run. Editor AI Playtest Remediation Queue v1 is the read-only post-evidence remediation model. `EditorAiPlaytestRemediationQueueModel` consumes only `EditorAiPlaytestEvidenceSummaryModel` rows, omits passed evidence, and classifies failed, blocked, missing, and host-gated evidence into remediation rows with recipe id, evidence status, host gates, blockers, readiness dependency, remediation category, and next-action text. Editor AI Playtest Operator Workflow v1 is the consolidated read-only loop: package diagnostics -> validation recipe preflight -> readiness -> operator handoff -> evidence summary -> remediation queue -> remediation handoff. `EditorAiPlaytestOperatorWorkflowReportModel` is the structured report surface for operators and agents: it reuses the existing models and reports the operator workflow stage, source model, source row count, source row ids, status, host gates, blockers, and diagnostic text. ## AI Workflow

Agents should follow this sequence:

1. Run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1`.
2. Read `engine/agent/manifest.json.aiOperableProductionLoop`.
3. Select a recipe id and verify its `status`, `validationRecipes`, `hostGates`, `unsupportedClaims`, and `followUpCapability`.
4. Inspect `aiCommandSurfaces` before making manifest, package, scene, prefab, material, or asset changes; use only ready request modes, and report planned/blocked command surfaces as unsupported diagnostics.
5. Read the target game's `game.agent.json`.

## Manifest Readiness Contract

`engine/agent/manifest.json` exposes:

- `runtimeBackendReadiness`: validated, optional, blocked, and planned backend status for graphics, audio, UI, physics, and platform lanes.
- `importerCapabilities`: first-party source document and cooked artifact support plus optional `MK_tools` external importer adapters.
- `packagingTargets`: source-tree, desktop GUI, desktop game runtime, release package, Android, and iOS target status, including optional GUI, focused desktop runtime smoke, and toolchain-gated mobile templates.
- `validationRecipes`: repository commands that prove a selected capability.
- `aiOperableProductionLoop`: recipe ids, AI command descriptor surfaces, authoring/package surfaces, unsupported production gaps, host gates, and validation recipe mapping for generated-game and engine-production work.

Each `games/<game_name>/game.agent.json` must mirror the selected subset through `backendReadiness`, `importerRequirements`, `packagingTargets`, optional `runtimePackageFiles`, and `validationRecipes`. Do not select a backend, importer, or package target whose status is only `planned`, `blocked`, or not validated for the chosen host unless the task is explicitly implementing that engine capability. A game that selects `desktop-runtime-release` must also select `desktop-game-runtime`, be registered through `MK_add_desktop_runtime_game` with a matching `GAME_MANIFEST`, declare finite smoke metadata, declare runtime package files in `runtimePackageFiles` and register them with `PACKAGE_FILES_FROM_MANIFEST` unless a literal `PACKAGE_FILES` registration intentionally mirrors the manifest, and include a package validation recipe for the selected target.
