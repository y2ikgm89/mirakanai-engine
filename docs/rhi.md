# Renderer Hardware Interface

## Current Scope

`MK_rhi` is the low-level graphics API boundary. It is intentionally separate from `MK_renderer`:

- `MK_renderer`: high-level renderer-facing contracts.
- `MK_rhi`: backend capability/probe contracts, Renderer/RHI Resource Foundation v1 through `RhiResourceLifetimeRegistry` resource ids, owner labels, debug names, generation-checked handles, deferred-release records, frame-indexed retirement, diagnostics, marker-style lifetime events, NullRHI lifetime wiring, D3D12/Vulkan deferred native teardown, GPU debug marker/timestamp-frequency rows, read-only memory diagnostics, Upload/Staging Foundation v1 through foundation-only `RhiUploadStagingPlan` staging allocation rows, buffer/texture copy rows, submitted `FenceValue` tracking, completed-fence retirement, and `RhiUploadDiagnosticCode::stale_generation` same-id/wrong-generation checks, device, resources, samplers, swapchains, explicit swapchain frame acquire/release, swapchain resize, shaders, descriptor set layouts, descriptor sets, upload-buffer writes, copy/upload/readback commands with explicit texture footprints, CPU buffer readback, transient resource leases, presentation ownership, graphics pipelines with explicit vertex input layouts/attributes plus backend-neutral depth-test/write/compare state, render passes with optional first-party depth attachments, vertex/index buffer binding, direct and indexed draw calls, transitions, explicit texture aliasing barrier intent, submission, fence semantics, and backend-neutral fence lifecycle stats.
- backend modules: future Direct3D 12, Vulkan, and Metal implementations.

The API-independent backend contract includes `BackendCapabilityProfile`, `BackendProbePlan`, and `BackendProbeResult`. These describe host support, shader artifact format (`dxil`, `spirv`, `metallib`, or none), native presentation features, and the deterministic bootstrap steps a backend must complete before the editor can call it available. Vulkan probes are modeled around runtime loading, physical device enumeration, queue family inspection, logical device creation, and shader artifact verification. Metal probes are modeled around runtime loading, default device creation, command queue creation, and Metal library verification.

The current backend is `NullRhiDevice`. It does not create GPU objects. It validates resource, sampler, descriptor, upload-buffer write, copy footprint, CPU buffer readback, transient lease, presentation, explicit swapchain frame acquire/release, swapchain resize, pipeline, vertex input layout/attribute, vertex buffer, index buffer, indexed draw, and RHI Depth Attachment Contract v0 descriptions. Optional depth render-pass attachments must be owned by the device, include `TextureUsage::depth_stencil`, use `Format::depth24_stencil8`, match the color target extent, be in `ResourceState::depth_write`, and provide finite clear-depth values in `[0, 1]`; depth textures may additionally include `TextureUsage::shader_resource` for sampled-depth use after an explicit `depth_write` to `shader_read` transition, but `NullRhiDevice` rejects using a shader-read depth texture as a depth attachment until it transitions back. Bound graphics pipelines must match the active color and depth formats. `NullRhiDevice` rejects unknown handles, incompatible descriptor set bindings, invalid write/copy ranges/usages, invalid readback ranges/usages, invalid vertex/index buffer ranges/usages, invalid sampler descriptors, invalid swapchains, invalid swapchain frame sequencing, released transient resources, and shader stage mismatches, records command list and render pass lifecycle, stores written upload-buffer bytes for deterministic copy/readback tests, counts write/copy/read/draw/indexed-draw/present/resize/frame/transient/sampler intent, tracks fence waits, wait failures, last submitted fence values, and last completed fence values, and signals fences immediately.

## Backend Model

The RHI vocabulary follows modern explicit graphics APIs:

- command lists are recorded before submission
- work is submitted to command queues
- resource state transitions are explicit
- graphics pipeline descriptions own explicit vertex input layouts/attributes before backend PSO creation
- buffer copies and buffer/texture transfers are explicit commands recorded outside render passes
- buffer/texture transfers use `BufferTextureCopyRegion`; `buffer_row_length` and `buffer_image_height` follow the common zero-means-tight convention used by modern copy APIs
- `buffer_texture_copy_required_bytes` computes the required host/staging buffer footprint and rejects invalid row or image pitches before backend code is written
- CPU readback is explicit through `IRhiDevice::read_buffer` and requires a buffer with `copy_destination` usage
- render pass clear values are explicit through `ClearColorValue` on `RenderPassColorAttachment`
- transient buffers and textures are acquired through explicit leases and become invalid after `release_transient`
- descriptor set layouts describe shader-visible resource tables before pipeline layout creation
- descriptor sets are allocated from a descriptor set layout, updated with typed buffer/texture resources, and bound against a compatible pipeline layout
- swapchain resize is explicit through `IRhiDevice::resize_swapchain` and is rejected while an acquired or rendered frame is pending
- swapchain frame ownership is explicit through `IRhiDevice::acquire_swapchain_frame` and `IRhiDevice::release_swapchain_frame`; render passes and presents use `SwapchainFrameHandle`, not the persistent `SwapchainHandle`
- swapchain presentation is an explicit graphics command through `SwapchainFrameHandle`, not an implicit side effect of ending a render pass; a frame must complete a render pass before `present`
- backend-neutral swapchains currently accept one acquired frame at a time; a second acquire is rejected until the prior frame is manually released or presented and submitted
- render passes bracket graphics pipeline binding and draw submission
- fence values represent submitted GPU work
- `RhiStats` exposes fence wait counts, wait failures, last submitted fence values, and last completed fence values so fence lifecycle coverage can be asserted across `NullRhiDevice` and native backends

This maps directly onto Direct3D 12 command queues/lists/fences plus root signature descriptor tables, resource barriers, future placed/aliased resources, and copy commands; Vulkan command buffers/queues/synchronization plus descriptor set layouts, buffer/image copy regions, and transient attachments; and Metal command buffers/queues plus argument tables, argument buffers, heaps, and blit passes.

## Backend Assumptions

- Game code should not depend on API-specific handles.
- Native handles are backend implementation details unless an explicit interop design is accepted later.
- Direct3D 12 is the Windows-specific backend path.
- Backend capability/probe state is API-independent and lives in `mirakana/rhi/backend_capabilities.hpp`; native backends must update this contract before editor availability or AI agent manifests claim support.
- `ShaderDesc::bytecode` is optional for GPU-free validation backends such as `NullRhiDevice`; real shader-owning backends require a non-null bytecode pointer and non-zero `bytecode_size`.
- Shader compile command planning is backend-aware but execution is not implicit. Commands are represented as executable-plus-argument vectors, not shell strings. The current runner validation allowlists compiler executables `dxc`, `metal`, and `metallib`, rejects unsafe output paths, records dry-run invocations, captures compiler stdout/stderr in `ShaderToolRunResult`, and can convert shader includes into asset dependency graph edges. Platform process execution is available through `mirakana::ProcessCommand` and `mirakana::Win32ProcessRunner`; `mirakana::ShaderToolProcessRunner` maps shader commands to that runner, supports reviewed executable overrides for PATH names, project-relative paths, or absolute installed SDK/tool paths, and enforces artifact output-root policy before invocation. `mirakana::discover_shader_tools` finds deterministic `dxc`, `spirv-val`, `metal`, and `metallib` descriptors with an explicit `dxc.spirv-codegen` capability marker for deterministic toolchain roots, and `mirakana::evaluate_shader_toolchain_readiness` reports D3D12 DXIL, Vulkan SPIR-V, DXC SPIR-V CodeGen, and Metal library readiness before backend promotion. `tools/check-shader-toolchain.ps1` additionally reads Process/User/Machine environment state, accepts `MK_SHADER_TOOLCHAIN_ROOT`, `VULKAN_SDK`/`VK_SDK_PATH`, vcpkg tool roots, and PATH tools before falling back to Windows SDK DXC, then runs a real `dxc -spirv -fspv-target-env=vulkan1.3` probe so Windows SDK DXC builds that expose `-spirv` but lack compiled SPIR-V CodeGen are reported as Vulkan blockers. `mirakana::make_spirv_shader_validation_command`, `mirakana::ShaderArtifactValidationProcessRunner`, and `mirakana::execute_shader_artifact_validation_action` provide a reviewed `spirv-val` path for compiled SPIR-V artifacts. `mirakana::build_shader_artifact_provenance`, `mirakana::is_shader_artifact_cache_valid`, `mirakana::ShaderArtifactCacheIndex`, and `mirakana::execute_shader_compile_action` record compiler/tool/input fingerprints, skip valid cached artifacts, write `.geprovenance` files, create deterministic artifact markers for tests, and require a real artifact when marker writes are disabled for compiler-backed runs. `mirakana::build_shader_pipeline_cache_plan` batches selected compile requests into deterministic hot-reload, compile-required, pipeline-recreation, cache-entry-current, and cache-index-repair rows; - Descriptor bindings are typed and stage-visible so D3D12 root signatures can stay small, Vulkan set layouts can preserve compatibility, and Metal can choose per-stage argument table binding or argument buffers.
Transient leases are exposed through first-party handles. The D3D12 `IRhiDevice` bridge backs single transient texture leases with backend-private one-heap-per-lease placed resources, invalidates the public resource handle on release, and retires the placed heap/resource when the deferred lifetime fence is complete. Backend-neutral alias-group texture leases now return one lease with distinct first-party texture handles per alias-group resource; D3D12 backs multi-resource groups with backend-private same-heap same-offset placed textures and explicit non-null aliasing-barrier proof. The Frame Graph RHI texture executor can automatically insert aliasing barriers from transient texture lifetimes, while Vulkan/Metal alias memory, data inheritance/content preservation, and native handles remain future allocator/executor work.
- Synchronization must be explicit and testable at the RHI boundary before a real GPU backend is added.
