# Engine Essential Gap Analysis

## Goal

Assess whether GameEngine has the essential capabilities expected from a usable C++ game engine, identify missing production features, and set a clean no-backward-compatibility implementation order.

## Verdict

The engine has a broad and coherent foundation for prototyping, AI-generated sample games, headless validation, package validation, and early desktop/mobile runtime work. It is not yet production-complete as a game engine.

These are production/commercial gaps and next-phase candidates. They are not blockers for closing `core-first-mvp` as the verified MVP scope described in `docs/superpowers/plans/2026-05-01-core-first-mvp-closure.md`.

The current foundation is strongest in architecture, C++23 policy, validation, RHI contracts, D3D12 proof paths, Vulkan/Metal scaffolding, asset identity/cooking plans, deterministic gameplay foundations, runtime UI contracts, editor-core models, Android packaging, and AI-readable manifests.

The main gap is not one missing module. The gap is the production loop that turns authored content into a shippable game:

1. Cooked assets with real payloads and runtime loading.
2. GPU material/texture/mesh binding across visible backends.
3. Editor authoring that edits those assets and scenes directly.
4. Runtime services that games require every session: save data, settings, localization, input actions, profiling diagnostics, and resource lifetime.
5. Editor-independent desktop runtime shell support so `games/<game_name>` can run in a window without borrowing the editor shell.
6. Platform store-quality validation for Apple and release distribution.

## Classification

- **P0:** Required before calling the engine production-usable for most games.
- **P1:** Required for a serious editor/gameplay workflow but can follow P0.
- **P2:** Genre, scale, or platform dependent.
- **Gated:** Requires host SDK, hardware, platform signing, or an accepted dependency audit.

## P0 Production Gaps

### 1. Runtime Asset Payloads And Loading

Current asset work records identity, metadata, dependency edges, source documents, cooked package indexes, and deterministic recook decisions. The engine still needs runtime-loadable cooked payloads that game code can consume without tools or source formats.

Recommended clean implementation:

- Add a runtime asset package reader that mounts cooked packages through `IFileSystem`.
- Add stable runtime handles for texture, mesh, audio, material, and scene payloads.
- Add payload version fields and strict migration rejection for unsupported versions.
- Keep source importers in `MK_tools`; runtime only reads cooked payloads.
- Add tests for invalid package versions, missing dependencies, hash mismatch, and safe replacement.

### 2. Production Material, Texture, And Mesh Binding

The engine has material data and explicit binding metadata, but production GPU binding is not complete. A real game needs textures, meshes, material parameters, descriptor updates, and shader layouts to meet at render time.

Recommended clean implementation:

- Promote material binding metadata into renderer/RHI descriptor layout creation.
- Build on the implemented backend-neutral mesh vertex/index payload upload, derived position-only and lit tangent-space cooked mesh layout contracts, renderer-visible mesh/material command binding, public sampler descriptor contract, material factor uniform upload, shader reflection metadata, scene GPU binding palette, runtime package-to-GPU-palette bridge, native D3D12/Vulkan sampler descriptor mapping, visible texture-sampling backend proofs, packaged SDL3 D3D12/Vulkan scene GPU injection, SDL3 Vulkan game-window presentation, Lit Material v0 sample proof, Shadow Map Foundation v0 planning contract, Depth Sampling Contract v0 with D3D12 readback plus Vulkan create/barrier/descriptor coverage and environment-gated visible sampled-depth readback proof, directional shadow receiver readback proof, selected package-visible depth-aware postprocess smoke for `sample_desktop_runtime_game`, and selected package-visible fixed 3x3 PCF directional shadow filtering; next add broader material/shader authoring policy, Metal presentation, production shadow quality, richer post-processing, render graph, and sampler-backed runtime material texture sampling coverage where missing.
- Add shader reflection only if it can stay tool-owned; otherwise generate explicit binding metadata during cook.
- Keep D3D12/Vulkan/Metal native handles private and validate each backend with readback tests.

### 3. Authoring Loop For Scenes, Materials, And Assets

The editor shell and editor-core models exist, but the production authoring loop is still thin. The engine needs tools that create and edit the same content the runtime consumes.

Recommended clean implementation:

- Continue from the implemented `MK_editor_core` material authoring model v0, which edits first-party `MaterialDefinition` source data through deterministic factor rows, fixed texture-slot rows, registry-backed texture validation, text-store save/load, dirty/staged state, dependency extraction, and undoable actions before GUI widgets.
- Continue from the implemented `MK_editor_core` scene/prefab authoring model v0, which edits `mirakana::Scene` hierarchy state through deterministic rows, selected-node dirty state, text-store scene/prefab save/load, registry-backed mesh/material/sprite diagnostics, package candidate rows, and UndoStack-backed create/rename/reparent/delete/duplicate/transform/component/prefab-instantiation actions before GUI widgets.
- Promote scene asset save/load workflows into the desktop editor shell and generated-game package registration once the model rows are wired.
- Continue from the implemented dependency-free prefab variant/override contract, `MK_editor_core` authoring document, and visible prefab variant GUI adapter into nested prefab propagation, conflict/merge UX, and instance link tracking.
- Add viewport gizmo/edit operations that remain editor-core testable.
- Keep Dear ImGui as an adapter over tested editor-core models.

### 4. Runtime Session Services

Most real games need save data, settings, localization, input action mapping, and runtime diagnostics. The first host-independent foundation is now split correctly by dependency direction: `MK_runtime` owns deterministic save/settings/localization/input-action documents through `IFileSystem`, `GameEngine.RuntimeInputActions.v4` context-scoped keyboard/pointer/gamepad button and scalar key-pair/gamepad-axis action documents evaluated through `RuntimeInputStateView`, `RuntimeInputContextStack`, default-context `bind_key`, `bind_pointer`, `bind_gamepad_button`, `bind_key_axis`, `bind_gamepad_axis`, and evaluate helpers, explicit `bind_key_in_context`, `bind_pointer_in_context`, `bind_gamepad_button_in_context`, `bind_key_axis_in_context`, and `bind_gamepad_axis_in_context` methods, context-aware `action_down` / `axis_value` overloads over first-party virtual input state, deterministic scalar clamp/deadzone handling, `RuntimeInputRebindingFocusCaptureRequest` / `RuntimeInputRebindingFocusCaptureResult` and `capture_runtime_input_rebinding_action_with_focus` for a narrow armed rebinding capture focus/consumption guard, plus `RuntimeDiagnosticReport` helpers for package load, payload inspection, scene material resolution, and session document failures; `MK_core` owns the dependency-free observability contract for `DiagnosticEvent`, `CounterSample`, `ProfileSample`, bounded `DiagnosticsRecorder`, manual/steady profile clocks, `ScopedProfileZone`, summary aggregation, and deterministic Trace Event JSON subset export. `MK_runtime_host` adds the first adapter on top of that foundation: host-supplied recorders can capture `runtime_host.frame` profile samples and backend-neutral renderer counters from `IRenderer::stats()`. `MK_editor_core` now adapts `DiagnosticCapture` into deterministic editor profiler summary/profile/counter/event rows while `MK_editor` records an `editor.frame` zone and editor counters. RHI GPU marker/timestamp-frequency and read-only memory diagnostics foundations exist; telemetry upload, crash reporting, production allocator hooks/enforcement, binary/cloud saves, richer defaults/override layering, broad UI focus integration, global input consumption/bubbling beyond focused rebinding capture, radial stick deadzones/response curves/full rebinding UI/input glyphs/per-device profiles, production trace import/export workflows, and sample-game integration remain follow-up work.

Recommended clean implementation:

- Expand the implemented `MK_runtime` session services, `MK_core` observability foundation, `MK_runtime_host` frame/counter adapter, recorder-backed editor profiler panel model, RHI GPU debug marker/timestamp-frequency hooks, and RHI memory diagnostics snapshots into sample coverage, telemetry/trace diagnostics, binary/cloud save adapters, settings defaults/override layering, broad UI focus integration beyond the focused rebinding capture guard, global input consumption/bubbling beyond focused rebinding capture, radial stick deadzones, response curves, full rebinding UI, input glyphs, per-device profiles, allocator policy adapters, and crash-report adapters.
- Use platform storage roots and `IFileSystem`; do not expose platform paths to game code.
- Treat save/settings schemas as versioned data with strict validation.
- Keep localization text keys separate from UI layout and renderer code.
- Add public samples that use these services.

### 5. Apple And Metal Production Validation

Apple packaging and Metal ownership are partly implemented, but visible backend integration and actual macOS/Xcode hosted execution are still gated.

Recommended clean implementation:

- Run the existing iOS Simulator CI lane on macOS/Xcode and keep failures as real blockers.
- Finish Metal shader library creation and backend-neutral visible clear/draw/readback/present.
- Add device signing validation only through protected CI secrets or a self-hosted Mac.

### 6. Editor-Independent Desktop Game Runtime Shell

The editor has the richest authoring desktop path, but game executables need their own host. The first slices are now implemented: `MK_runtime_host` drives `mirakana::GameApp` through public `IWindow`, virtual input, lifecycle, and `IRenderer` contracts, optional `MK_runtime_host_sdl3` owns SDL event polling for the `desktop-runtime` lane, and optional `MK_runtime_host_sdl3_presentation` owns renderer selection diagnostics plus the reusable `mirakana::SdlDesktopGameHost` wrapper. `games/sample_desktop_runtime_shell` opens a window through `SdlDesktopGameHost` and receives its renderer from host-owned `SdlDesktopPresentation`, which privately probes SDL3 Win32 window properties for D3D12 and Vulkan surface candidates, can create D3D12 or Vulkan swapchain-backed `RhiFrameRenderer` instances when host code supplies precompiled vertex/fragment shader bytecode through first-party descriptors, validates Vulkan surface capabilities/formats/present modes before swapchain creation, and falls back to `NullRenderer` when native surface/backend/runtime shader policy is unavailable. `games/sample_desktop_runtime_game` proves a non-shell windowed game target can use the same host path, validate through deterministic `NullRenderer` fallback, read a selected-target packaged config file through `mirakana::RootedFileSystem`, load a bundled cooked scene package through `mirakana::runtime::load_runtime_asset_package`, instantiate it through `mirakana::instantiate_runtime_scene_render_data`, and submit the resulting packet through `mirakana::submit_scene_render_packet` during installed smoke. It ships target-specific scene DXIL and SPIR-V artifacts for the selected package lane and can require a real SDL3/Win32 D3D12 or Vulkan presentation with host-owned scene GPU bindings; installed strict smokes validated `renderer=d3d12` or `renderer=vulkan`, `scene_gpu_status=ready`, and two resolved mesh/material GPU binding submissions on a ready Windows/toolchain host. `MK_runtime_scene_rhi` provides the host-independent package/packet-to-`SceneGpuBindingPalette` bridge that uploads unique scene mesh, texture, and material payloads through `MK_runtime_rhi` while retaining RHI resources and descriptor bindings, and `MK_runtime_host_sdl3_presentation` owns the private D3D12/Vulkan device/pipeline/palette injection paths. The first native-presentation foundation is also in place: swapchain-backed `RhiFrameRenderer::resize` calls `IRhiDevice::resize_swapchain`, and `DesktopGameRunner` can drive that path from a public window extent. `tools/package-desktop-runtime.ps1` can now select a registered desktop runtime game target for install and installed smoke validation; the default `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/package-desktop-runtime.ps1` path packages the DXIL/SPIR-V-backed sample shell. Metal game-window renderer creation and broader authored/cooked asset conventions remain separate host/backend binding work.

Recommended clean implementation:

- Keep gameplay code on `mirakana::GameApp`; the host supplies `IWindow`, virtual input/lifecycle, and renderer services.
- Keep `SDL_PollEvent`, native window handles, GPU swapchain handles, and editor APIs in host/backend modules, not public game code.
- Use `SdlDesktopGameHost` for normal SDL3 windowed game hosts and `SdlDesktopPresentation` for lower-level host adapter work so renderer selection, native surface probing, and fallback diagnostics stay in host code before backend-specific presentation is promoted.
- Promote native RHI-backed game-window presentation further only after the current shader artifact package metadata, swapchain diagnostics, resize behavior, and fallback policy are tested per backend.

## P2 Or Genre-Dependent Gaps

These are not mandatory for every engine MVP, but become mandatory for specific game types:

- Networking and replication.
- Navigation mesh assets, spline/funnel smoothing beyond the implemented conservative grid line-of-sight smoother, full crowd simulation beyond the explicit-neighbor local avoidance helper, richer AI decision controllers beyond the memoryless behavior tree plus typed blackboard-condition foundation, scene/physics/navigation integration, and editor visualization beyond the implemented deterministic grid pathfinding, grid dynamic replan helpers, grid path smoothing helper, single-agent path following/arrive steering, value-type movement controller foundation, local avoidance helper, and behavior tree blackboard evaluator.
- Terrain.
- Particles/VFX authoring.
- Advanced 2D tilemap workflow.
- Scripting or hot-reloadable gameplay modules.
- Console platform support.
- Cloud saves, achievements, leaderboards, purchases.

## Recommended Priority Order

1. **Runtime Asset Package Reader and Runtime Handles**
  Host-independent and foundational for rendering, audio, scene loading, editor preview, hot reload, and packaging.
2. **Texture/Mesh Payload Cooking and Upload Path**
  Builds directly on current importer adapters and RHI copy/readback contracts.
3. **Material GPU Binding And Texture Sampling**
  Converts the existing material metadata into visible game rendering.
4. **Runtime Session Services Expansion: Diagnostics, Samples, Pointer/Gamepad Actions**
  The first save/settings/localization/keyboard-action foundation exists; expand it into complete game-facing workflows.
5. **Desktop Visible Game Runtime Shell And Native Presentation**
