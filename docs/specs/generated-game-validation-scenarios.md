# Generated Game Validation Scenarios

Use these scenarios when an AI agent creates or expands a game under `games/<game_name>/`. The selected scenario must be reflected in the game's `game.agent.json` through `backendReadiness`, `importerRequirements`, `packagingTargets`, and `validationRecipes`.

First read `engine/agent/manifest.json.aiOperableProductionLoop` or the `productionLoop` section from `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1`. The currently enforced recipe ids are `headless-gameplay`, `desktop-runtime-config-package`, `desktop-runtime-cooked-scene-package`, `desktop-runtime-material-shader-package`, `ai-navigation-headless`, `runtime-ui-headless`, `2d-playable-source-tree`, `2d-desktop-runtime-package`, `3d-playable-desktop-package`, `native-gpu-runtime-ui-overlay`, `native-ui-textured-sprite-atlas`, `native-ui-text-glyph-atlas`, `native-ui-atlas-package-metadata`, and `future-3d-playable-vertical-slice`.

Also inspect `aiCommandSurfaces` before changing game manifests, runtime package files, scene/prefab/material data, source asset registrations, package cooking, or validation command execution. Command surfaces are descriptor contracts with `schemaVersion`, `requestModes`, `requestShape`, `resultShape`, `requiredModules`, `capabilityGates`, `hostGates`, `validationRecipes`, `unsupportedGapIds`, and placeholder `undoToken` fields. Ready apply modes are narrow: `register-runtime-package-files` for reviewed `runtimePackageFiles` appends, `register-source-asset` for `GameEngine.SourceAssetRegistry.v1` source registry rows plus deterministic `GameEngine.AssetIdentity.v2` and import-metadata projections, `cook-registered-source-assets` for explicitly selected source registry rows into deterministic cooked artifacts plus `.geindex` updates through reviewed import/package helpers, `migrate-scene-v2-runtime-package` for reviewed `GameEngine.Scene.v2` plus source-registry rows into the existing `GameEngine.Scene.v1` `.scene` plus `.geindex` update surface, `create-scene` / `add-scene-node` / `add-or-update-component` / `create-prefab` / `instantiate-prefab` for authored Scene/Prefab v2 source rows, `update-ui-atlas-metadata-package` for cooked `.uiatlas` plus `.geindex` rows, `create-material-instance` for first-party `.material` plus `.geindex` material rows, and `update-scene-package` for first-party `GameEngine.Scene.v1` `.scene` plus `.geindex` scene dependency rows. `validate-runtime-scene-package` is ready only as non-mutating runtime scene package validation through `mirakana::plan_runtime_scene_package_validation` and `mirakana::execute_runtime_scene_package_validation`, using package load plus `mirakana_runtime_scene` diagnostics without package cooking or renderer/RHI claims. `run-validation-recipe` is ready only for allowlisted validation recipes through `tools/run-validation-recipe.ps1`; it does not evaluate arbitrary shell or raw manifest command strings, and free-form validation commands are unsupported. `prefabScenePackageAuthoringTargets` rows can sequence those reviewed surfaces for generated 3D packages, but they are descriptors rather than broad orchestration commands. Other descriptors must produce planned/blocked diagnostics until their own focused slices land.

For editor-assisted packaged playtests, use Editor Playtest Package Review Loop v1: review editor package candidate and package registration draft diagnostics, apply only reviewed `runtimePackageFiles` additions, select `validate-runtime-scene-package` inputs from `runtimeSceneValidationTargets`, run that non-mutating validation, then run the selected host-gated desktop smoke. Do not skip from editor package candidates directly to package smoke, and do not use this loop to claim broad package cooking, runtime source parsing, renderer/RHI residency, package streaming, editor productization, Metal readiness, native handles, or general renderer quality.

## Headless Loop

- Production recipe: `headless-gameplay`.
- Use for: rules prototypes, simulations, turn-based games, deterministic gameplay tests.
- Modules: `mirakana_core`, `mirakana_math`, optional `mirakana_assets`, optional `mirakana_scene`.

- Importer requirements: `sourceFormats=["first-party"]`, `cookedOnlyRuntime=true`.
- Proof: executable output or unit tests plus `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1`.

- Packaging target: `desktop-game-runtime` plus `desktop-runtime-release` only for registered targets whose package validation passes.
- Scaffold: use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntime2DPackage` for generated packaged 2D gameplay.
