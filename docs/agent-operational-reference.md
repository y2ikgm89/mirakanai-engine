# Agent operational reference (English)

This document expands validation, tooling, editor-shell, and IDE-integration guidance that `AGENTS.md` indexes in short form. Keep `AGENTS.md` concise enough for Codex's default project-doc budget when practical, and keep needles required by `tools/check-ai-integration.ps1` synchronized when you change policy here.

## Testing and validation (expanded)

- Before production C++ behavior, add or update tests first when the environment can run them; target the smallest externally meaningful guarantee for the behavior/API/regression rather than adding tests by habit.
- Prefer updating an existing test when it already covers the contract. Avoid tests that merely mirror implementation details, duplicate an existing guarantee, or over-specify incidental ordering unless deterministic ordering is the public contract.
- During implementation, use the smallest relevant focused loop first: targeted CMake build/test commands for the changed target, plus only the static checks that match the files touched. Avoid rerunning full `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` after every code or docs edit.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` before reporting completion.
- Run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` once near the end of a coherent slice, after code, docs, manifest, and static-check updates are settled. Rerun it only if later edits can affect validated behavior or checked metadata.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` when diagnosing local CMake, CTest, CPack, Visual Studio, MSBuild, or `PATH` issues.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/cmake.ps1 ...` and `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/ctest.ps1 ...` for focused local CMake/CTest loops; these wrappers normalize child `Path` for Windows CMake/MSBuild/CTest and `PATH` elsewhere before launching the resolved tools.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1 -RequireDirectCMake` before direct `cmake --preset ...` work when the task depends on raw `cmake` being on `PATH`; checked-in presets normalize the Visual Studio/MSBuild `PATH`/`Path` handoff for visible configure/build presets.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1 -RequireVcpkgToolchain` when the selected vcpkg-backed lane must fail fast on a missing `external/vcpkg` checkout.
- Checked-in CMake configure/build presets must inherit `normalized-configure-environment` / `normalized-build-environment`; `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` enforces the preset contract and reports parent path-casing diagnostics.
- Treat generated CMake/MSBuild trees under `out/build/` as disposable outputs, not configuration source. When `PATH`/`Path` casing or stale configure state poisons a tree, recover through `tools/cmake.ps1 --preset <preset>` or a verified removal of that `out/build/<preset>` directory; do not patch generated project files or CMake cache entries directly.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-format.ps1` / `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/format.ps1` for repository formatting. Raw `clang-format --dry-run ...` commands require `clang-format` on `PATH`; check `direct-clang-format-status` from `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` before treating a raw lookup failure as a formatting blocker.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-public-api-boundaries.ps1` after changing public headers or backend interop.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-shader-toolchain.ps1` when shader, RHI, or viewport work depends on local `dxc`, `spirv-val`, `metal`, or `metallib` availability.
- On Windows native crash, dump, or debugger-path work, use the official Debugging Tools for Windows (`cdb`, `windbg`) from the Windows SDK and the Microsoft public symbol server through `_NT_SYMBOL_PATH`; do not require `gdb`/`lldb` for the MSVC lane.
- On Windows D3D12 debug-layer work, use the official Windows Graphics Tools capability and verify `d3d12SDKLayers.dll`; use PIX on Windows for D3D12 GPU captures/counter investigations, and Windows Performance Toolkit (`wpr`, `wpa`, `xperf`) for ETW/CPU performance work.
- Optional repository helper for launching the PIX UI from a non-repo scratch directory: `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/launch-pix-host-helper.ps1` (resolves `WinPix.exe` under `Program Files\Microsoft PIX\<version>\`, legacy `PIX.exe` when present, `PATH` fallbacks, or override `MIRAKANA_PIX_EXE`; use `-SkipLaunch` to verify resolution only). See `docs/ai-integration.md` for editor vs operator capture handoff.
- Treat Debugging Tools for Windows, Windows Graphics Tools, PIX on Windows, and Windows Performance Toolkit as host diagnostics, not repository runtime dependencies. Their absence is a blocker only when the selected task needs that diagnostic lane.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1` after changing shared C++ implementation patterns; it verifies `.clang-tidy`, configures the active CMake preset when needed, accepts native Makefile/Ninja `compile_commands.json`, or **synthesizes** `compile_commands.json` under the preset `binaryDir` from CMake File API codemodel data for the default Windows Visual Studio `dev` preset. `tools/validate.ps1` avoids duplicate configure/build work by running independent static checks through bounded PowerShell background jobs with per-check timeouts, having `build.ps1` request File API codemodel data before generation, then running the tidy smoke with explicit reply reuse unless `-SkipTidySmoke` is explicitly passed and CTest with `test.ps1 -SkipBuild`; standalone wrappers remain conservative by default. Use `tools/validate.ps1 -StaticJobs <N>` only when a local host needs an explicit static-check throttle, and use `-StaticCheckTimeoutSeconds <N>` only to diagnose or lower a host-specific static-check timeout. Hosted CI uses `tools/validate.ps1 -StaticOnly -StaticJobs 1 -StaticCheckTimeoutSeconds 120` for the dedicated static lane and `tools/validate.ps1 -SkipStaticChecks -SkipTidySmoke` for the Windows MSVC lane so repository static checks and full clang-tidy are not duplicated there; raw Linux, sanitizer, and macOS lanes pass explicit official `--parallel` job counts to CMake build and CTest. Tooling must treat `Get-RepoRoot` as a **string** path (use `$root` in `Join-Path` / string replace—never `$root.Path`, which is null on strings and breaks preset path resolution). Missing CMake or `clang-tidy` remains a local tool blocker. For a narrow loop on one or more translation units, use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1 -Files engine/path/to/file.cpp,...` after `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/cmake.ps1 --preset dev` so entries exist in the compile database. **`check-tidy.ps1 -Files` accepts only `.cc`, `.cpp`, and `.cxx`**; to tidy-check a public header, pass the primary implementation TU that includes it (for example `editor/core/src/scene_authoring.cpp` for `editor/core/include/mirakana/editor/scene_authoring.hpp`) or rely on IDE `clangd` diagnostics on the header. `.clang-tidy` enables `misc-*`, including `misc-include-cleaner` (IWYU-style direct `#include` guidance); when touching a source file, prefer fixing includes there rather than disabling checks without an explicit project decision—see `docs/cpp-style.md`.
- For PR static-analysis failures, treat `warnings generated` volume as a diagnostic signal until proven otherwise. `.clang-tidy` `HeaderFilterRegex` must match absolute compile-database paths on Windows and Linux, strict CI must keep `--warnings-as-errors=*`, and wrapper log cleanup may suppress only frontend summary lines like `NN warnings generated.` after preserving real diagnostics and exit codes. Use `-ShardCount` / `-ShardIndex` plus `-Jobs 0` for sharded full hosted lanes and `-Files` for focused local TUs.
- Large unit translation units under `tests/unit/` should match the same clang/clang-tidy hygiene as production code where practical: designated aggregate initialization (including fields added later and declaration-order designated lists when the toolchain warns), C++23 `std::string::contains` / `starts_with` and the same **`std::string_view`** membership/prefix helpers where applicable, file-local helpers in anonymous namespaces (for very large TUs, a single anonymous namespace through the test bodies with `int main()` left at global scope is acceptable—see **Unit tests** in `docs/cpp-style.md`), and `class` test doubles with private state plus `const` getters when guideline checks fire on public counters. For production `editor/core` and similar adapters with a custom destructor and deleted copy operations, complete **Rule of Five** (explicit deleted or defined move members) when clang-tidy special-member rules apply—see **Unit tests** / production notes in `docs/cpp-style.md`.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-coverage.ps1` for Linux GCC/Clang coverage work; non-Linux hosts report an explicit blocker.
- Dependency policy changes must pass `tools/check-dependency-policy.ps1`, which is included in `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1`.
- If CMake, `clang-format`, or a compiler is missing, report that validation is blocked by missing local tools and include the exact failing command.
- For C++ changes, the intended validation loop is:
  - `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1`
  - `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/cmake.ps1 --preset dev`
  - `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/cmake.ps1 --build --preset dev`
  - `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/ctest.ps1 --preset dev --output-on-failure`
- Direct `cmake --preset ...` commands assume CMake is on `PATH`; repository wrappers remain the preferred local loop because they use the resolved full path reported by `check-toolchain.ps1` and normalize before launch, while CMake presets also normalize the configure/build environment for raw preset commands.
- Direct `clang-format --dry-run ...` commands assume `clang-format` is on `PATH`; repository format wrappers use the resolved tool reported by `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1`, including Visual Studio Build Tools LLVM tools when available.

## Deferred visible editor shell (`MK_editor`)

The visible `MK_editor` shell is deferred during SDL3 removal. The removed SDL3/Dear ImGui shell sources (`editor/src/main.cpp`, `editor/src/material_preview_gpu_cache.cpp`, `editor/src/sdl_viewport_texture.cpp`, and editor-local `compile_flags.txt` fallbacks) are no longer active guidance. `MK_editor_core` remains the supported editor logic target and is covered by the default validation lane.

Future visible editor shell work must start from a focused native shell plan, use first-party Win32/D3D12 adapters, keep backend handles private, and add its own ImGui/native-adapter coding conventions plus validation gates before claiming editor readiness. Until then, `tools/build-gui.ps1`, `tools/evaluate-cpp23.ps1 -Gui`, and `MK_ENABLE_DESKTOP_GUI=ON` intentionally fail closed instead of rebuilding an SDL3/Dear ImGui lane.

## AI development workflow (expanded)

- Start documentation navigation from `docs/README.md`.
- Use `docs/roadmap.md` for current engine status and priorities.
- Use `docs/superpowers/plans/README.md` as the implementation plan registry before creating, editing, or extending plans.
- Use `docs/superpowers/plans/README.md`, the production-completion master plan, specs/ADRs, and `99-historical-verdict-archive.md` to reconcile manifest pointers and historical dated plan evidence without bulk-rewriting ledger prose; foundation layout changes are clean-break and are not a substitute for `unsupportedProductionGaps` burn-down.
- Use `docs/specs/` for feature designs and `docs/superpowers/plans/` for implementation plans.
- Write new and updated implementation plans under `docs/superpowers/plans/` in **English** (headings, prose, tables) so they stay consistent with the production-completion master plan and remain easy to grep and review across tooling; localized explanations may still live in `docs/roadmap.md` or other human-facing docs when needed.
- **Dated plan and spec files:** Use `YYYY-MM-DD` in filenames and in the first heading (for example `# Title v1 (YYYY-MM-DD)`) for the **authoritative calendar date at authoring time**. When working inside Cursor or another agent host, treat the session **`Today's date`** field (or an equivalent user-stated date) as authoritative—do not infer dates from model defaults or UTC-only boundaries. If that field is missing, confirm the date with the operator or run a local `Get-Date -Format yyyy-MM-dd` (or host equivalent) before choosing the prefix; keep filename date and heading date identical.
- Keep the live plan stack shallow: one active roadmap, one active gap-cluster burn-down or milestone, and at most one active phase/child plan selected by `currentActivePlan`.
- Create a new dated plan for capability/gap-cluster/milestone work, then put phase behavior/API/validation boundary decisions, validation-only follow-up, docs synchronization, static-check edits, and substeps inside the active plan checklist.
- Prefer extending the active gap-cluster burn-down or milestone checklist over creating many tiny child plans. Use a phase-gated milestone plan when tightly related production slices share one end-to-end objective and repeated tiny active-plan hops would hide the decision.
- Do not append unrelated work to completed historical plans. Preserve completed plan files only while they remain referenced evidence for current docs, manifests, checks, or active decisions; remove unreferenced historical noise in explicit cleanup tasks and rely on Git history for deleted records.
- Keep active implementation plans concise enough to execute without excessive context. Put detailed historical evidence in the final validation table, not in repeated progress prose, and batch docs/manifest/static-check synchronization after behavior is green unless those files are the behavior under test.
- Prefer capability/gap-cluster/milestone plans over many tiny dated plans when the work shares one architecture decision, public API family, and validation surface; create child plans only when the next phase has a distinct architecture/validation/review boundary or is too large to execute, validate, and review safely in one context.
- Each active phase still needs Goal, Context, Constraints, Done When, and validation evidence; do not use gap burn-downs or milestones to broaden ready claims or weaken host gates.
- Read `engine/agent/manifest.json` (or a **targeted fragment** under `engine/agent/manifest.fragments/` when you know the slice) or run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1` before generating game code or changing engine APIs. Prefer `tools/agent-context.ps1 -ContextProfile Minimal` or `-ContextProfile Standard` when a smaller JSON summary is enough; default is `-ContextProfile Full` (full manifest-shaped output). Avoid loading full historical plans or broad docs when a targeted fragment, registry row, or focused search answers the current decision.
- Use repository skills in `.agents/skills/` when the task matches their description (Codex). Keep overlapping topics in **behavioral parity** with `.claude/skills/` (Claude Code): update both trees in the same task when shared workflows or validation commands change.
- Use Cursor Agent Skills in `.cursor/skills/` when working inside Cursor. Prefer thin `.cursor/skills/gameengine-*` folders that name-match `.claude/skills/` and link readers to the full `.claude/skills/` / `.agents/skills/` bodies; keep IDE-only guidance there and avoid duplicating long prose that already lives in `AGENTS.md` or shared skills. Cursor-only skill folders that intentionally have no Claude twin stay limited to `gameengine-cursor-baseline` and `gameengine-plan-registry` unless `tools/check-agents.ps1` is updated together.
- After changing agent surfaces (`tools/*.ps1` dot-source pairs, `tools/static-contract-ledger.ps1` line budgets or numeric-prefix discovery, skill folders, or validation scripts), follow **Repository consistency checklist** in `docs/workflows.md` (toolchain → `check-agents` → `check-ai-integration` → public API checks → `validate.ps1` as appropriate).
- `MK_tools` implementation sources live under `engine/tools/{shader,gltf,asset,scene}/` with public headers in `engine/tools/include/mirakana/tools/` (see `docs/specs/2026-05-11-directory-layout-target-v1.md`). The Windows `dev` preset keeps the visible editor shell disabled during SDL3 removal and still builds `MK_editor_core`; run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1` before vcpkg-backed lanes when `vcpkg_installed/` is missing. Treat a missing database as “configure the dev preset first,” not a signal to change public include style.
- Use native worktree support for parallel file isolation: Codex app Worktree/Handoff for Codex threads, Claude Code `--worktree` for separate Claude sessions, and `isolation: worktree` for write-capable Claude subagents. Manual `git worktree` is a fallback; inspect `git worktree list`, run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/prepare-worktree.ps1` inside the linked worktree before configure, register auto-merge with `gh pr merge --auto --merge --match-head-commit <headRefOid>` without `--delete-branch`, and use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/remove-merged-worktree.ps1 -WorktreePath <path> [-BaseRef origin/main] [-BaseBranch main] [-Remote origin] [-LocalCheckoutPath <path>] [-DeleteLocalBranch] [-DeleteRemoteBranch]` after merge so the local base checkout fast-forwards before guarded cleanup. The cleanup script accepts standard roots from the script checkout, `-LocalCheckoutPath`, and Git's main worktree porcelain record, checks local and remote branch ancestry before deletion, then unlinks worktree-local `external/vcpkg` and `vcpkg_installed` reparse points before `git worktree remove`; if Windows long paths block Git removal, the fallback remains inside the guarded script after clean/merged/root checks and must not follow reparse points. If no linked worktree or local branch remains after merge, there is no cleanup target to pass.
- Publication-capable Codex work needs more than worktree file writes: linked worktrees write Git metadata under the parent repository `.git/worktrees/<name>`, and PR publication also needs remote network access plus readable host-local Git/GitHub authentication. For tasks that include commit, push, PR, merge, or cleanup, start or switch to a trusted local/full-access session before publishing; if sandbox policy blocks `index.lock`, `git ls-remote`, or `gh auth status`, record the blocker instead of bypassing GitHub Flow with API object writes or loosening command gates.
- Use project subagents in `.codex/agents/` and `.claude/agents/` for bounded independent work such as parallel read-only exploration, plan lifecycle audits, agent-surface drift audits, code review, build failures, gameplay implementation, and rendering-specific audits when the user explicitly asks for subagent delegation or parallel agent work. Give each delegated task clear ownership, keep immediate blocking work local, close completed/obsolete/no-longer-needed delegated agents promptly after their results are consumed, and leave useful independent work running.
- Every task should define: Goal, Context, Constraints, Done when.

- The production-completion historical verdict archive now lives in `docs/superpowers/master-plans/production-completion-v1/99-historical-verdict-archive.md` and preserves retained static-check evidence. Do not remove or hollow it without running `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` afterward.
- Use this reusable execution prompt when resuming production-completion work: finish the selected `currentActivePlan` or production gap with official-best-practice, clean breaking implementation; keep context small with focused reads, focused validation, and bounded subagents only when they improve speed or confidence; keep the tree free of temporary files, dead code, dead files, and unnecessary folders; do not trade away tests, docs, manifest truth, static checks, host gates, or ready-claim evidence for speed.
- Re-read the master plan, plan registry, and manifest after user edits or context resumes before changing files.
- Finish the selected production gap as implemented, host-gated, blocked with evidence, or explicitly excluded from the 1.0 ready surface before switching to another gap.
- Prefer official documentation, Context7, OpenAI developer docs, Anthropic docs, project skills, and clean breaking greenfield designs over compatibility shims, broad ready claims, or undocumented shortcuts.
- Use focused validation while iterating, then `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` at the coherent slice-closing gate; do not weaken static checks, host gates, or ready-claim evidence to save time.
- Use subagents only for bounded independent investigation, plan lifecycle audit, agent-surface drift audit, review, build-failure triage, or disjoint implementation that improves speed or confidence; keep immediate blocking work local. Close completed, obsolete, or no-longer-needed agents promptly after consuming their results and before spawning replacements; do not wait for the session to approach its subagent limit. Leave still-useful independent work running. Keep fixed Codex model defaults limited to stable roles (`explorer` on `gpt-5.3-codex-spark`, `agent-surface-auditor` on `gpt-5.4-mini`, and `cpp-reviewer` / `engine-architect` / `planning-auditor` / `rendering-auditor` on `gpt-5.4`), and use inherited models or per-invocation overrides for write-capable delegated work.
- Before reporting a production slice complete, reconcile code, tests, docs, plans, manifest, static checks, completed gap, remaining gaps, next active plan, and host-gated blockers against actual validation evidence.

## AI-driven game development (expanded)

- Games live under `games/<game_name>/`.
- `game_name` and `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name>` values must match `^[a-z][a-z0-9_]*$`; keep source-tree game directories and `runtimePackageFiles` path segments lowercase snake_case.
- JSON manifest IDs, display names, and external package identifiers may use their ecosystem formats, including kebab-case when required; do not convert those identifiers back into source paths.
- Every game must include `game.agent.json` so AI agents can understand the game contract, including backend readiness, importer requirements, packaging targets, runtime package files, runtime scene validation targets, and validation recipes.
- Most current sample games are headless validation executables. `sample_desktop_runtime_shell` and `sample_desktop_runtime_game` are the optional windowed SDL3 desktop runtime samples; do not promise a visible game window outside the desktop-runtime lane unless a runtime windowed host is implemented. Use `mirakana_editor` for the broader visible desktop/editor smoke path.
- Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name>` to scaffold a new C++ game. Shared scaffolding helpers live in `tools/new-game-helpers.ps1`, and generated source/runtime template functions live in `tools/new-game-templates.ps1` (both dot-sourced by `new-game.ps1`); update the matching helper/template file when changing formatting, string-literal helpers, or generated scaffold content.
- Register default source-tree games through `games/CMakeLists.txt` with `MK_add_game`. Register optional windowed desktop runtime games with `MK_add_desktop_runtime_game`.
- When a game selects the optional `desktop-game-runtime` lane, validate with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate-desktop-game-runtime.ps1`; use `tools/package-desktop-runtime.ps1 -GameTarget <target>` when validating a selected registered desktop-runtime package target. Registered desktop runtime game targets must declare source-tree smoke args, package smoke args, shader-artifact requirements, and package files through CMake metadata when runtime config/assets must ship beside the executable; keep `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/package-desktop-runtime.ps1` as the default sample-shell package lane.
- When a game or engine task selects `android-gameactivity`, run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-mobile-packaging.ps1` first. If Android reports ready, validate Debug packages with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/build-mobile-android.ps1 -Game <game_name> -Configuration Debug`; validate Release signing with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-android-release-package.ps1 -Game <game_name> -UseLocalValidationKey` for local non-repository keys or CI secrets for real upload keys; validate emulator launch with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/smoke-android-package.ps1 -Game <game_name> -Configuration Release -SkipBuild -StartEmulator -AvdName Mirakanai_API36` when the configured AVD is available. Do not store Android keystores, passwords, SDKs, Gradle caches, or AVD images in the repo.
- Game code should include only public engine headers unless implementing engine internals.
