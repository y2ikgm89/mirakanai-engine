## Play-In-Editor

Play-In-Editor Session Isolation v1 is represented in `MK_editor_core` by `mirakana::editor::EditorPlaySession`. Starting a session snapshots the active `SceneAuthoringDocument` source scene into an isolated simulation scene copy, exposes begin/pause/resume/tick/stop transitions, reports the source and simulation node counts through `EditorPlaySessionReport`, blocks source-scene edit claims while active, counts simulation ticks only in play state, and discards the simulation copy on stop without writing it back to the authored scene.

Play-In-Editor Gameplay Driver v1 adds `mirakana::editor::IEditorPlaySessionDriver` and `mirakana::editor::EditorPlaySessionTickContext`. A caller may start a session with a reviewed driver; editor core invokes `on_play_begin`, `on_play_tick`, and `on_play_end` against the isolated simulation scene only. Tick calls require a finite positive delta, include the current frame index and delta seconds in the context, update `last_delta_seconds`, and reject ticks while paused.

Play-In-Editor Visible Viewport Wiring v1 adds `mirakana::editor::EditorPlaySessionControlsModel` and command rows for Play/Pause/Resume/Stop availability. A future visible `MK_editor` Run menu and Viewport toolbar use that model to start, pause, resume, and stop `EditorPlaySession`, render the isolated simulation scene while active, tick the play session once per viewport frame in play state, and reject source-scene authoring plus undo/redo while `source_scene_edits_blocked()` is true.

## UI Strategy

The current supported editor surface is `MK_editor_core` plus retained `MK_editor_ui` models, with `MK_editor` as an optional native developer shell. The visible shell now uses private Win32 first-party retained `mirakana::ui` / `MK_ui_renderer` code through the dependency-free `desktop-editor` lane; First-Party Editor Shell v1 completed the agent/static closeout after deleting inactive Dear ImGui implementation files without compatibility forwarding, and runtime game UI stays on public `mirakana::ui` rather than editor-private or middleware APIs.

The removed SDL3/Dear ImGui shell sources (`material_preview_gpu_cache.*`, `sdl_viewport_texture.*`, and editor-local `compile_flags.txt` fallbacks) are no longer active guidance. For **`editor/core`** (GUI-independent sources), follow **Editor core C++** in the editor skills plus **Unit tests** / production adapter notes in `docs/cpp-style.md` (`std::string_view::contains` / `starts_with`, Rule of Five on C API adapters, `check-tidy.ps1 -Files editor/core/src/<file>.cpp`). Additional native shell work must stay in focused plan phases and keep Win32/D3D12 implementation details private to `editor/src` targets.

UI work is tracked in `docs/ui.md` and Wave 8 of `Engine Excellence Roadmap`: first-party retained-mode `MK_ui` contracts cover game HUDs and menus, while `MK_editor_ui` models cover complex editor panels. The completed clean-break editor plan follows the same separation of concerns as Unity's retained Editor UI guidance, Unreal's engine-owned Slate/UMG layering, and Godot's first-party Control/TextServer split: editor tooling, runtime game UI, low-level text/font/IME/accessibility adapters, and debug surfaces are separate systems.

Rules:

- Runtime game code must not depend on `editor`, SDL3, or Dear ImGui.
- New complex editor panels should first add GUI-independent state/model contracts in `editor/core`, then expose durable retained models through `MK_editor_ui` before wiring any future visible-shell adapter.
- Do not hand-roll mature low-level font, text shaping, IME, accessibility, image decoding, or platform integration systems inside editor panels; route them through engine adapters after an accepted dependency/license decision.
- Do not add Qt, NoesisGUI, Slint, RmlUi, fonts, or UI middleware until `license-audit`, `docs/dependencies.md`, `vcpkg.json`, and `THIRD_PARTY_NOTICES.md` are updated.

## Project Text Format

The current project format is deterministic line-based text:

```text
format=GameEngine.Project.v4
project.name=sample
project.root=games/sample
project.asset_root=assets
project.source_registry=source/assets/package.geassets
project.game_manifest=game.agent.json
project.startup_scene=scenes/start.scene
project.render_backend=auto
project.shader_tool.executable=dxc
project.shader_tool.working_directory=.
project.shader_tool.artifact_output_root=out/editor/shaders
project.shader_tool.cache_index=out/editor/shaders/shader-cache.gecache
```

Rules:

- The first supported format version is `GameEngine.Project.v1`.
- The current write format is `GameEngine.Project.v4`.
- All current fields are required when authoring a new project file.
- Project fields must not contain newlines or `=`.
- Project paths must be project-relative and must not contain `..` traversal.
- `project.source_registry` is a safe project-relative `.geassets` path. New projects default it to `source/assets/package.geassets`.
- Shader tool executable values must be a PATH-resolved tool name such as `dxc`, a project-relative reviewed tool path, or a reviewed absolute installed SDK/tool path. They must not contain `..` traversal, and shell executables are rejected.
- Shader tool working directory, artifact output root, and cache index path are project-relative and must not contain `..` traversal.
- Render backend preference values are `auto`, `null`, `d3d12`, `vulkan`, or `metal`.
- Unknown keys and unsupported formats fail fast.
- `migrate_project_document` upgrades `GameEngine.Project.v0` by filling the default asset root, game manifest path, and startup scene path.
- `migrate_project_document` upgrades `GameEngine.Project.v1` by filling default shader tool settings.
- `migrate_project_document` upgrades `GameEngine.Project.v2` by filling the default `auto` render backend preference.
- `migrate_project_document` upgrades `GameEngine.Project.v3` by filling the default project source registry path.

- Editor Prefab Variant Missing Node Cleanup v1 lets the authoring document load review-cleanup-repairable stale variants, marks missing-node override rows as explicit `Remove missing-node override` resolutions, and allows the visible editor to remove those stale rows through the same reviewed undoable action path before saving a valid variant.
- Editor Prefab Variant Node Retarget Review v1 records source-node hints on editor-created name/transform/component overrides and exposes an explicit `Retarget override to node N` resolution only when a missing-node row's hint uniquely matches one current base node and the retarget would not create a duplicate node/kind override. Retarget uses the same reviewed `resolve_prefab_variant_conflict` and `UndoStack` path as cleanup; it is never automatic.
- Editor Prefab Variant Source Mismatch Retarget Review v1 classifies an existing-node override as `source_node_mismatch` when its non-empty `source_node_name` no longer matches the current base node at `node_index`. It keeps strict `MK_scene` composition index-based, blocks editor conflict review composition until explicitly resolved, and exposes the same `Retarget override to node N` metadata only when the source-name hint uniquely identifies another current base node without creating a duplicate node/kind override.
- Editor Prefab Variant Source Mismatch Accept Current Review v1 adds `accept_current_node` resolution metadata for existing-node `source_node_mismatch` rows when no safe unique retarget exists. `Accept current node N` updates only `PrefabNodeOverride::source_node_name` to the current indexed base node name through the same reviewed single-row, batch, and `UndoStack` paths; it never changes `node_index` or runtime composition semantics.
- Editor Prefab Variant Batch Resolution Review v1 adds `PrefabVariantConflictBatchResolutionPlan`, `PrefabVariantConflictBatchResolutionResult`, `resolve_prefab_variant_conflicts`, and `make_prefab_variant_conflict_batch_resolution_action`. The plan reports retained `prefab_variant_conflicts.batch_resolution` ids and an explicit `Apply All Reviewed Resolutions` command. The resolver recomputes conflict rows after each reviewed cleanup/retarget so override indices cannot go stale, and the visible editor wraps a valid batch result as one variant-scoped `UndoStack` action.
- Editor Prefab Variant Native Dialog v1 adds `EditorPrefabVariantFileDialogModel`, `make_prefab_variant_open_dialog_request`, `make_prefab_variant_save_dialog_request`, `make_prefab_variant_open_dialog_model`, `make_prefab_variant_save_dialog_model`, and `make_prefab_variant_file_dialog_ui_model`. The model reviews accepted/canceled/failed `.prefabvariant` open/save results, rejects empty, multi-select, and wrong-extension selections, and emits retained `prefab_variant_file_dialog.open` / `prefab_variant_file_dialog.save` rows without bringing SDL3, Dear ImGui, native handles, filesystem IO, or project-store path conversion into editor core.
- Editor Prefab Variant Base Refresh Merge Review v1 adds `PrefabVariantBaseRefreshPlan`, `PrefabVariantBaseRefreshRow`, `PrefabVariantBaseRefreshResult`, `plan_prefab_variant_base_refresh`, `apply_prefab_variant_base_refresh`, `make_prefab_variant_base_refresh_action`, and `make_prefab_variant_base_refresh_ui_model`. The reviewed `prefab_variant_base_refresh` retained rows compare a current embedded base prefab with a refreshed base prefab, preserve or retarget name/transform/component overrides only through unique `PrefabNodeOverride::source_node_name` matches, block missing hints, missing or ambiguous refreshed source nodes, invalid inputs, and duplicate target `(node_index, kind)` keys, and apply the refreshed embedded base only through an explicit undoable action. This is not automatic nested prefab propagation or fuzzy merge/rebase UX.
- Registry-aware validation reports field-specific missing or wrong-kind mesh/material/sprite references on component overrides through `AssetRegistry`.
- Save/load helpers use `ITextStore`; registry-aware save should be used when an editor workflow has an `AssetRegistry` so missing or wrong-kind override references are not persisted accidentally.
- Undoable actions capture full prefab variant snapshots through `UndoStack`; owning shells must keep undo stacks scoped to the variant document lifetime and clear them when replacing a variant document.
- A future visible desktop editor may adapt this model in the Assets panel's Prefab Variant Authoring section. The shell can create a variant from the saved selected prefab, load/save `.prefabvariant` text assets, render override rows, diagnostics, conflict review rows, batch reviewed resolution status, native open/save dialog review rows, and composed-prefab review, route name/transform/component overrides through a variant-scoped `UndoStack`, launch `Browse Load Variant` / `Browse Save Variant` through `mirakana::IFileDialogService` / the native shell file-dialog service, convert accepted in-project `.prefabvariant` selections back to safe project-relative paths, and instantiate the composed prefab into the active `SceneAuthoringDocument` with the `.prefabvariant` path recorded as source-link metadata.
- This is not nested prefab propagation, fuzzy matching, automatic merge/rebase/resolution UX, a dedicated prefab browser, execution from the review model, broad editor native save/open dialogs outside Profiler, Scene, and Prefab Variant, or play-in-editor isolation; instance source links are diagnostic metadata only, retarget, accept-current, batch apply, and base refresh are limited to explicit reviewed rows/actions, and batch resolution recomputes the same reviewed model after each applied row.

## Shader Compile State

`mirakana::editor::ShaderCompileState` owns the GUI-independent view of editor shader compile work.

Rules:

- Compile requests come from `mirakana::ShaderCompileRequest`.
- Items are sorted deterministically by output path, then shader id.
- Updates change item status to `Pending`, `Cached`, `Compiled`, or `Failed` and preserve diagnostics for UI display.
- `mirakana::editor::ShaderCompileState::apply_execution_results` maps `mirakana::ShaderCompileExecutionResult` from the tool layer into editor-facing item status.
- Cache hits are explicit so the editor can distinguish reused artifacts from newly compiled outputs.
- `mirakana::editor::ShaderToolDiscoveryState` stores deterministic `discover_shader_tools` results plus `mirakana::evaluate_shader_toolchain_readiness` diagnostics for UI and AI workflows. Items keep the tool kind label, executable path or tool name, and discovered version string. Vulkan SPIR-V readiness requires DXC, proven DXC SPIR-V CodeGen support, and `spirv-val`.
- `mirakana::editor::ViewportShaderArtifactState` checks compiled shader artifacts produced by editor requests. The D3D12 viewport path requires non-empty vertex and fragment DXIL artifacts before the backend can become available. The Vulkan viewport path separately requires non-empty vertex and fragment SPIR-V artifacts generated through the engine DXC command plan with `-spirv` and `-fspv-target-env=vulkan1.3` before future Vulkan viewport promotion can proceed. The same state type can query readiness by shader id so viewport artifacts and material-preview variants cannot be confused when displayed together.
- A future visible desktop editor may expose this in the Assets panel with compile counts, per-item status, cache-hit state, output paths, diagnostics, and a `Compile Shader` button wired through editable project shader tool settings, `RootedFileSystem`, `Win32ProcessRunner`, `ShaderToolProcessRunner`, and `execute_shader_compile_action`. The default viewport shader request plan is generated by `mirakana::editor::make_viewport_shader_compile_requests` and queues D3D12 vertex/fragment DXIL plus Vulkan vertex/fragment SPIR-V artifacts from the same HLSL source. The material preview request plan is generated by `mirakana::editor::make_material_preview_shader_compile_requests` and queues factor-only plus base-color-textured vertex/fragment artifacts for both D3D12 and Vulkan from `material-preview.hlsl`.
- The Project Settings panel edits the viewport render backend preference plus shader tool executable, working directory, artifact root, and cache index path through `ProjectSettingsDraft`; Apply updates the active project, Save Project persists the current project bundle, and `Open Project...` / `Save Project As...` route `.geproject` selections through the reviewed project native dialog path.
- The Project Settings panel can refresh shader tool discovery from reviewed project-relative toolchain roots plus known installed SDK locations such as `MK_SHADER_TOOLCHAIN_ROOT`, `VULKAN_SDK`/`VK_SDK_PATH`, vcpkg tool roots, and Windows SDK `dxc.exe`, display DXC SPIR-V CodeGen readiness separately, then copy a discovered executable into the draft.

## Viewport

`mirakana::editor::ViewportState` owns GUI-independent viewport readiness, frame counters, play state, active transform tool, and resolved render backend selection.

Rules:

- Renderer name and non-zero extent are required before the viewport is ready.
- `mirakana::editor::choose_editor_render_backend` resolves `auto`, `null`, `d3d12`, `vulkan`, or `metal` against explicit backend availability and host platform policy.
- Automatic selection prefers Direct3D 12 on Windows, Vulkan on Linux/Android/unknown hosts, Metal on Apple hosts, and falls back to `NullRhiDevice` when no native viewport backend is enabled.
- A future visible desktop editor may expose render backend preference in Project Settings. On Windows GUI builds, D3D12 becomes a viewport candidate only after `ViewportShaderArtifactState` verifies non-empty vertex and pixel DXIL artifacts and the D3D12 runtime probe reports hardware or WARP support; failed native device or pipeline creation falls back to `NullRhiDevice`. Vulkan SPIR-V readiness is displayed separately and remains a prerequisite for the future Vulkan editor viewport backend.
- Focused and hovered state are explicit so editor tools can later route input safely.
- Frame count is tracked separately from renderer internals, keeping the editor shell testable without a GPU.
- Run mode is `Edit`, `Play`, or `Paused`; simulation ticks advance only while in `Play`.
- Stopping play mode returns to `Edit` and clears the simulation frame counter.
- The active viewport tool is one of Select, Move, Rotate, or Scale and is independent from play state.

A future visible desktop editor may record the visible viewport extent, focus, hover state, rendered frame count, simulation frame count, run mode, and active viewport tool while owning a renderer-backed `RhiViewportSurface`. The surface recreates its RHI color texture on panel resize, records explicit transitions plus either a clear render pass or a renderer-submitted draw frame, can prepare a shader-readable `RhiViewportDisplayFrame` for native display interop, and can still copy the color target into `RhiViewportReadbackFrame` when needed.

The old SDL3/ImGui viewport display path has been removed from active build lanes. `MK_editor_core` still owns the GUI-independent viewport state, shader artifact readiness rows, and retained models. A future visible shell must provide a first-party native display adapter, keep GUI/backend handles out of engine public APIs, and add fresh validation evidence before claiming low-latency viewport display readiness.

## Scene Transform Editing

`mirakana::editor::SceneNodeTransformDraft`, `mirakana::editor::SceneNodeComponentDraft`, and `mirakana::editor::ViewportTransformEdit` own GUI-independent scene editing.

Rules:

- Create editable node transform copies with `mirakana::editor::make_scene_node_transform_draft`.
- Apply absolute inspector edits with `mirakana::editor::apply_scene_node_transform_draft`.
- Apply active viewport tool deltas with `mirakana::editor::apply_viewport_transform_edit`.
- Create editable component copies with `mirakana::editor::make_scene_node_component_draft`.
- Apply camera, light, mesh renderer, and sprite renderer component changes with `mirakana::editor::apply_scene_node_component_draft`.
- Transform values must be finite, and scale values must stay positive on every axis.
- Component values are validated through the scene component validators before mutation.
- Select does not mutate transforms; Move changes position, Rotate changes radians-based Euler rotation, and Scale applies additive scale deltas.

A future visible desktop editor may expose this through Inspector `Position`, `Rotation`, `Scale`, Camera, Light, Mesh Renderer, and Sprite Renderer controls plus axis delta buttons in the Viewport toolbar. These edits are applied to the active `SceneAuthoringDocument` through `make_scene_authoring_transform_edit_action` and `make_scene_authoring_component_edit_action`, preserving selected-node state and undo/redo history without a parallel raw-scene editing model.

## Undo and Redo

Editor mutations that change document state should flow through `mirakana::editor::UndoStack`.

Rules:

- Use concise labels such as `Add Empty Node`.
- `redo` performs the new state.
- `undo` restores the previous state.
- Executing a new action clears redo history.
- View-only toggles may remain regular commands until layout history is intentionally supported.

A future visible desktop editor should wire scene replacement actions through this stack for New Scene, Add Empty Node, transform edits, Undo, and Redo.
