# Editor

## Current Shape

The editor is split into two layers:

- `MK_editor_core`: GUI-independent project documents, project shader tool settings, shader tool discovery state, project render backend preference/availability selection, project settings draft validation, project creation wizard state, project/workspace migration, project bundle IO, text stores, workspace state, panel visibility, workspace serialization, content browser state, asset import queue state, asset import execution result mapping including scene imported asset records, imported asset record registration helpers, asset hot reload event, recook request, applied result, failed rollback diagnostic state, and hot reload diagnostic labels, material authoring documents/models for first-party material factor and texture-slot edits, scene/prefab authoring documents, play-in-editor session isolation, prefab variant authoring documents/models for name/transform/component overrides plus read-only conflict review rows, shader compile queue/status state, shader compile execution result mapping, viewport and material-preview shader artifact request/readiness state, viewport state, viewport run controls, viewport tool state, scene transform drafts, scene component drafts, viewport transform edit application, command registry, command palette query model, first `MK_editor_ui` retained models backed by `MK_ui`, timeline control/event-row models backed by `MK_animation`, recorder-backed profiler panel models backed by `MK_core` diagnostics, resource panel models backed by caller-supplied plain RHI/resource diagnostics plus reviewed capture request and host-owned capture execution evidence rows, read-only AI command diagnostics models over existing AI workflow rows, deterministic editor UI model serialization, dirty tracking, and undo/redo history.
- `MK_editor`: optional SDL3 + Dear ImGui desktop shell.
- Native `.scene` open/save dialog review is split the same way: `MK_editor_core` builds/reviews `FileDialogRequest` / `FileDialogResult` data and retained `scene_file_dialog.open` / `scene_file_dialog.save` rows, while `MK_editor` owns SDL3 polling, path conversion, and scene load/save IO.

The default build tests `MK_editor_core` without third-party GUI dependencies. The optional `desktop-gui` preset builds and tests the SDL3/ImGui shell.

On Windows, when `MK_editor` is enabled, CMake links `MK_rhi_vulkan` and defines `MK_EDITOR_ENABLE_VULKAN`. The shell can construct a headless `VulkanRuntimeDevice` and bridge it to `IRhiDevice` using `mirakana::rhi::vulkan::minimal_irhi_device_mapping_plan` when the Vulkan loader probe succeeds and viewport SPIR-V shader artifacts are ready; viewport and material GPU preview shader bytecode follows the active render backend (DXIL for D3D12, SPIR-V for Vulkan). SDL GPU texture display for the Dear ImGui viewport remains primarily exercised through the D3D12 shared-texture path.

Editor-core package diagnostics include `mirakana::editor::make_editor_tilemap_package_diagnostics_model`, which summarizes selected cooked `GameEngine.Tilemap.v1` package rows through `MK_runtime` payload inspection and deterministic visible-cell sampling without executing package scripts, mutating manifests, depending on Dear ImGui, or exposing renderer/RHI handles.

## Play-In-Editor

Play-In-Editor Session Isolation v1 is represented in `MK_editor_core` by `mirakana::editor::EditorPlaySession`. Starting a session snapshots the active `SceneAuthoringDocument` source scene into an isolated simulation scene copy, exposes begin/pause/resume/tick/stop transitions, reports the source and simulation node counts through `EditorPlaySessionReport`, blocks source-scene edit claims while active, counts simulation ticks only in play state, and discards the simulation copy on stop without writing it back to the authored scene.

Play-In-Editor Gameplay Driver v1 adds `mirakana::editor::IEditorPlaySessionDriver` and `mirakana::editor::EditorPlaySessionTickContext`. A caller may start a session with a reviewed driver; editor core invokes `on_play_begin`, `on_play_tick`, and `on_play_end` against the isolated simulation scene only. Tick calls require a finite positive delta, include the current frame index and delta seconds in the context, update `last_delta_seconds`, and reject ticks while paused.

Play-In-Editor Visible Viewport Wiring v1 adds `mirakana::editor::EditorPlaySessionControlsModel` and command rows for Play/Pause/Resume/Stop availability. The visible `MK_editor` Run menu and Viewport toolbar use that model to start, pause, resume, and stop `EditorPlaySession`, render the isolated simulation scene while active, tick the play session once per viewport frame in play state, and reject source-scene authoring plus undo/redo while `source_scene_edits_blocked()` is true.

## UI Strategy

The current `MK_editor` target intentionally uses Dear ImGui as an optional developer-facing shell for fast iteration, diagnostics, and early tools. It is not the production runtime UI foundation.

Coding conventions for the visible shell (`editor/src/main.cpp` plus companion `editor/src/material_preview_gpu_cache.cpp` / `sdl_viewport_texture.cpp`)â€”ImGui varargs vs `std::string_view`, Windows TU include hygiene, environment-variable reads, material GPU preview and SDL viewport bridge (`editor/include/mirakana/editor/material_preview_gpu_cache.hpp`, `sdl_viewport_texture.hpp`), optional-RHI backend includes kept in `.cpp` when the public header does not need them (`misc-include-cleaner`), tracked **`editor/src/compile_flags.txt`** and **`editor/include/compile_flags.txt`** for clangd when headers or TUs miss `compile_commands.json`, and targeted `check-tidy`â€”are documented under **Testing** and **AI Development Workflow** in `AGENTS.md` and **Dear ImGui shell** / **MK_editor Windows and material-preview host cache** in `.claude/skills/gameengine-editor/SKILL.md` (Codex twin: `.agents/skills/editor-change/SKILL.md`); see also **Desktop editor shell** in `docs/cpp-style.md`. For **`editor/core`** (GUI-independent sources), follow **Editor core C++** in the same editor skills plus **Unit tests** / production adapter notes in `docs/cpp-style.md` (`std::string_view::contains` / `starts_with`, Rule of Five on C API adapters, `check-tidy.ps1 -Files editor/core/src/<file>.cpp`).

UI work is tracked in `docs/ui.md` and Wave 8 of `docs/superpowers/plans/2026-04-26-engine-excellence-roadmap.md`: first-party retained-mode `MK_ui` contracts cover game HUDs and menus, while `MK_editor_ui` models cover complex editor panels. This follows the same separation of concerns as Unreal's Slate/UMG/Common UI and Unity's UI Toolkit/uGUI/IMGUI families: editor tooling, runtime game UI, and debug/immediate-mode surfaces are separate systems.

Rules:

- Runtime game code must not depend on `editor`, SDL3, or Dear ImGui.
- New complex editor panels should first add GUI-independent state/model contracts in `editor/core`, then expose durable retained models through `MK_editor_ui` before wiring a Dear ImGui adapter.
- Do not hand-roll mature low-level font, text shaping, IME, accessibility, image decoding, or platform integration systems inside editor panels; route them through engine adapters after an accepted dependency/license decision.
- Do not add Qt, NoesisGUI, Slint, RmlUi, fonts, or UI middleware until `license-audit`, `docs/dependencies.md`, `vcpkg.json`, and `THIRD_PARTY_NOTICES.md` are updated.

## Project Text Format

The current project format is deterministic line-based text:

```text
format=GameEngine.Project.v4
project.name=sample
project.root=games/sample
project.asset_root=assets
project.source_registry=source/assets/package.geassets
project.game_manifest=game.agent.json
project.startup_scene=scenes/start.scene
project.render_backend=auto
project.shader_tool.executable=dxc
project.shader_tool.working_directory=.
project.shader_tool.artifact_output_root=out/editor/shaders
project.shader_tool.cache_index=out/editor/shaders/shader-cache.gecache
```

Rules:

- The first supported format version is `GameEngine.Project.v1`.
- The current write format is `GameEngine.Project.v4`.
- All current fields are required when authoring a new project file.
- Project fields must not contain newlines or `=`.
- Project paths must be project-relative and must not contain `..` traversal.
- `project.source_registry` is a safe project-relative `.geassets` path. New projects default it to `source/assets/package.geassets`.
- Shader tool executable values must be a PATH-resolved tool name such as `dxc`, a project-relative reviewed tool path, or a reviewed absolute installed SDK/tool path. They must not contain `..` traversal, and shell executables are rejected.
- Shader tool working directory, artifact output root, and cache index path are project-relative and must not contain `..` traversal.
- Render backend preference values are `auto`, `null`, `d3d12`, `vulkan`, or `metal`.
- Unknown keys and unsupported formats fail fast.
- `migrate_project_document` upgrades `GameEngine.Project.v0` by filling the default asset root, game manifest path, and startup scene path.
- `migrate_project_document` upgrades `GameEngine.Project.v1` by filling default shader tool settings.
- `migrate_project_document` upgrades `GameEngine.Project.v2` by filling the default `auto` render backend preference.
- `migrate_project_document` upgrades `GameEngine.Project.v3` by filling the default project source registry path.

## Workspace Text Format

The current workspace persistence format is deterministic line-based text:

```text
format=GameEngine.Workspace.v1
project.name=sample
project.root=games/sample
panel.scene=visible
panel.inspector=visible
panel.assets=hidden
panel.console=visible
panel.viewport=visible
panel.resources=hidden
panel.ai_commands=hidden
panel.profiler=visible
panel.project_settings=hidden
panel.timeline=hidden
```

Rules:

- The first supported format version is `GameEngine.Workspace.v1`.
- Project fields must not contain newlines or `=`.
- Panel values are `visible` or `hidden`.
- Unknown keys, unknown panels, duplicate panel entries, and unsupported formats fail fast.
- `migrate_workspace` upgrades `GameEngine.Workspace.v0` into the current default panel set before applying stored panel visibility overrides.

These formats are intentionally small until the editor project model is ready for a stable JSON schema and a reviewed JSON dependency.

## Project Creation

`mirakana::editor::ProjectCreationWizard` owns the GUI-independent new-project flow.

Rules:

- Steps are `Identity`, `Paths`, and `Review`.
- Project name and root path are required before leaving `Identity`.
- Asset root, source registry path, game manifest path, and startup scene path default to `assets`, `source/assets/package.geassets`, `game.agent.json`, and `scenes/start.scene`.
- Draft fields reject newlines and `=` so they can safely become `GameEngine.Project.v4` fields.
- `create_project_document` validates through the same project document rules used by project file IO.
- The desktop editor exposes this through `File > New Project` and creates a clean workspace plus startup scene in memory before saving to disk.

## Project Settings Drafts

`mirakana::editor::ProjectSettingsDraft` owns editable project settings before they mutate the active `ProjectDocument`.

Rules:

- Create drafts with `ProjectSettingsDraft::from_project`.
- Draft setters update the candidate shader tool executable, working directory, artifact output root, cache index path, and render backend preference.
- `set_shader_tool_descriptor` copies a discovered `mirakana::ShaderToolDescriptor` executable into the draft while leaving path validation to `apply`.
- `validation_errors` reports field-specific editor errors without mutating the source project.
- `apply` returns a validated `ProjectDocument`; invalid settings throw and must not be written into the active project.
- The desktop editor uses this model in the Project Settings panel, then rebuilds shader compile requests when shader tool paths change and recomputes the viewport backend selection when the render backend preference changes.

## Project Bundle IO

`mirakana::editor::save_project_bundle` and `mirakana::editor::load_project_bundle` persist three deterministic text documents:

- project document
- workspace state
- startup scene text

Storage is abstracted through `ITextStore` so tests and agents can use `MemoryTextStore`, while the desktop editor can use `FileTextStore`. `FileTextStore` only accepts root-relative paths and rejects absolute paths or `..` traversal.

`DocumentDirtyState` tracks editor document changes by revision. Save operations mark the current revision clean; editing, undo, and redo mark the document dirty.

Editor Project Native Dialog v1 adds `EditorProjectFileDialogModel`, `EditorProjectFileDialogMode`, `make_project_open_dialog_request`, `make_project_save_dialog_request`, `make_project_open_dialog_model`, `make_project_save_dialog_model`, and `make_project_file_dialog_ui_model`. The model reviews accepted/canceled/failed `.geproject` open/save results, rejects empty, multi-select, and wrong-extension selections, and emits retained `project_file_dialog.open` / `project_file_dialog.save` rows without bringing SDL3, Dear ImGui, native handles, filesystem IO, or project-store path conversion into editor core. The visible desktop editor exposes `Open Project...` and `Save Project As...` from the File menu and Project Settings panel through `mirakana::SdlFileDialogService`; accepted in-store `.geproject` selections are converted to safe store-relative paths before reusing `load_project_bundle` / `save_project_bundle`.

`mirakana::editor::save_shader_artifact_manifest` and `mirakana::editor::load_shader_artifact_manifest` persist deterministic shader artifact manifests through the same `ITextStore` abstraction, including shader descriptor reflection rows for set, binding, resource kind, stage, count, and semantic. Paths must be root-relative and must not contain absolute segments, `..`, newlines, or `=`.

## Viewport

`mirakana::editor::ViewportState` owns GUI-independent viewport readiness, frame counters, play state, active transform tool, and resolved render backend selection.

Rules:

- Run mode is `Edit`, `Play`, or `Paused`; simulation ticks advance only while in `Play`.
- Stopping play mode returns to `Edit` and clears the simulation frame counter.

## Undo and Redo

Rules:

- Use concise labels such as `Add Empty Node`.

- Executing a new action clears redo history.
