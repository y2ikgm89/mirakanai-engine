# GameEngine Game Development

## Workflow

1. Run or inspect `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1 -ContextProfile Minimal` or `Standard` first; use `Full` only when generated-game/API decisions need the complete manifest-shaped output.
2. Read targeted `engine/agent/manifest.fragments/*.json` or the composed `engine/agent/manifest.json` for engine capabilities. Engine contract maintenance uses fragments + `tools/compose-agent-manifest.ps1 -Write` (gameplay work normally only **reads** the composed manifest).
3. Read `docs/cpp-style.md` before adding public names or new files.
4. Create a headless game with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name>` when starting fresh (shared string/format helpers live in `tools/new-game-helpers.ps1`, dot-sourced by `new-game.ps1`; edit both when changing those helpers), use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntimePackage` for a config-only package-ready SDL3 desktop runtime scaffold, use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntimeCookedScenePackage` for a package-ready scaffold with first-party cooked scene/material/mesh/texture files and `--require-scene-package` validation, use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntime2DPackage` for a generated host-gated packaged 2D gameplay scaffold with cooked sprite/material/audio/orthographic scene payloads, use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntime3DPackage` for a generated host-gated packaged 3D gameplay scaffold with cooked texture/mesh/morph_mesh_cpu/material/animation_float_clip/animation_quaternion_clip/perspective scene/physics-collision payloads, transform/morph/quaternion scene-transform package smoke counters, selected safe-point package streaming smoke counters, selected scene collision package smoke counters through `--require-scene-collision-package`, selected generated 3D renderer quality counters, selected generated 3D directional shadow package smoke counters, selected D3D12 native UI overlay HUD box smoke counters, and primary camera/controller package smoke, or use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/new-game.ps1 -Name <game_name> -Template DesktopRuntimeMaterialShaderPackage` only when the scaffold needs source material/HLSL authoring inputs plus host-built desktop shader artifacts.
5. Read or write `games/<game_name>/game.agent.json`.
6. Keep `game_name` and `new-game -Name` values matching `^[a-z][a-z0-9_]*$`; source-tree game directories and `runtimePackageFiles` path segments must be lowercase snake_case.
7. JSON manifest IDs, display names, and external package identifiers may use ecosystem formats, including kebab-case when required.
8. Keep `backendReadiness`, `importerRequirements`, `packagingTargets`, and `validationRecipes` aligned with the selected engine manifest capabilities.
9. Keep gameplay code under `games/<game_name>/`.
10. Use only public `mirakana::` headers unless the task explicitly changes the engine.
11. Prove the smallest externally meaningful gameplay/package guarantee with focused tests or executable output while iterating, then validate the coherent slice with `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1`.

## Current Runtime

- Implement `mirakana::GameApp`.
- Run with `mirakana::HeadlessRunner` for default source-tree validation.
- Use `mirakana::SdlDesktopGameHost` when a game selects the optional `desktop-game-runtime` lane and needs the reusable SDL3 windowed host path.

- Treat most `games/sample_*` executables as headless proof programs. `sample_input_renderer` uses `mirakana::NullRenderer`; running it proves renderer submissions and input state but does not open a window. `sample_ai_navigation` is the headless generated-game proof that composes `mirakana_ai` perception blackboard projection and blackboard-driven behavior tree conditions with `mirakana_navigation` `plan_navigation_grid_agent_path` output and value-type agent movement through public APIs only. `sample_desktop_runtime_shell` is the optional windowed runtime shell sample and uses `mirakana::SdlDesktopGameHost`, which owns SDL runtime/window/event-pump/presentation/runner wiring;
