# GameEngine Agent Integration

## Rules

- `AGENTS.md` is the shared baseline for Codex and other AGENTS-aware tools; long-form validation and workflow tables live in `docs/agent-operational-reference.md`. Cursor loads `.cursor/rules/*.mdc` plus `.cursor/skills/`.
- `CLAUDE.md` imports `AGENTS.md` and project rule files for Claude Code.
- `engine/agent/manifest.json` is the machine-readable **canonical** engine contract (compose output only). **Edit** `engine/agent/manifest.fragments/*.json` (see `engine/agent/manifest.fragments/README.md` and `docs/adr/0002-agent-manifest-fragments-compose.md`), then run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/compose-agent-manifest.ps1 -Write`. `tools/check-json-contracts.ps1` verifies the committed canonical file matches compose output.
- `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/agent-context.ps1` prints the current agent context as JSON, including public headers, module ownership, sample games, asset/importer capabilities, platform targets, and validation recipes. Use `-ContextProfile Full|Standard|Minimal` (default `Full`) when you want a smaller summary.
- `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-ai-integration.ps1` validates agent-facing files.
- thin static-contract ledger entrypoints: `tools/static-contract-ledger.ps1`.
- Every implementation change, improvement, bug fix, refactor, and architecture/toolchain/workflow/validation/packaging change includes a targeted **agent-surface drift check** before completion. When durable guidance or AI-operable contracts are stale, update the relevant `AGENTS.md`, `CLAUDE.md`, docs, Codex/Claude/Cursor skills, rules, settings, subagents, manifest fragments plus compose output, schemas, validation checks, and tracked `.clangd` in the same task; do not leave that synchronization as a separate follow-up or load every agent surface when no durable guidance changed.
- **`MK_tools` paths:** Implementation `.cpp` files live under `engine/tools/{shader,gltf,asset,scene}/`; public headers stay `engine/tools/include/mirakana/tools/` (see `docs/adr/0003-directory-layout-clean-break.md`, `docs/specs/2026-05-11-directory-layout-target-v1.md`). Moving those sources requires updating `tools/check-json-contracts.ps1`, `tools/check-ai-integration.ps1`, and the matching `engine/tools/*/CMakeLists.txt` needles in the same change. When new tool implementation code depends on additional `mirakana/*` modules, add the corresponding `MK_*` to that **cluster's** `OBJECT` target in `engine/tools/<cluster>/CMakeLists.txt`; the `MK_tools` umbrella keeps the full `PUBLIC` link set for consumers (spec invariant 4).
- Every tracked `tools/*.ps1` starts with `#requires -Version 7.0` then `#requires -PSEdition Core`; use **`pwsh`** only. Scripts are **UTF-8 without BOM** and must parse as PowerShell; `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-agents.ps1` enforces both. `tools/check-format.ps1` / `tools/format.ps1` cover C++ `clang-format` plus tracked text; `tools/check-text-format.ps1` / `tools/format-text.ps1` are text-only loops for LF, UTF-8 without BOM, one final newline, and no EOF blanks. Use PowerShell approved verbs so **PSScriptAnalyzer** stays clean; prefer **`ConvertTo-*` / `Join-*` / `Resolve-*`** over **`Normalize-*`**. Shared helpers live in `tools/common.ps1` (`Get-RepoRoot` returns a **string**; use `$root`, never `$root.Path`).
- **PSScriptAnalyzer hygiene for `tools/*.ps1`:** do not reuse **`$input`**, **`$matches`**, **`$IsWindows` / `$IsMacOS` / `$IsLinux`**, or other automatic variables as locals; use **`$null = ...`** for intentional discard; prefer **`Write-Information ... -InformationAction Continue`** over **`Write-Host`**; avoid empty **`catch { }`**; use **`SupportsShouldProcess`** and **`ShouldProcess`** gates for host-visible mutation. Match **`AGENTS.md` -> Repository command entrypoints** and run **`Invoke-ScriptAnalyzer`** when available.
- `tools/check-agents.ps1` also enforces Codex ↔ Claude `gameengine-*` skill folder twins via `claudeToCodexSkillMap`, initial-load budgets for `AGENTS.md`, Codex/Claude/Cursor `SKILL.md`, and Codex/Claude subagents, and Cursor thin-pointer `gameengine-*` folders matching `.claude/skills/` names except `gameengine-cursor-baseline` and `gameengine-plan-registry`. Register new pairs when adding shared skills; see `docs/workflows.md` **Repository consistency checklist**.
- `.codex/rules/*.rules` contains project-local Codex escalation policy. Keep `match` / `not_match` rules narrow, aligned with **PowerShell 7** (`pwsh`), and never broad. Agent publishing follows official GitHub Flow: commit task-owned validated phases, push non-forced topic-branch checkpoints after remote-state inspection, keep one PR per focused capability/gap-cluster/milestone, use early `--draft` PRs, and avoid PRs per commit/checklist item. Task-owned `gh pr create`, `gh pr merge --auto --merge --delete-branch`, and guarded `tools/remove-merged-worktree.ps1` cleanup run after validation/merge evidence plus PR preflight (`state`, `isDraft`, `baseRefName`, `headRefOid`, `mergeable`, `mergeStateStatus`, `reviewDecision`, `statusCheckRollup`); pending-only `UNSTABLE` / `BLOCKED` may proceed, prefer `--match-head-commit <headRefOid>`. A final completion report must not stop after local validation when task-owned changes can be published. Direct default-branch pushes are forbidden; force-push, immediate merges, `gh pr ready`, raw worktree cleanup, and other PR state changes remain prompt-gated. Keep merge/delete-branch, post-merge remote-tracking cleanup, post-merge worktree cleanup, and checkout fast-forward examples in docs. Treat command policy as session-scoped; after rule edits, rely on policy reload or a new session, and use GitHub Web/Desktop or an approval-capable session when approvals are unavailable.
- `.claude/settings.json` contains shared Claude Code project permissions with the official JSON schema. Deny secret-bearing files and direct default-branch pushes; allow read-only PR preflight, task-owned PR creation including draft PR creation, auto-merge registration after validation checkpoints and official GitHub Flow PR preflight, and guarded merged-worktree cleanup with local checkout fast-forward; ask before raw destructive cleanup, network, dependency-bootstrap, mobile signing/smoke, force-push, immediate PR merge, `gh pr ready`, or other GitHub PR state-change commands. Keep `.claude/settings.local.json`, `.mcp.json`, and `AGENTS.override.md` uncommitted unless project governance intentionally changes that.
- CI check selection changes must keep branch-protection-required checks always-running or behind an always-running aggregate gate. Do not make path-filtered required checks. `tools/classify-pr-validation-tier.ps1` owns PR touched-surface classification and `tools/check-ci-matrix.ps1` exercises docs-only, static-policy, runtime, workflow, and non-PR cases. Run the full hosted matrix for default-branch/release/scheduled/workflow-dispatch runs; for PRs, use the cheapest proving tier. Docs/agent/rules/subagent-only changes run formatting plus agent/static guards, not Windows/MSVC, macOS, or full repository clang-tidy unless they also change CI execution, CMake/vcpkg, build/test/package tooling, runtime code/assets, packaging, or public/runtime contracts.
- Hosted PR failure hardening: when a PR/CI check fails because guidance or a static guard drifted, update the relevant `AGENTS.md`, docs, Codex/Claude/Cursor skills, subagents, and scoped `tools/check-ai-integration.ps1` Needles together. Static-analysis drift includes `.clang-tidy` `HeaderFilterRegex`, strict `--warnings-as-errors=*`, `NN warnings generated.` handling, and hosted `-Jobs 0` parallelism. Keep `.codex/rules` and `.claude/settings.json` as command/permission gates, not troubleshooting playbooks.
- Git/GitHub authentication stays host-local through Git Credential Manager, GitHub CLI, SSH agent, or a browser session; do not add repository requirements for `GITHUB_TOKEN` or personal access tokens. If warnings mention missing helpers such as `credential-manager-core`, inspect `git config --show-origin --get-all credential.helper`, prefer current Git for Windows GCM helper `manager`, and fix host/user Git config rather than adding repository overrides.
- Worktree/parallel development follows official surfaces: prefer Codex app Worktree/Handoff or Claude Code `--worktree` / subagent `isolation: worktree`; keep `.worktrees/` and `.claude/worktrees/` ignored; run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/prepare-worktree.ps1` in manual linked worktrees; use Claude `worktree.baseRef = "head"` when isolated subagents need current branch context; use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/remove-merged-worktree.ps1 -WorktreePath <path> [-BaseRef origin/main] [-BaseBranch main] [-Remote origin] [-LocalCheckoutPath <path>] [-DeleteLocalBranch]` for guarded post-merge local checkout fast-forward plus worktree cleanup instead of raw deletion; Windows long-path fallback must remain inside that guarded script after clean/merged/root checks and must not follow reparse points.
- Cursor Agent Skills live under `.cursor/skills/`; use thin `gameengine-*` folders that name-match `.claude/skills/` and point at Claude/Codex skill routers and detailed references. Cursor-only summaries stay in `gameengine-cursor-baseline` and `gameengine-plan-registry`. Keep content aligned with `AGENTS.md` when validation or toolchain defaults change. Tracked `.clangd` points clangd at `out/build/dev` after `cmake --preset dev`.
- `docs/README.md` is the documentation entrypoint.
- When **`editor/core`** public aggregates such as **`mirakana::editor::ScenePrefabInstanceRefreshPolicy`** gain members, refresh **`tests/unit/editor_core_tests.cpp`** (and any other designated braced sites) in the **same task** with **full declaration-order designated initializers** per **`docs/cpp-style.md`** (**Unit tests** → **Aggregate literals**) so clang/clangd `-Wmissing-field-initializers` stays clean.
- For **AI-driven Windows D3D12 GPU diagnostics** (Resources panel is review/evidence only; PIX is operator-installed; optional `tools/launch-pix-host-helper.ps1` resolves `WinPix.exe` / legacy `PIX.exe` or `MIRAKANA_PIX_EXE`; `-SkipLaunch` for path smoke), read `docs/ai-integration.md` § **AI-driven workflows and Windows GPU diagnostics** and follow § **Recommended workflow (operator PIX, AI analysis)**—bundle `pixtool --help`, optional `-SkipLaunch`, then operator UI launch and capture; analyze only from operator-supplied evidence.
- `docs/superpowers/plans/README.md` is the implementation plan registry; read it before creating, editing, or extending plans.
- Keep the live plan stack shallow: one active roadmap, one active gap-cluster burn-down or milestone, and at most one active phase/child plan selected by `currentActivePlan`.
- Create dated capability/gap-cluster/milestone plans; put phase behavior/API/validation boundary decisions, validation-only follow-up, docs/manifest/static-check sync, small cleanup, and checklist substeps inside the active plan.
- Preserve completed plans as historical implementation evidence and reduce navigation noise through the registry or evidence indexes rather than deleting traceability.
- A phase-gated milestone plan is allowed when tightly related production slices share one end-to-end objective; each phase still needs focused evidence and validation.
- Treat checkboxes as fine-grained execution steps, not plan-file boundaries. Plan files may be broader than phase/PR width: prefer capability/gap-cluster/milestone plans over tiny dated plans when work shares one architecture decision, public API family, validation surface, and review purpose; create child plans only for distinct architecture/validation/review boundaries or phases too large for one safe context.
- Keep tracked instructions specific, concise, verifiable, and durable. Keep `AGENTS.md` below Codex's default 32 KiB project-doc budget, keep selected `SKILL.md` files as concise routers, and keep subagents narrowly scoped; `tools/check-agents.ps1` enforces these budgets. Put long procedures in skill-local `references/*.md` or docs, path-specific guidance in rules, specialized behavior in subagents, and machine-readable status claims in the composed `engine/agent/manifest.json` (maintain via `engine/agent/manifest.fragments/` + `tools/compose-agent-manifest.ps1 -Write`).
- Keep agent-driven development efficient: use targeted reads/manifest fragments, targeted drift checks, focused target builds/tests, and file-owned static checks during implementation. Run full `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` only for C++/runtime/build/packaging/public-contract slice gates, or when narrower checks cannot prove the changed surface.
- Keep tests scoped to the smallest externally meaningful behavior/API/regression guarantee. Prefer existing-test updates when they already cover the contract; avoid implementation-mirroring, duplicate, or incidentally order-sensitive tests.
- For production-completion work, drive from `currentActivePlan`, `recommendedNextPlan`, and `unsupportedProductionGaps`; re-read the master plan, registry, and manifest after user edits or context resumes; burn down one selected gap at a time; keep clean breaking greenfield designs, official documentation, focused validation, and actual evidence ahead of compatibility shims or broad ready claims; keep context small with focused reads/tests and use bounded independent subagents only when they improve speed or confidence.
- Each `unsupportedProductionGaps` row includes `oneDotZeroCloseoutTier` (`foundation-follow-up`, `package-evidence`, or `closeout-wedge`) as machine-readable 1.0 closeout grouping alongside `status` and `requiredBeforeReadyClaim`; keep it aligned with the master plan when re-scoping gaps (edit `engine/agent/manifest.fragments/010-aiOperableProductionLoop.json`, then `tools/compose-agent-manifest.ps1 -Write`).
- `docs/superpowers/plans/2026-05-03-production-completion-master-plan-v1.md` may end with an HTML comment archive of historical verdict needles for `tools/check-ai-integration.ps1`; do not delete or hollow that block without running `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` afterward.
- Do not loosen `.codex/rules` or `.claude/settings.json` approval gates merely to save time; speed should come from narrower work slices and focused validation, not broader command trust.
- Use Context7 MCP for live library, SDK, build-system, and toolchain documentation lookups. Keep Context7 API keys in user-level configuration only.
- Use the OpenAI developer documentation MCP, or official OpenAI documentation when MCP is unavailable, for OpenAI API, Codex, ChatGPT Apps SDK, OpenAI agent, and OpenAI model questions. Use official Anthropic documentation for Claude Code memory, settings, permissions, hooks, skills, and subagent behavior.
- Game source naming guidance must stay synchronized: `game_name` and `new-game -Name` values match `^[a-z][a-z0-9_]*$`, source-tree game directories and `runtimePackageFiles` path segments stay lowercase snake_case, and JSON manifest IDs, display names, or external package identifiers may keep ecosystem formats including kebab-case.

## When Changing Integration

**Engine manifest:** edit `engine/agent/manifest.fragments/*.json`, run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/compose-agent-manifest.ps1 -Write`, and keep `schemas/` / `tools/check-ai-integration.ps1` needles aligned when literals change; never hand-edit `engine/agent/manifest.json`.

1. Update Codex (`.agents/skills/`), Claude Code (`.claude/skills/`), and Cursor (`.cursor/skills/`, `docs/ai-integration.md`) surfaces when cross-tool behavior changes; keep overlapping `.agents` / `.claude` skill bodies equivalent.
2. Keep always-loaded files, selected `SKILL.md` bodies, and subagent instructions within `tools/check-agents.ps1` budgets.
3. Put reusable procedures in skills, and move deep procedures to skill-local `references/*.md` when a `SKILL.md` would stop being a concise router.
4. Put specialized workers in `.codex/agents` and `.claude/agents`.
5. If implementation work reveals missing or stale agent guidance or AI-operable contract claims, update the relevant instructions, docs, skills, rules, settings, subagents, manifests, schemas, and validation checks in the same task before reporting completion.
6. Keep reviewer, explorer, architect, and auditor subagents read-only by default; write-capable tools belong only on builder/fixer roles that are expected to change files.
7. Keep engine/game manifest `runtimeBackendReadiness`, `importerCapabilities`, `packagingTargets`, `runtimePackageFiles`, `runtimeSceneValidationTargets`, and `validationRecipes` honest: do not mark planned or dependency-gated work as implemented.
8. When public `mirakana::` diagnostics, counters, profile zones, trace export, or other cross-module capabilities change, update the engine agent manifest (`engine/agent/manifest.fragments/*.json` + `tools/compose-agent-manifest.ps1 -Write`), generated-game guidance, and Codex/Claude subagents so agents do not place shared observability in `mirakana_runtime`, renderer, platform, editor, SDL3, or backend modules by mistake.
9. When dependency bootstrap behavior changes, update `AGENTS.md`, CMake/license/feature skills, build-fixer subagents, the engine agent manifest (`engine/agent/manifest.fragments/*.json` + `tools/compose-agent-manifest.ps1 -Write`), and validation checks together. Keep `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1` as the vcpkg install entrypoint and keep CMake configure lanes on `VCPKG_MANIFEST_INSTALL=OFF`.
10. When CMake/toolchain preflight or static-analysis behavior changes, update `AGENTS.md`, CMake skills, build-fixer subagents, Claude rules, the engine agent manifest (fragments + `compose-agent-manifest.ps1 -Write`), validation checks, `.cursor/skills/`, and `.clangd` when needed. Keep `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/prepare-worktree.ps1` as linked-worktree readiness, `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` as the CMake/CTest/CPack/`clang-format`/Visual Studio/MSBuild guard for visible configure/build presets inheriting `normalized-configure-environment` / `normalized-build-environment`, and `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1 -RequireDirectCMake` only for direct CMake PATH requirements. Keep `direct-clang-format-status`, `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1`, CMake File API synthesis, `.clang-tidy` `HeaderFilterRegex`, scoped `check-tidy.ps1 -Files`, and `tools/check-agents.ps1` script hygiene aligned.
11. When Windows host diagnostics guidance changes, update `docs/dependencies.md`, `docs/workflows.md`, `docs/testing.md`, `AGENTS.md`, Codex/Claude debugging/rendering/CMake skills, build-fixer and rendering-auditor subagents, `.codex/rules`, `.claude/settings.json`, the engine agent manifest (`engine/agent/manifest.fragments/*.json` + `tools/compose-agent-manifest.ps1 -Write`), schemas, and validation checks together. Keep Debugging Tools for Windows, Windows Graphics Tools, PIX on Windows, and Windows Performance Toolkit as host-local diagnostics, not default build dependencies.
12. For new production work, update `docs/superpowers/plans/README.md` and prefer one dated capability/gap-cluster/milestone plan. Put phase behavior/API/validation boundary decisions, validation-only follow-up, docs/manifest/static-check synchronization, small cleanup, and substeps inside the active plan; create child plans only for distinct architecture/validation/review boundaries or phases too large for one safe context. Plan width may exceed PR width, but phases/PRs stay one reviewable purpose. Avoid copying long completed-slice prose into active sections or `recommendedNextPlan.completedContext`; do not extend completed historical plans, delete evidence to reduce file count, broaden ready claims, or weaken host gates.
13. When mobile packaging changes, keep Android and Apple host gates distinct. Android GameActivity has host-validated Debug/Release/signing/emulator smoke lanes on configured hosts; Apple/iOS remains macOS/Xcode gated. Update `AGENTS.md`, Codex/Claude game-development skills, gameplay/build subagents, `docs/ai-game-development.md`, generated-game validation scenarios, and the engine agent manifest (`engine/agent/manifest.fragments/*.json` + `tools/compose-agent-manifest.ps1 -Write`) together.
14. When reviewed AI-operable content mutation or metadata-repair surfaces change, such as source asset registration, scene/prefab authoring, runtime package registration, material instance package updates, material graph package binding, shader pipeline cache metadata reconciliation, or UI atlas package updates, update `docs/ai-game-development.md`, Codex/Claude game-development skills, relevant subagents, the engine agent manifest (`engine/agent/manifest.fragments/*.json` + `tools/compose-agent-manifest.ps1 -Write`), schema/static checks, and validation recipes together. Keep arbitrary shell, free-form edit/eval, importer execution, shader compiler execution, cooked package writes, renderer/RHI residency, and package streaming claims rejected unless the reviewed surface implements and validates them.
15. When Git/GitHub publishing workflow changes or PR CI check selection changes, update `AGENTS.md`, `docs/workflows.md`, `docs/testing.md`, Codex/Claude/Cursor skills, command-policy rules/settings only when permissions change, relevant subagents, and `tools/check-ai-integration.ps1` needles together; keep purpose/checkpoint-based cadence: validated phase commits, checkpoint pushes after remote-state inspection, one PR per focused capability/gap-cluster/milestone, early draft PRs, PR create/auto-merge registration, and guarded cleanup after official GitHub Flow PR preflight. Keep branch-protection-required checks always-running or aggregate-gated, docs/agent-only PRs use lightweight static validation, direct default-branch pushes forbidden, force-push/immediate-merge/ready/other PR state changes prompt-gated, and merge/delete-branch plus stale remote-tracking cleanup documented.
16. When worktree or parallel-agent workflow changes, update `AGENTS.md`, `docs/workflows.md`, `docs/ai-integration.md`, Codex/Claude/Cursor skills, `.gitignore`, `.codex/rules`, `.claude/settings.json`, Claude write-capable subagents, and static checks together; prefer native Codex/Claude worktree surfaces over manual fallback, keep raw worktree cleanup reviewed, and expose only guarded post-merge cleanup for automatic removal.
17. Run the smallest relevant agent/static check while iterating, usually `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-ai-integration.ps1` for manifest or instruction changes, then run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` at the slice-closing gate.
18. When extending `tools/check-ai-integration.ps1` with new Needles for retained editor UI ids, manifest slices, or editor-shell literals, edit the scoped hashtable entry for the target `Path` (for example `engine/agent/manifest.json`, `.agents/skills/editor-change/SKILL.md`, `.claude/skills/gameengine-editor/SKILL.md`, `editor/src/main.cpp`) so each needle remains unambiguous; update sibling skill/manifest text in the same task.
