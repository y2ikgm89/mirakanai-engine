Games live under `games/<game_name>/` with `game.agent.json`; see `AGENTS.md` **AI-Driven Game Development**.

Runtime package payloads are byte-hashed. New text cooked/runtime extensions or `runtimePackageFiles` entries need matching `runtime/.gitattributes` `text eol=lf`, scaffold/static-check parity, and the narrowest package smoke before slice completion.

Missing generated-game art/audio should use reviewed engine tooling such as `mirakana::PlaceholderAssetBundleRequest` / `mirakana::plan_placeholder_asset_bundle` to return a `PlaceholderAssetBundlePlan` with deterministic first-party placeholder source documents, changed-file hashes, provenance, and fail-closed diagnostics; package-authoring workflows can use `mirakana::PlaceholderAssetCookPackageRequest` / `mirakana::plan_placeholder_asset_cook_package` to route generated source documents through registered source cook/package planning. Do not download external assets, bypass cook/package validation, or parse source assets at runtime.

Reviewed generated 2D source atlas rows should use `mirakana::SpriteAtlasSourceFrameDesc` / `mirakana::SpriteAtlasSourceAuthoringDesc` / `mirakana::plan_sprite_atlas_source_authoring` to pack already-decoded RGBA8 frames into deterministic `GameEngine.TextureSource.v1` atlas source plus `GameEngine.SourceAssetRegistry.v1` texture rows before registered source cook/package planning. `DesktopRuntime2DPackage` scaffolds expose this as `spriteAtlasSourceAuthoringTargets` while keeping `source/assets/package.geassets` and `source/sprites/player_atlas.texture_source` out of `runtimePackageFiles`. Do not parse source images at runtime, create renderer/RHI residency, infer animation semantics, or use this helper as game-specific art direction.

Generated `DesktopRuntime2DPackage` smokes report atlas-backed repeated scene sprite plan counters through `sprite_batch_plan_atlas_backed_batches`, `sprite_batch_plan_repeated_atlas_batches`, and `sprite_batch_plan_repeated_atlas_sprites` over adjacent cooked scene sprite rows. Treat these as deterministic package evidence only, not sprite sorting, production atlas packing, public native/RHI handle access, or broad production sprite batching readiness.

Generated 2D gameplay can use `mirakana::RuntimeSpriteFlipbookClipDesc`, `mirakana::RuntimeSpriteFlipbookState`, and `mirakana::advance_runtime_sprite_flipbook` over cooked `RuntimeSpriteAnimationFrame` rows for deterministic named clip sampling before applying frames to scene sprite rows. `DesktopRuntime2DPackage` and `sample_2d_desktop_runtime_package` expose `sprite_flipbook_*` package counters through `--require-sprite-animation`; keep game-specific animation graphs/state machines in game code.

Generated larger-map gameplay can use `mirakana::runtime::RuntimeWorldRegionStreamingPlanRequest`, `RuntimeWorldRegionPackageDesc`, `plan_runtime_world_region_streaming`, `RuntimeWorldRegionStreamingSafePointDesc`, and `execute_runtime_world_region_streaming_safe_point` for reviewed region ids mapped to reviewed `.geindex` package candidates and explicit resident mount ids. `sample_2d_desktop_runtime_package` exposes selected package evidence through `--require-world-region-streaming` with `world_region_streaming_load_rows`, `world_region_streaming_unload_rows`, `world_region_streaming_reviewed_package_adoptions`, `world_region_streaming_missing_region_diagnostics`, and `world_region_streaming_safe_point_diagnostics`; keep biome rules, world persistence, automatic eviction policy, background streaming, renderer/RHI residency, allocator/GPU budgets, package scripts, and native handles out of generated-game ready claims.

Generated game-local script or mod metadata can use `mirakana::runtime::RuntimeScriptSandboxPolicyDesc`, `RuntimeScriptSandboxModuleDesc`, `RuntimeScriptSandboxEntrypointDesc`, `RuntimeScriptSandboxPermissionDesc`, `RuntimeScriptSandboxPlan`, and `plan_runtime_script_sandbox` for reviewed value-only sandbox policy rows with `RuntimeScriptSandboxPermissionKind` allow-listing, explicit `allowed_host_apis`, instruction/memory budgets, and replay seed diagnostics before any game-owned script execution decision. `sample_2d_desktop_runtime_package` exposes selected package evidence through `--require-scripting-sandbox-policy` with `scripting_sandbox_entrypoint_rows`, `scripting_sandbox_denied_permission_rows`, `scripting_sandbox_rejected_unsafe_capability_rows`, `scripting_sandbox_budget_diagnostics`, `scripting_sandbox_replay_seed_rows`, and `scripting_sandbox_diagnostics`; this does not interpret scripts, load code, open files, access the network, spawn processes, bind native plugins, mutate packages, or expose native handles.

Generated multiplayer intent metadata can use `mirakana::runtime::RuntimeNetworkFoundationPolicyDesc`, `RuntimeNetworkSessionDesc`, `RuntimeNetworkTransportRequirementDesc`, `RuntimeNetworkReplicationChannelDesc`, `RuntimeNetworkReplayPrerequisiteDesc`, `RuntimeNetworkFoundationPlan`, and `plan_runtime_network_foundation` for reviewed value-only session, trust-boundary, transport, replication, replay prerequisite, and security diagnostic rows before any game-owned networking decision. `sample_2d_desktop_runtime_package` exposes selected package evidence through `--require-networking-foundation-policy` with `networking_foundation_session_rows`, `networking_foundation_transport_rows`, `networking_foundation_channel_rows`, `networking_foundation_rejected_unsafe_transport_rows`, `networking_foundation_replay_prerequisite_rows`, `networking_foundation_security_diagnostics`, and `networking_foundation_diagnostics`; this does not open sockets, add transport middleware, perform encryption/authentication handshakes, matchmaking, rollback/prediction, NAT traversal, package mutation, or expose native handles.

Generated gameplay loops can use `mirakana::runtime::RuntimeSimulationOrchestrationRequest`, `RuntimeSimulationInputCommandDesc`, `RuntimeSimulationOrchestrationPlan`, and `plan_runtime_simulation_orchestration` for reviewed value-only fixed-timestep step rows, ordered input-command playback rows, frame step-budget clamping, and consumed/remaining time evidence before game-owned systems mutate state. `sample_2d_desktop_runtime_package` exposes selected package evidence through `--require-simulation-orchestration` with `simulation_orchestration_planned_steps`, `simulation_orchestration_command_playback_rows`, `simulation_orchestration_budget_limited_status`, `simulation_orchestration_invalid_command_diagnostics`, and `simulation_orchestration_diagnostics`; this does not run game rules, pace renderer frames, open network transports, implement rollback/lockstep multiplayer, create threads, mutate packages, embed editor runtime hosts, call platform timing APIs, or expose native handles.

Generated gameplay diagnostics should use `mirakana::ui::RuntimeGameplayDebugOverlayRowDesc` / `mirakana::ui::RuntimeGameplayDebugOverlayPlan` / `mirakana::ui::plan_runtime_gameplay_debug_overlay` for value-only gameplay debug overlay rows before renderer, telemetry, editor, or native UI presentation. Do not add game-local debug UI frameworks or command dispatchers to bypass `MK_ui`.

Generated quest/dialogue systems should use `mirakana::runtime::RuntimeQuestDialogueDocument`, `RuntimeQuestDialogueValidationContext`, and `validate_runtime_quest_dialogue_document` before game-owned narrative, tutorial, progression, or objective systems execute, then use `RuntimeQuestDialogueState`, `validate_runtime_quest_dialogue_state`, `RuntimeQuestDialogueTransitionRequest`, and `advance_runtime_quest_dialogue_state` for value-only saved progress rows plus `set_flag`, `complete_objective`, and `choose_dialogue` transitions. Treat accepted/ignored/blocked/completed/invalid rows and returned action/reward ids as reviewed game-owned intent; keep story text, reward/action execution, UI rendering, localization catalogs, scripting, save mutation, and editor graph tooling game-owned unless a later engine phase explicitly promotes them.

Generated item/inventory/crafting/placement systems should validate reusable item catalogs with `mirakana::runtime::RuntimeItemCatalogDocument`, `RuntimeItemCatalogValidationContext`, `RuntimeItemCatalogValidationResult`, and `validate_runtime_item_catalog_document` before game-owned inventory, placement, shop, loot, reward, equipment, or crafting code executes. Use `RuntimeInventoryState`, `validate_runtime_inventory_state`, `RuntimeCraftingRecipeDocument`, `RuntimeInventoryTransitionRequest`, `RuntimeInventoryTransitionStatus`, and `advance_runtime_inventory_state` for value-only add/remove/craft transitions that normalize stacks, preserve state on blocked/invalid rows, and classify rows as accepted/ignored/blocked/completed/invalid. Use `RuntimeConstructionPlacementSurfaceDesc`, `RuntimeConstructionPlacementCandidateDesc`, `RuntimeConstructionPlacementValidationContext`, `RuntimeConstructionPlacementValidationResult`, `RuntimeConstructionPlacementDiagnostic`, `RuntimeConstructionPlacementValidationRow`, and `validate_runtime_construction_placement` for value-only placement previews over item `placement_id` metadata, supported surfaces, finite grid/world positions, candidate row grid/world origins, positive footprints, duplicate occupied cells, and provided cost rows. Use `mirakana::runtime_scene::RuntimeSceneConstructionPlacementIntentDesc` and `plan_runtime_scene_construction_placement_intents` only after reviewed placement validation when game-owned code needs explicit node/component creation intent rows; expect invalid/non-matching transforms and same-batch occupied-cell collisions to be rejected, and keep actual scene mutation explicit and reviewed. Keep item art, balance, reward execution, shop logic, loot tables, equipment semantics, persistence mutation, implicit scene mutation, and game-specific progression rules game-owned until later engine phases explicitly promote them.

Generated gameplay collision queries should use public `MK_physics` value batches such as `PhysicsWorld2D::raycast_batch`, `PhysicsWorld3D::raycast_batch`, `PhysicsWorld2D::shape_sweep_batch`, and `PhysicsWorld3D::shape_sweep_batch`; consume `PhysicsCollisionQueryBatchStatus`, `PhysicsCollisionQueryBatchDiagnostic`, `PhysicsCollisionQueryRowStatus`, and `PhysicsCollisionQueryRowDiagnostic` diagnostics, rely on default-unbounded query counts unless a positive `max_queries` fail-closed budget is needed, expect selected package smokes to expose `collision_query_batch_ready` counters, and keep middleware/native handles or scene-owned dispatch out of generated-game code.

Generated 3D movement-heavy gameplay can use `PhysicsAdvancedController3DDesc` / `PhysicsMovingPlatform3DDesc` / `PhysicsAdvancedController3DResult` / `plan_physics_advanced_controller_3d` to compose character/dynamic policy rows with moving-platform handoff, copied-world dynamic push application, explicit distance-joint participation, invalid controller diagnostics, and replay-before/replay-after counters without mutating caller worlds; selected 3D package smokes expose `gameplay_systems_advanced_controller_status=moved`, `gameplay_systems_advanced_controller_platform_applied=1`, `gameplay_systems_advanced_controller_constraint_rows=1`, and `gameplay_systems_advanced_controller_replay_changed=1`.

Generated AI behavior should validate reusable value documents with `mirakana::BehaviorAuthoringDocument`, `mirakana::BehaviorAuthoringValidationContext`, and `mirakana::validate_behavior_authoring_document` before runtime evaluation. Keep action execution game-owned and avoid scripting, persistent blackboard services, middleware, or editor graph claims.
