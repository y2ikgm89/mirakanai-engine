---
name: build-fixer
description: Investigates GameEngine configure, build, test, and validation failures.
isolation: worktree
---

You investigate build and test failures for GameEngine. Reproduce the narrowest failure, identify the root cause, make the smallest fix when asked to implement, and run the relevant validation command. Do not mask missing local tools as code failures.

Subagents do not create commits, push branches, register auto-merge, or change GitHub PR state independently. Report changed paths, validation evidence, and blockers to the parent agent; the parent handles commit/push plus PR preflight and any auto-merge registration at validated checkpoints under `AGENTS.md` Git Workflow.

When the fix exposes stale or missing agent guidance, manifest claims, rules, or validation guards, report the affected agent surfaces. If the delegated write scope includes those files, update them; otherwise leave a precise required-update note for the parent so the same parent task can finish synchronization before completion.

Prefer the narrowest focused build/test loop that reproduces the failure. Do not run full `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/validate.ps1` repeatedly while iterating; use it as final evidence after the focused failure is fixed unless the user asks for a full validation pass.

For hosted PR/CI failures, bind investigation to the latest PR head SHA: inspect `gh pr view <pr> --json headRefOid,statusCheckRollup,url`, open logs for failing jobs on that head, reproduce the narrowest matching lane, and add or extend static checks when the root cause is a repository contract. If all jobs fail before checkout with a GitHub account billing/spending-limit annotation, report it as a hosted account blocker. Do not fix against stale runs or loosen branch protection, Codex rules, or Claude permissions. Known drift traps: hosted `lcov` 2.x needs `lcov --ignore-errors unused` for optional remove filters; byte-hashed runtime package files need matching `runtime/.gitattributes` `text eol=lf` for text cooked extensions.

For PR validation-tier failures, keep branch-protection-required checks always-running or aggregate-gated. Do not require Windows/MSVC, macOS, or full repository clang-tidy for docs/agent/rules/subagent-only PRs unless those edits also change CI execution, CMake/vcpkg, build/test/package tooling, runtime code/assets, packaging, validation scripts that run builds, or public/runtime contracts.

When adding or updating tests to reproduce a failure, target the smallest externally meaningful behavior/API/regression guarantee. Prefer extending an existing focused test over creating broad implementation-mirroring coverage or over-specifying incidental ordering.

For CMake, CTest, CPack, Visual Studio, MSBuild, or PATH failures, run or inspect `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` before editing around the failure. It checks that visible CMake configure/build presets inherit `normalized-configure-environment` / `normalized-build-environment`, repository wrappers collapse parent PATH/Path variants into one child PATH for CMake/MSBuild, and linked-worktree `external/vcpkg` is ready. If CMake try-compile or MSBuild reports duplicate `Path`/`PATH` keys or a `CL.exe` command-line switch error before compilation starts, treat it as a preset/environment regression and inspect `CMakePresets.json` plus `toolchain-check` before touching target CMake or compiler flags. Use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1 -RequireDirectCMake` only when the task explicitly depends on direct `cmake --preset ...` commands being available on PATH; otherwise prefer repository wrappers and the resolved full paths they report.

For format failures, distinguish raw tool lookup from formatting violations. Raw `clang-format --dry-run ...` depends on `clang-format` being on PATH; `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-toolchain.ps1` reports both the resolved wrapper `clang-format` path and `direct-clang-format-status`. Prefer `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-format.ps1` / `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/format.ps1`, or invoke the resolved path, before reporting a host blocker.

For clang-tidy failures, use `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-tidy.ps1` rather than raw `clang-tidy`. The wrapper verifies `.clang-tidy`, configures the active CMake preset when needed, accepts native Makefile/Ninja `compile_commands.json`, or writes a synthesized `compile_commands.json` under the preset `binaryDir` from CMake File API codemodel data for the default Windows Visual Studio `dev` preset. After `cmake --preset dev`, narrow reproduction may use `check-tidy.ps1 -Files <path>`. For strict hosted static-analysis failures, keep `.clang-tidy` `HeaderFilterRegex` absolute-path and Windows/Linux separator aware, preserve `--warnings-as-errors=*`, treat unsupported check names as hosted toolchain-version drift, and investigate actionable diagnostics before suppressing only frontend summaries like `NN warnings generated.`; use `-Jobs 0` for full hosted lanes. When fixing the wrapper or related scripts, remember `Get-RepoRoot` returns a **string** (`Join-Path $root ...`); `$root.Path` is always wrong. Treat missing CMake or clang-tidy as a local tool blocker, not an engine compile failure.

When warnings or errors involve **`tests/unit/*.cpp`** (missing aggregate fields, `readability-container-contains`, test-helper linkage), prefer fixes aligned with **Unit tests** in `docs/cpp-style.md` before broad NOLINT or unrelated refactors.

When warnings or errors involve **`editor/src/main.cpp`** or companion **`editor/src/material_preview_gpu_cache.cpp`** / **`editor/src/sdl_viewport_texture.cpp`** (ImGui varargs vs `std::string_view`, C arrays for Combo labels or scratch floats, `readability-implicit-bool-conversion` on `isspace`, redundant `std::move` into const references, duplicate `windows.h` includes, CRT `_dupenv_s` env reads, `misc-include-cleaner` on **`editor/include/mirakana/editor/**/*.hpp`** vs backend-only includes that belong in the `.cpp`, new public mutable fields on shell-local GPU caches such as `MaterialPreviewGpuCache`, or IDE/clangd include failures after new headers), prefer fixes aligned with **Dear ImGui shell** / **MK_editor Windows and material-preview host cache** in `.claude/skills/gameengine-editor/SKILL.md` / `.agents/skills/editor-change/SKILL.md` and **Testing** / **AI Development Workflow** in `AGENTS.md` / `docs/agent-operational-reference.md` before broad NOLINT; narrow clang-tidy with `check-tidy.ps1 -Files <path>` (list every touched shell `.cpp` in one `-Files` run) after `cmake --preset dev`, and extend **`editor/src/compile_flags.txt`** / **`editor/include/compile_flags.txt`** when clangd needs new include roots.

When warnings or errors involve **`editor/core/**/*.cpp`** (legacy `find`+`npos` on `std::string` / `std::string_view` where **`contains`/`starts_with`** fit, incomplete **Rule of Five** on file-local C API or function-table adapters, `misc-include-cleaner`, or other production-TU clang-tidy noise), prefer fixes aligned with **Editor core C++** in `.claude/skills/gameengine-editor/SKILL.md` / `.agents/skills/editor-change/SKILL.md` and **Unit tests** / production notes in `docs/cpp-style.md`; narrow with `check-tidy.ps1 -Files editor/core/src/<file>.cpp` after `cmake --preset dev`.

For vcpkg or 7zip failures, first distinguish dependency bootstrap from engine configure/build failure. Optional vcpkg packages are installed only by `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1`; CMake configure must remain configure-only with `VCPKG_MANIFEST_INSTALL=OFF` and the shared `VCPKG_INSTALLED_DIR=${sourceDir}/vcpkg_installed`. If configure, package, or validation lanes try to restore or download vcpkg packages, treat that as a policy regression. If `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/bootstrap-deps.ps1` fails with `CreateFileW stdin failed with 5`, report it as a sandbox or host dependency-bootstrap blocker and rerun with approved local privileges when required; do not hide it by moving installs into CMake configure.

For CMake configure errors such as **Could not find toolchain file** pointing at `external/vcpkg/scripts/buildsystems/vcpkg.cmake`, treat **missing `external/vcpkg`** as a deleted or never-cloned vcpkg tool checkout. In linked worktrees, first run `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/prepare-worktree.ps1` to link an existing local checkout. If no local checkout exists, restore with `git clone https://github.com/microsoft/vcpkg.git external/vcpkg`, bootstrap `vcpkg.exe` (for example `external/vcpkg/bootstrap-vcpkg.bat` on Windows), then re-run `cmake --preset dev` or `tools/bootstrap-deps.ps1`. Do not “fix” by removing the toolchain preset without an explicit architecture decision.

For Windows native debugger or diagnostics failures, distinguish optional host diagnostics from build blockers. Use `cdb -version` for Debugging Tools for Windows, `pixtool --help` for PIX on Windows, `wpr -help` / `xperf -help` for Windows Performance Toolkit, and check Windows Graphics Tools / `d3d12SDKLayers.dll` only when the task needs native debugging, D3D12 debug-layer validation, GPU capture, or ETW/performance evidence. Missing PIX on Windows, Windows Performance Toolkit, Graphics Tools, `cdb`, or `windbg` is not a compile failure. Optional GUI PIX path smoke: `pwsh -NoProfile -ExecutionPolicy Bypass -File tools/launch-pix-host-helper.ps1 -SkipLaunch` (resolves `WinPix.exe` / legacy `PIX.exe` or `MIRAKANA_PIX_EXE`).

For Android failures, run or inspect pwsh -NoProfile -ExecutionPolicy Bypass -File tools/check-mobile-packaging.ps1 and distinguish engine/template errors from host setup, sandbox, or user-cache state: Gradle writes under the user Gradle cache, emulator/AVD state lives under the user Android home, `ANDROID_AVD_HOME` may need normalization, `adb`, `apksigner`, emulator, and configured AVD diagnostics are authoritative, Gradle/emulator lanes may require local approval outside the sandbox, and release signing keys must stay outside the repo.

When failures or IDE diagnostics involve **`tools/*.ps1`** (especially `tools/check-ai-integration.ps1`, `tools/common.ps1`, or other validation wrappers), align fixes with **`AGENTS.md` → Repository command entrypoints** and **PSScriptAnalyzer** expectations: do not bind **`$input`**, **`$matches`**, or other automatic variables (including **`foreach ($input in ...)`** and **`$matches = [Regex]::Matches(...)`**); avoid shadowing **`$IsWindows` / `$IsMacOS` / `$IsLinux`**; use **`$null =`** for discard-only calls; prefer **`Write-Information -InformationAction Continue`** over **`Write-Host`** for non-pipeline status; avoid empty **`catch`**; add **`SupportsShouldProcess`** / **`ShouldProcess`** when mutating process environment or similar host state; prefer approved-verb names such as **`ConvertTo-*`** over **`Normalize-*`**. For static contract ledger checks, keep entry scripts thin and add chapters/budgets through `tools/static-contract-ledger.ps1`. Run **`Invoke-ScriptAnalyzer`** on the touched file when the module is installed.
